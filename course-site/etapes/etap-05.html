§ I · ТемыTopics

Что значит «отдать проект миру»What “shipping a project to the world” means

Git как «машина времени» для проектаGit as a project’s “time machine”

Git — это программа, которая фиксирует состояние вашего проекта в любой момент. Каждое сохранение называется коммит — это «фотография» состояния всех файлов.

Что это даёт:

  • Можете в любой момент откатиться к старой версии: «верни как было два дня назад».
  • Видите историю: что и когда менялось, кем (если работали с кем-то).
  • Можете экспериментировать: создаёте «параллельную линию» (ветку), пробуете что-то, и если не понравилось — просто отбрасываете.
  • Можете делиться: загружаете на GitHub — и у всех есть полная история проекта.

Аналогия: «Версии файлов» в macOS или «Журнал версий» в Google Docs — но для всего проекта целиком, и на стероидах.

Минимум команд (которые Claude использует за вас):

# создать репозиторий в текущей папкеinit a repo in the current folder
$ git init

# сохранить состояние (коммит)save state (commit)
$ git add .
$ git commit -m "первый коммит"

# отправить на GitHubsend to GitHub
$ git push

Запоминать их не надо. Достаточно понимать смысл: добавить в индекс → зафиксировать как коммит → отправить на удалённый.

Git is a program that records your project’s state at any moment. Each save is called a commit — a “photo” of all files.

What that gives:

  • Roll back at any time: “revert to two days ago.”
  • See history: what changed when, by whom (if collaborating).
  • Experiment: create a “parallel line” (branch), try things, drop if you dislike.
  • Share: push to GitHub — everyone has the full history.

Analogy: “Versions” in macOS or “Version history” in Google Docs — but for the whole project, on steroids.

The minimum commands (Claude runs them for you):

# init a repo in the current folder
$ git init

# save state (commit)
$ git add .
$ git commit -m "first commit"

# send to GitHub
$ git push

You don’t need to memorise them. Just understand the meaning: stage → commit as a snapshot → push to the remote.

Репозиторий, ветки и удалённыйRepository, branches and remote

Репозиторий — это папка, в которой git ведёт историю. Внутри есть скрытая папка .git — именно там лежат все коммиты.

Ветка — это параллельная линия истории. Главная называется main (раньше — master). Когда хотите попробовать что-то рискованное — создаёте новую ветку (например, experiment/dark-theme), коммитите туда, и если получилось — сливаете (merge) обратно в main. Если не получилось — просто удаляете ветку.

Удалённый (remote) — это копия вашего репозитория где-то на сервере. Обычно — на GitHub. Команда push отправляет ваши коммиты туда, pull — забирает изменения оттуда.

Типичный диалог:

> создай в этой папке git-репозиторий, сделай первый коммит «init».
> создай ветку experiment/dark-theme.
> в ней сделай dark.html (тёмная версия моей страницы).
> покажи изменения.
> если выглядит нормально — слей в main и удали ветку.

A repository is a folder git keeps history for. Inside is a hidden .git folder — that’s where commits live.

A branch is a parallel history line. The main one is main (used to be master). For something risky you make a new branch (e.g. experiment/dark-theme), commit there, and on success merge back to main. If it failed — drop the branch.

A remote is a copy of your repo on a server. Usually GitHub. push uploads your commits, pull fetches changes from there.

Typical dialogue:

> init a git repo in this folder, first commit "init".
> create branch experiment/dark-theme.
> in it add dark.html (a dark version of my page).
> show me the changes.
> if it looks fine — merge into main and delete the branch.
GitHub Pages: бесплатный хостинг для ваших страницGitHub Pages: free hosting for your pages

GitHub Pages — это «магия»: загружаете HTML/CSS/JS в репозиторий на GitHub, включаете одну галочку в настройках — и через минуту ваш сайт доступен по адресу https://<username>.github.io/<repo>/.

Что подходит:

  • статичные сайты без бэкенда (всё, что мы делали в этапах 00–04);
  • портфолио, лендинги, дашборды на CSV-данных;
  • демо ваших калькуляторов и игр.

Что не подходит:

  • сайты с серверной логикой (этап 06 будет про них — через Vercel или Railway);
  • проекты с секретами, которые требуют живой работы с API (вызовы к Alpha Vantage с ключом нельзя делать прямо в браузере).

Полный диалог публикации:

> создай на github новый публичный репозиторий "fi-forecast".
> запушь туда текущий проект.
> включи GitHub Pages с источником main / root.
> дай мне публичную ссылку и проверь через playwright,
> что страница открывается и не сломалась.

Через минуту — ссылка вида https://yourname.github.io/fi-forecast/, которую можно показать кому угодно.

GitHub Pages is the “magic”: upload HTML/CSS/JS to a GitHub repo, flip one toggle, and a minute later your site is live at https://<username>.github.io/<repo>/.

Fits:

  • static sites, no backend (everything from etapes 00–04);
  • portfolios, landing pages, CSV-data dashboards;
  • your calculator and game demos.

Doesn’t fit:

  • sites with server logic (etape 06 — Vercel or Railway);
  • projects with secrets that need live API calls (Alpha Vantage requests with a key cannot run directly in the browser).

Full publishing dialogue:

> create a new public github repo "fi-forecast".
> push the current project there.
> enable GitHub Pages with source main / root.
> give me the public URL and verify via playwright
> that the page opens and isn't broken.

A minute later — a URL like https://yourname.github.io/fi-forecast/, shareable with anyone.

README и .gitignore: «обложка» и «не показывать»README and .gitignore: the “cover” and the “do not show”

README.md — первое, что видит человек, открывший ваш репозиторий на GitHub. Без него странно, как лекция без названия.

Минимум для хорошего README:

  1. Заголовок и одна строка о чём проект.
  2. Демо-ссылка — на GitHub Pages или скриншот.
  3. Что это умеет — 3–5 пунктов.
  4. Как запустить локально — 2–3 команды.
  5. Откуда данные / какие нужны секреты — если применимо.
  6. Лицензия — MIT по умолчанию хорошо.

Claude составит README за вас одной командой:

> составь README.md для этого проекта.
> опиши: что делает, как запустить, демо-ссылка
> (https://...github.io/...), какие нужны секреты,
> лицензия MIT. сделай двуязычным: ru + en.

.gitignore — список «что git не должен видеть»: .env, data/private/, node_modules/, .DS_Store. Без него можно случайно опубликовать API-ключ в интернет — и через час кто-то будет тратить ваши деньги через ваш аккаунт.

> составь .gitignore для этого проекта.
> обязательно: .env, data/personal/, *.log,
> макросные файлы macOS/Windows.
> добавь раздел "private" с моими личными исключениями.

README.md — the first thing anyone sees opening your repo. Without one it feels like a lecture without a title.

Minimum for a good README:

  1. Title and a one-liner.
  2. Demo link — to GitHub Pages or a screenshot.
  3. What it does — 3–5 bullets.
  4. How to run locally — 2–3 commands.
  5. Data sources / required secrets — if any.
  6. Licence — MIT is a good default.

Claude drafts a README in one prompt:

> draft README.md for this project.
> describe: what it does, how to run, demo URL
> (https://...github.io/...), required secrets,
> MIT licence. make it bilingual: ru + en.

.gitignore — the list of “what git must not see”: .env, data/private/, node_modules/, .DS_Store. Without it you may accidentally publish an API key — and within an hour someone burns your money through your account.

> draft .gitignore for this project.
> required: .env, data/personal/, *.log,
> macOS/Windows junk files.
> add a "private" section with my personal exclusions.
Защита секретов: что нельзя коммититьSecret hygiene: what never to commit

Самая дорогая ошибка непрофессионала — случайно отправить API-ключ или пароль в публичный репозиторий. Боты сканируют GitHub каждые несколько минут — ключ найдут за часы.

Чего НЕ делать никогда:

  • не писать ключи прямо в HTML/JS-файлах;
  • не коммитить .env;
  • не класть пароли в CLAUDE.md;
  • не коммитить дампы баз с личными данными;
  • не копировать SSH-ключ в repo.

Как защититься:

  1. .gitignore с .env, secrets/, data/private/;
  2. в коде — всегда читаем секреты из окружения (process.env.X, os.getenv("X"));
  3. в README — пример .env без значений (.env.example);
  4. перед каждым коммитом — «покажи git diff, проверь, не утекают ли секреты»;
  5. регулярные сканы — через hooks (этап 08a) или внешние сервисы (например, gitleaks).

Дисциплина:

> перед тем как пушить, прогон проверки:
> git diff --staged | grep -iE "key|secret|token|password"
> если что-то нашлось — откати и поправь, перед коммитом не пускай.

Если случайно опубликовали — не тяните: отозвать ключ через сервис, поменять пароль, и только потом думать про «как удалить из истории git» (это сложнее, чем кажется).

The costliest amateur mistake — accidentally pushing an API key or password to a public repo. Bots scan GitHub every minute — they find keys in hours.

Never:

  • write keys directly in HTML/JS files;
  • commit .env;
  • store passwords in CLAUDE.md;
  • commit DB dumps with personal data;
  • copy your SSH key into the repo.

Defenses:

  1. .gitignore with .env, secrets/, data/private/;
  2. in code — always read secrets from environment (process.env.X, os.getenv("X"));
  3. in README — provide an empty .env.example;
  4. before every commit — “show git diff, check that no secrets leak.”;
  5. regular scans — via hooks (etape 08a) or external services (e.g. gitleaks).

Discipline:

> before pushing, run a check:
> git diff --staged | grep -iE "key|secret|token|password"
> if anything matches — revert and fix before commit.

If accidentally published — don’t delay: revoke the key, change the password, then think about “erase from git history” (harder than it looks).

Worktrees: параллельные ветки без путаницыWorktrees: parallel branches without confusion

Обычная работа с ветками — «переключайся туда-сюда»: git checkout main, потом git checkout experiment/dark-theme. Состояние папки меняется: одни и те же файлы превращаются в разные версии.

Worktrees — альтернатива: каждая ветка живёт в своей папке. Не надо переключаться — вы просто открываете другую папку.

Когда полезно:

  • Хотите делать две вещи параллельно — например, исправить баг в main и дописать фичу в experiment;
  • Запускаете агента-стажёра в другой ветке (этап 07 — subagents часто работают в worktree, чтобы не мешать вам);
  • Не хотите случайно потерять незакоммиченные изменения при переключении.

Команды (Claude делает за вас):

# создать worktree в папке-сестре
$ git worktree add ../my-project-dark experiment/dark-theme

# список всех worktrees
$ git worktree list

# удалить worktree, когда не нужен
$ git worktree remove ../my-project-dark

Skill superpowers:using-git-worktrees добавляет дисциплину — Claude правильно создаёт worktree и убирает за собой.

Default branch work is “toggle in place”: git checkout main, then git checkout experiment/dark-theme. The folder’s state shifts: the same files become different versions.

Worktrees are an alternative: each branch lives in its own folder. No checkout dance — you just open another folder.

Useful when:

  • You want to do two things in parallel — fix a bug on main and add a feature on experiment;
  • You run an intern agent on another branch (etape 07 — subagents often live in worktrees so as not to bother you);
  • You don’t want to lose uncommitted changes on a checkout.

Commands (Claude runs them):

# add a worktree as a sibling folder
$ git worktree add ../my-project-dark experiment/dark-theme

# list worktrees
$ git worktree list

# remove a worktree
$ git worktree remove ../my-project-dark

The superpowers:using-git-worktrees skill adds discipline — Claude creates worktrees properly and cleans them up.

Форки и Pull Requests — на будущееForks and Pull Requests — for later

На этом этапе мы работаем со своими репозиториями. Но стоит знать два понятия для будущего:

  • Форк — кнопка «Fork» на чужом репо. Вы получаете свою копию, в которой можете делать что угодно. Полезно, если хотите доработать чужую тему сайта или взять чей-то проект как основу.
  • Pull Request (PR) — способ предложить владельцу оригинального репозитория ваши изменения. Сделали форк, исправили опечатку в README, послали PR — владелец смотрит и либо принимает, либо отклоняет.

Зачем это нужно сейчас знать: на капстоуне, возможно, будете публиковать собственный плагин для Claude Code. Когда другие пользователи захотят вам помочь — они сделают форк и PR. Через GitHub CLI это работает за 30 секунд.

This etape works on your own repos. Still, two terms to know for later:

  • Fork — the “Fork” button on someone else’s repo. You get your own copy and can do anything in it. Useful when you want to adapt a theme or take a project as a base.
  • Pull Request (PR) — a way to propose changes to the original owner. You forked, fixed a README typo, sent a PR — the owner reviews and accepts or declines.

Why know this now: at the capstone you may publish your own Claude Code plugin. When others want to help — they’ll fork and PR. Through the GitHub CLI it’s a 30-second flow.

§ V · Словарь этапаEtape glossary

Что добавилось к словарюNew entries in the glossary

Git
Программа контроля версий. «Машина времени» для проекта — фиксирует каждое изменение, позволяет откатываться, разделять историю на ветки и делиться. A version-control program. A “time machine” for a project — records every change, allows rollback, branches and sharing.
РепозиторийRepository
Папка проекта, которую отслеживает git. Внутри — скрытая .git с историей всех изменений. A project folder tracked by git. Inside — a hidden .git with the change history.
КоммитCommit
«Фотография» проекта в момент сохранения. Имеет автора, дату и описание. Можно вернуться к любому коммиту в истории. A “snapshot” of the project at save time. Has an author, date and message. You can return to any commit in history.
ВеткаBranch
Параллельная линия истории. Главная — main. На экспериментальные ветки делают рискованные изменения, потом сливают в main или отбрасывают. A parallel history line. The main one — main. Risky edits live on experiment branches, then merge or get dropped.
Удалённый (remote)Remote
Копия вашего репозитория на стороннем сервере (обычно GitHub). git push отправляет туда коммиты, git pull забирает изменения оттуда. A copy of your repo on a third-party server (usually GitHub). git push sends commits, git pull fetches changes.
GitHub Pages
Бесплатный хостинг от GitHub: загрузил HTML/CSS/JS в репозиторий, включил галочку — и сайт доступен по адресу https://<username>.github.io/<repo>/. Free GitHub hosting: upload HTML/CSS/JS, flip a switch — your site is live at https://<username>.github.io/<repo>/.
README.md
Главная «обложка» репозитория. Markdown-файл в корне. GitHub показывает его на странице репо. Туда — что это, как запустить, демо-ссылка, скриншоты, лицензия. A repo’s “cover.” A markdown file at the root. GitHub renders it on the repo page. Put: what it is, how to run, demo URL, screenshots, licence.
.gitignore
Файл со списком того, что git должен игнорировать: .env, секреты, кэши, временные файлы, node_modules/. Без него легко случайно опубликовать API-ключ. A file listing what git should ignore: .env, secrets, caches, temp files, node_modules/. Without it, you easily leak an API key.
Worktree
Изолированная копия репозитория в отдельной папке — для параллельной работы над несколькими ветками одновременно. Создаётся через git worktree add, удаляется через git worktree remove. An isolated repo copy in a separate folder — for parallel work across several branches at once. Created via git worktree add, removed via git worktree remove.
ФоркFork
Ваша личная копия чужого репозитория на GitHub. Делается одной кнопкой. Вы можете в ней что угодно — оригинал не трогается. Your personal copy of someone else’s GitHub repo. One-click. You can do anything in yours — the original is untouched.
Pull Request (PR)
Способ предложить владельцу другого репозитория ваши изменения. Сделали форк, поправили, послали PR — владелец смотрит, принимает или отклоняет. A way to propose changes to another repo’s owner. You forked, edited, opened a PR — the owner reviews and accepts or rejects.
gh (GitHub CLI)
Командная утилита от GitHub: создавать репозитории, делать PR, смотреть issues — всё из терминала. Claude использует gh для большинства действий с GitHub. A command-line tool by GitHub: create repos, open PRs, browse issues — all from the terminal. Claude uses gh for most GitHub actions.
§ VI · Quiz

Семь вопросов на закреплениеSeven questions for reinforcement

  1. Чем коммит отличается от «сохранить» в Word?How is a commit different from “save” in Word?

    показать ответshow answer

    «Сохранить» в Word перезаписывает файл — старая версия теряется. Коммит — «фотография» проекта в момент сохранения, со всеми файлами, автором, датой и описанием. Все коммиты остаются в истории, можно вернуться к любому.

    “Save” in Word overwrites the file — the old version is gone. A commit — a “snapshot” of the project at save time, with all files, author, date and message. Every commit stays in history; you can return to any.

  2. Что точно нельзя коммитить в публичный репозиторий?What must never be committed to a public repo?

    показать ответshow answer

    API-ключи, пароли, токены, дампы баз с личными данными, личные финансовые выписки, SSH-ключи. Всё это идёт в .env и .gitignore. Боты сканируют GitHub каждые несколько минут — утёкший ключ найдут за часы.

    API keys, passwords, tokens, DB dumps with personal data, personal financial statements, SSH keys. All those go to .env and .gitignore. Bots scan GitHub every minute — a leaked key is found in hours.

  3. Что такое GitHub Pages и для каких проектов он подходит?What is GitHub Pages and which projects does it fit?

    показать ответshow answer

    Бесплатный хостинг для статичных сайтов. Подходит: HTML/CSS/JS без серверной логики, портфолио, дашборды на CSV, лендинги, демо. Не подходит: проекты с серверным кодом, проекты, которые должны делать API-вызовы с секретным ключом (для них — Vercel/Railway, этап 06).

    Free hosting for static sites. Fits: HTML/CSS/JS without server logic, portfolios, CSV dashboards, landings, demos. Doesn’t fit: projects with server code, ones that need to call APIs with a secret key (for those — Vercel/Railway, etape 06).

  4. Зачем нужен worktree, если есть обычный git checkout?Why a worktree if normal git checkout exists?

    показать ответshow answer

    Чтобы работать над двумя ветками одновременно, не переключаясь. Каждый worktree — своя папка. Не нужно прятать незакоммиченные изменения, не нужно ждать, пока всё закоммитится. Особенно полезно для subagents (этап 07): агент работает в своей папке, не мешая вам.

    To work on two branches at the same time, no checkout dance. Each worktree — its own folder. No stashing uncommitted changes, no waiting to commit. Especially useful for subagents (etape 07): an agent works in its own folder without bothering you.

  5. Что обязательно должно быть в хорошем README?What must a good README contain?

    показать ответshow answer

    Шесть пунктов: заголовок и строка о чём проект; демо-ссылка или скриншот; что это умеет (3–5 пунктов); как запустить локально; откуда данные / какие нужны секреты; лицензия. Через 30 секунд чтения постороннему должно быть понятно, чем проект полезен.

    Six items: title and a one-liner; demo URL or screenshot; what it does (3–5 bullets); how to run locally; data source / required secrets; licence. In 30 seconds a stranger should grasp what the project is good for.

  6. Что делать, если случайно опубликовали API-ключ в публичный репо?What to do if you accidentally pushed an API key to a public repo?

    показать ответshow answer

    Сразу — отозвать ключ через сервис, выпустивший его. Поменять пароль если ключ был связан. Только потом думать про «удалить из истории git» (это сложнее, чем кажется — даже после git rebase копии могут остаться у ботов, скачавших ключ за первые минуты).

    First — revoke the key at its issuing service. Change the related password. Only then think “remove from git history” (harder than it looks — even after git rebase, bots that grabbed the key in the first minutes have copies).

  7. Какой минимум команд git нужно понимать (даже если их запускает Claude)?Which minimal git commands should you understand (even if Claude runs them)?

    показать ответshow answer

    git init (создать репо), git add (добавить в индекс), git commit (зафиксировать как коммит), git push (отправить на удалённый), git pull (забрать изменения), git status (что сейчас), git log (история), git branch (ветки), git checkout (переключиться).

    git init (create repo), git add (stage), git commit (record commit), git push (send to remote), git pull (fetch), git status (current state), git log (history), git branch (branches), git checkout (switch).
