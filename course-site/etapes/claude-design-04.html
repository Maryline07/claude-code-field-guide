§ Этап 04Etape 04  ·  Интерактивный прототипInteractive Prototype

Не mockup.
Кликаемая штука. Not a mockup.
A clickable thing.

Mockup отвечает на вопрос «как это выглядит». Прототип отвечает «как это работает». В Claude Design прототип — это многоэкранная композиция с графом переходов и состояниями каждого экрана. На этом этапе делаем кликабельные прототипы, которые можно дать живому человеку и посмотреть, что он будет нажимать.

A mockup answers “how it looks.” A prototype answers “how it works.” In Claude Design a prototype is a multi-screen composition with a transition graph and states per screen. This etape ships clickable prototypes you can hand to a real person and watch where they tap.

Прототип vs mockup: что меняетсяPrototype vs mockup: what changes

Mockup — статичный экран. «Вот как выглядит главная». Хорош, когда нужно показать стиль, типографику, иерархию — до того, как обсуждать поведение.

Прототип — набор экранов с переходами. «Вот как пользователь идёт от входа к экспорту через 4 клика». Хорош, когда нужно проверить пользовательские решения: не «красиво ли», а «понятно ли», «найдут ли», «не запутаются ли».

В Claude Design прототип строится той же дискуссией, что mockup. Вы описываете в промпте экраны и переходы:

> интерактивный прототип бронирования столика в кофейне «Северный край».
> формат: мобильный 375×812.
> стиль: workspace tokens (загруженные).
> экраны и переходы:
>   1. главный (логотип, «забронировать столик» кнопка)
>       → клик «забронировать» → экран 2.
>   2. выбор даты и времени (календарь, доступные слоты)
>       → выбор слота → экран 3.
>   3. данные гостя (имя, тел, кол-во гостей, особые пожелания)
>       → клик «подтвердить» → экран 4.
>   4. подтверждение + опция «добавить в календарь».
> для каждого экрана покажи default-состояние.
> для экрана 2 покажи также «ничего нет на сегодня» (empty-state).
> для экрана 3 покажи validation-error (телефон неверного формата).

На холсте появятся 4 экрана с явными стрелками-переходами между ними, плюс альтернативные состояния. Это и есть прототип.

Mockup — a static screen. “Here is what the home looks like.” Good when you need to show style, typography, hierarchy — before discussing behaviour.

Prototype — a set of screens with transitions. “Here is how the user goes from log-in to export in 4 taps.” Good when you need to verify user decisions: not “is it pretty,” but “is it clear,” “will they find it,” “will they get lost.”

In Claude Design a prototype is built by the same dialogue as a mockup. You describe screens and transitions in the prompt:

> interactive table-booking prototype for café "Northern Coast."
> format: mobile 375×812.
> style: workspace tokens (loaded).
> screens and transitions:
>   1. home (logo, "book a table" button)
>       → tap "book" → screen 2.
>   2. date / time picker (calendar, available slots)
>       → slot tap → screen 3.
>   3. guest details (name, phone, party size, notes)
>       → "confirm" → screen 4.
>   4. confirmation + "add to calendar" option.
> show default state per screen.
> on screen 2 also show "nothing available today" (empty state).
> on screen 3 show validation error (bad phone format).

The canvas shows the 4 screens with explicit arrows between them, plus alternative states. That is the prototype.

Граф навигации: рисуем поток до экрановNavigation graph: draw the flow before the screens

Главная ошибка новичков — начать с «сделай главный экран», потом постепенно нарастить переходы. Это даёт неаккуратный прототип, в котором каждый экран нарисован «сам по себе».

Правильный путь — сначала граф:

> не делай дизайн пока. помоги составить граф навигации
> для прототипа бронирования столика. опиши:
>   — узлы (экраны)
>   — рёбра (переходы между ними)
>   — для каждого ребра: какое действие пользователя его запускает
>   — альтернативные пути и тупики (back-buttons, error-states)
> формат вывода: список узлов + таблица переходов.

Claude вернёт что-то вроде:

УЗЛЫ:
  S1: home
  S2: date/time picker
  S3: guest details
  S4: confirmation
  S5: error / "no slots today"

ПЕРЕХОДЫ:
  S1 →(tap "book")→ S2
  S2 →(tap slot)→ S3
  S2 →(no slots loaded)→ S5
  S3 →(tap "confirm", valid)→ S4
  S3 →(tap "confirm", invalid phone)→ S3 (error state, stay)
  S4 →(tap "add to calendar")→ external
  S2 →(back)→ S1
  S3 →(back)→ S2

Теперь у вас понятная карта. Дальше — промпт «нарисуй экраны по этому графу». Каждый экран будет осознанной частью целого, а не «просто красивая картинка».

Если граф из 3+ экранов и 5+ переходов — обязательно проговорите его словами с Claude до дизайна. Несколько минут на граф экономят 30 минут на «ой, тут забыли back».

The newcomer mistake — start with “make the home screen,” then bolt on transitions. Gives a sloppy prototype where each screen is drawn “on its own.”

The right path — graph first:

> do not design yet. help me draft a navigation graph
> for a table-booking prototype. describe:
>   — nodes (screens)
>   — edges (transitions)
>   — what user action triggers each edge
>   — alternative paths and dead-ends (back, errors)
> output format: list of nodes + transition table.

Claude returns something like:

NODES:
  S1: home
  S2: date/time picker
  S3: guest details
  S4: confirmation
  S5: error / "no slots today"

EDGES:
  S1 →(tap "book")→ S2
  S2 →(tap slot)→ S3
  S2 →(no slots loaded)→ S5
  S3 →(tap "confirm", valid)→ S4
  S3 →(tap "confirm", invalid phone)→ S3 (error state, stay)
  S4 →(tap "add to calendar")→ external
  S2 →(back)→ S1
  S3 →(back)→ S2

Now you have a clear map. Then — the prompt “draw the screens per the graph.” Each screen is a conscious part of the whole, not “just a pretty picture.”

If the graph has 3+ screens and 5+ transitions — always talk it through with Claude before designing. Minutes on the graph save 30 minutes on “oh, we forgot the back button.”
Состояния экрана: default — это только 1 из 5Screen states: default is just 1 of 5

Каждый один экран в продакшене существует в нескольких состояниях. Если в прототипе только default — вы тестируете только идеальный сценарий. Реальные пользователи попадают в остальные.

Пять каноничных состояний:

  1. Default — данные есть, всё в порядке. Это «парадная» картинка.
  2. Empty — данных ещё нет. «Вы ещё не добавили ни одной позиции в портфель» — что показать вместо пустоты? Хороший empty-state даёт onboarding: пример, призыв к действию, ссылку на помощь.
  3. Loading — данные загружаются. Спиннер, скелетон, или контент-плейсхолдер.
  4. Error — что-то пошло не так. Сетевой, валидационный, серверный. Хороший error-state объясняет что произошло и что делать.
  5. Success — действие совершено. Тосты, чекмарки, переход на экран подтверждения.

В промпте к Claude Design можно явно запросить альтернативные состояния:

> для экрана «портфель»:
>   default — 8 позиций;
>   empty — пользователь только зарегистрировался, ничего не добавил;
>   loading — данные тянутся с брокера;
>   error — токен брокера истёк, нужно переавторизоваться.
> покажи все 4 рядом на холсте.

Claude разместит четыре варианта одного экрана рядом, чтобы вы видели всю «семью». Это критично для нормального user-testing — пользователю покажете не только «когда всё хорошо», но и «когда что-то не так».

Every single screen in production exists in several states. If your prototype only has default, you’re testing the ideal scenario. Real users land in the others.

Five canonical states:

  1. Default — data present, all good. The “press” picture.
  2. Empty — no data yet. “You haven’t added any positions to your portfolio” — what shows instead of emptiness? A good empty state offers onboarding: example, CTA, help link.
  3. Loading — data loading. Spinner, skeleton, or content placeholder.
  4. Error — something went wrong. Network, validation, server. A good error state explains what happened and what to do.
  5. Success — action completed. Toasts, checkmarks, jump to confirmation.

In the prompt you can ask explicitly for alt states:

> for the "portfolio" screen:
>   default — 8 positions;
>   empty — user just signed up, added nothing;
>   loading — pulling from broker;
>   error — broker token expired, needs reauth.
> show all 4 side by side on canvas.

Claude lays the four next to each other so you see the whole “family.” Critical for proper user testing — show users not only “happy path” but also “something off.”

User testing: 5 живых людей за один вечерUser testing: 5 real people in one evening

Прототип без тестирования — просто красивая картинка. Тестирование — это где прототип обретает смысл. Канон Якоба Нильсена: 5 пользователей ловят 80% проблем юзабилити. Не нужно 50, не нужно фокус-группы — пять живых людей и два часа.

Подготовка (10 минут):

  1. Откройте Internal URL вашего прототипа в Claude Design — это и есть «продакшен» для теста.
  2. Сформулируйте задачу, которую попросите выполнить: «забронируй столик на четверг 19:00 на двоих». Не объясняйте, как — пусть пробуют сами.
  3. Подготовьте 3 вопроса для после-задачи: «было ли что-то непонятно?», «что заметно лишнее?», «чего не хватает?».

Сессия с одним человеком (15–20 минут):

  1. Дайте задачу. Просите проговаривать вслух мысли: «вижу кнопку «забронировать» — нажму». Это «think-aloud»-протокол — основа.
  2. Не помогайте. Если запутался, говорите: «представьте, что меня нет рядом — что бы вы сделали?». Запутанность — это и есть данные.
  3. Записывайте: где спотыкается, что говорит вслух, какие неожиданные места кликает.
  4. В конце — три вопроса.

Итерация (30–60 минут):

  1. После 5 человек выпишите общие проблемы. Если 3+ человека спотыкаются на одном — это надо чинить.
  2. Возвращайтесь в Claude Design, описываете правки в диалоге: «пользователи путались на экране 3, потому что не понимали, что «notes» — опциональное поле. сделай его явно опциональным — добавь подпись «по желанию».
  3. Тестируете повторно на 3 новых людях. Если общая проблема ушла — готово.

An untested prototype is just a pretty picture. Testing is where it earns meaning. Nielsen’s canon: 5 users catch 80% of usability issues. No need for 50, no focus groups — five real humans and two hours.

Prep (10 min):

  1. Open the Internal URL of your prototype — that’s “production” for the test.
  2. Frame a task: “book a table for Thursday 7pm, two people.” Don’t explain how — let them try.
  3. Three post-task questions: “was anything unclear?”, “what feels redundant?”, “what’s missing?”

One-person session (15–20 min):

  1. Give the task. Ask them to think aloud: “I see ‘book’ — I’ll tap.” That’s the think-aloud protocol — the foundation.
  2. Don’t help. When stuck, say: “pretend I’m not here — what would you do?” Confusion is the data.
  3. Note where they stumble, what they say, unexpected taps.
  4. At the end — three questions.

Iteration (30–60 min):

  1. After 5 people, list common issues. 3+ people stumbling at the same point — needs fixing.
  2. Back to Claude Design, describe fixes in the dialogue: “users got confused on screen 3 because they didn’t see ‘notes’ was optional. Make it explicitly optional — add ‘optional’ label.”
  3. Re-test on 3 new people. If the common issue is gone — done.

  1. Возьмите простой пример: «отправить файл другу через ссылку». 3 экрана: загрузка → ссылка готова → копирование/расшаривание.
  2. В Claude Design новый проект. Промпт-граф: «помоги составить граф навигации для отправки файла по ссылке. узлы, рёбра, действия. не делай дизайн пока».
  3. Получите граф. Поправьте, если что-то не то.
  4. Промпт-дизайн: «нарисуй экраны по этому графу, мобильный 375×812, в workspace tokens. для экрана 1 покажи default + drag-and-drop hover. для экрана 2 покажи default + loading».
  5. Откройте Internal URL. Покликайте сами — почувствуйте, что прототип живой.
  1. Pick a simple example: “send a file to a friend via link.” 3 screens: upload → link ready → copy/share.
  2. New Claude Design project. Graph prompt: “help draft a navigation graph for sending a file via link. nodes, edges, actions. don’t design yet.”
  3. Get the graph. Patch what’s off.
  4. Design prompt: “draw the screens per this graph, mobile 375×812, in workspace tokens. for screen 1 show default + drag-and-drop hover. for screen 2 show default + loading.”
  5. Open the Internal URL. Click through yourself — feel the prototype live.

Что мы тренируем: порядок «граф → экраны → состояния». Это нормальный workflow для всего, что сложнее одной страницы.

What we are training: the graph → screens → states order. The normal workflow for anything beyond a single page.

Проект A · по инструкции · трек кофейняProject A · guided · café track

Бронирование столикаTable booking

Цель: 5-экранный мобильный прототип бронирования с empty/error/success-состояниями. Тестируется на 5 живых людях.

Goal: a 5-screen mobile booking prototype with empty/error/success states. Tested on 5 real people.

  1. Workspace «Северный край» (с tokens из этапа 03 или загруженный заново).
  2. Промпт-граф: попросите Claude составить граф для бронирования (5 экранов: home → date → guests → confirm → done; alternate: empty-day, validation-error, success-toast).
  3. Сверьтесь с графом. Поправьте.
  4. Промпт-дизайн: «нарисуй экраны по графу, мобильный 375×812, workspace tokens. для каждого экрана: default. для экрана 2: empty («ничего нет на сегодня, посмотрите завтра»). для экрана 3: validation-error на поле «телефон». для экрана 5: success-toast «бронь подтверждена ✓»».
  5. Откройте Internal URL. Поклассифицируйте сами по 3 раза, отметьте свои собственные «спотыкания».
  6. Тестируйте на 5 живых людях по think-aloud протоколу. Записывайте проблемы.
  7. Если 3+ человека спотыкаются на одном — правьте через диалог Claude. Re-тест на 3 новых.
  8. Финальный экспорт: handoff bundle (для этапа 05).
  1. Workspace “Northern Coast” (with etape-03 tokens or reloaded).
  2. Graph prompt: ask Claude to draft a graph for booking (5 screens: home → date → guests → confirm → done; alternates: empty-day, validation-error, success-toast).
  3. Verify the graph. Patch.
  4. Design prompt: “draw screens per the graph, mobile 375×812, workspace tokens. each screen: default. screen 2: empty (‘nothing today, try tomorrow’). screen 3: validation-error on the ‘phone’ field. screen 5: success toast ‘booking confirmed ✓’.”
  5. Open the Internal URL. Click through 3 times yourself, note your own stumbles.
  6. Test on 5 real people via think-aloud. Note issues.
  7. If 3+ people trip on the same thing — fix via Claude dialogue. Re-test on 3 new ones.
  8. Final export: handoff bundle (for etape 05).

  3. Сверьтесь.
  4. Промпт-дизайн: «нарисуй экраны по графу, мобильный портрет 1080×1920. стиль workspace. в character-select покажи 4 персонажа карточками. в battle — верх HP-bars, низ кнопки действий. в inventory — сетка 4×6. в shop — 3 категории слева, товары сетка справа».

  6. Тестируйте на 3 живых человеках, кто играет в мобильные платформеры. Замеряйте время до первой победы в battle: если >30 секунд — иерархия экрана плохая.
  7. Экспорт: PNG-набор + handoff bundle.
  1. Workspace “Aurora Pixel” (from etape 03).
  2. Graph prompt: “navigation graph for a platformer main menu. Nodes: main, character-select, battle, inventory, shop. Edges both ways (back), plus from battle there is an exit to inventory without returning to main. Describe.”
  3. Verify.
  4. Design prompt: “screens per graph, mobile portrait 1080×1920. Workspace style. In character-select show 4 cards. In battle — HP bars top, action buttons bottom. In inventory — 4×6 grid. In shop — 3 categories left, item grid right.”
  5. States: battle default + low-HP warning. Shop default + “not enough coins.”
  6. Test on 3 real people who play mobile platformers. Measure time to first battle win: if >30 sec — screen hierarchy is off.
  7. Export: PNG set + handoff bundle.

Проект C · самостоятельно · трек квартираProject C · solo · apartment track

Конфигуратор отделки квартирыApartment finishes configurator

Цель: 5-экранный прототип для дизайн-бюро интерьеров: выбор стиля → комната → материалы → смета → сохранение. Главное упражнение — many-to-many переходы (можно прыгать между комнатами).

Goal: a 5-screen prototype for an interior bureau: style → room → materials → estimate → save. The main exercise is many-to-many transitions (jumping between rooms).

  • Граф здесь сложнее линейного — пользователь должен иметь возможность вернуться на «комната» и поправить материал в любой момент. Спецификация графа критична.
  • Empty/error/success обязательны. На экране «комната» — empty («вы ещё ничего не выбрали»). На экране «смета» — success при сохранении.
  • Тестируйте на будущих заказчиках или знакомых, у кого был ремонт.
  • The graph is non-linear — user can return to “room” and tweak materials at any time. Graph spec is critical.
  • Empty/error/success required. “Room” screen — empty (“you haven’t picked yet”). “Estimate” — success on save.
  • Test on prospective clients or friends who’ve done renovations.

Личный инвест-кабинетPersonal Investment Office

Цель: 5-экранный мобильный прототип кабинета: вход → дашборд → детали позиции → прогноз → экспорт. Продолжение проекта 06 основного курса в дизайн-форме.

Goal: a 5-screen mobile prototype of the office: log-in → dashboard → position details → forecast → export. A continuation of main-course etape 06 in design form.

  • Empty-state дашборда («ещё нет позиций — давайте добавим первую») критичен — это onboarding-точка.
  • Loading-state на «детали позиции» (тянем котировку) и «прогноз» (считаем сценарии).

  • Тестируйте на 3 живых людях, у кого есть инвест-портфель.
  • Dashboard empty state (“no positions yet — add the first”) is critical — an onboarding moment.
  • Loading on “position details” (pulling quote) and “forecast” (computing scenarios).
  • Error “broker token expired” — allows recovery testing without sign-in.

§ V · Словарь этапаEtape glossary

Новые словаNew words

Прототип vs mockupPrototype vs mockup
Mockup — статичный экран, отвечает «как выглядит». Прототип — набор связанных экранов с переходами, отвечает «как работает». Mockup тестирует стиль, прототип — пользовательские решения. A mockup is one static screen, answers “how it looks.” A prototype is connected screens with transitions, answers “how it works.” Mockup tests style; prototype tests user decisions.
Граф навигацииNavigation graph
Карта прототипа: узлы (экраны), рёбра (переходы), описание действий пользователя на каждом ребре. Составляется ДО дизайна экранов — иначе прототип получается «каждый экран сам по себе». The prototype map: nodes (screens), edges (transitions), user-action notes per edge. Drafted BEFORE designing screens, otherwise the prototype reads as “each screen on its own.”
Состояния экранаScreen states
Пять каноничных: default, empty, loading, error, success. Прототип только с default тестирует только идеальный сценарий. Реальные пользователи попадают в остальные. The canonical five: default, empty, loading, error, success. A default-only prototype tests only the happy path. Real users land elsewhere.
Empty-state как onboardingEmpty state as onboarding
Хороший пустой экран — не «ничего нет», а призыв к первому действию: пример, кнопка «давайте добавим», ссылка на помощь. Это первый момент, где новичок учится продукту. A good empty screen is not “nothing here,” it’s a CTA for the first action: an example, “let’s add one” button, help link. The first moment a newcomer learns the product.
Think-aloud протоколThink-aloud protocol
Канонический способ user-testing: пользователь решает задачу и проговаривает вслух мысли. «Вижу кнопку, нажимаю». Запутанность — это и есть данные. The canonical user-testing method: the user solves a task and speaks thoughts aloud. “I see a button, I tap.” Confusion is data.
Правило 5 пользователейRule of 5 users
Эмпирическое (Якоб Нильсен): 5 живых тестировщиков ловят 80% проблем юзабилити. Не нужно 50, не нужно фокус-группы. Если 3+ человека спотыкаются на одном — точно надо чинить. Empirical (Nielsen): 5 real testers catch 80% of usability problems. No need for 50, no focus groups. If 3+ trip on the same point — fix it.
§ VI · Quiz

Семь вопросовSeven questions

  1. Чем прототип отличается от mockup?Prototype vs mockup?

    show

    Mockup — один статичный экран, отвечает «как выглядит». Прототип — связанные экраны с переходами и состояниями, отвечает «как работает». Mockup тестирует стиль, прототип — пользовательские решения.

    Mockup is one static screen, “how it looks.” Prototype is connected screens with transitions and states, “how it works.” Mockup tests style; prototype tests user decisions.

  2. Зачем составлять граф ДО экранов?Why graph before screens?

    show

    Если рисовать экраны без графа, каждый получается «сам по себе». Граф фиксирует, как они связаны, какие действия запускают переходы, где альтернативные пути и тупики. Несколько минут на граф экономят 30 минут на «ой, забыли back-кнопку».

    Skipping the graph leaves each screen disconnected. The graph fixes how they relate, which actions trigger transitions, where alt paths and dead-ends sit. Minutes on graph save 30 min on “oh, forgot back.”

  3. Какие пять состояний у экрана?Five screen states?

    show

    Default (всё в порядке), empty (нет данных), loading (тянутся), error (что-то не так), success (сделано). Прототип только с default тестирует только happy path.

    Default, empty, loading, error, success. Default-only tests only the happy path.

  4. Что должен показывать хороший empty-state?What should a good empty state show?

    show

    Не «ничего нет», а призыв к первому действию: пример, кнопка «давайте добавим», ссылка на помощь. Это первый момент onboarding для новичка.

    Not “nothing here,” but a CTA for the first action: example, “let’s add one” button, help link. The first onboarding moment.

  5. Что говорит «правило 5 пользователей»?Rule of 5 users?

    show

    5 живых тестировщиков ловят 80% проблем юзабилити (Якоб Нильсен). Не нужно 50. Если 3+ человека спотыкаются на одном — обязательно чинить.

    5 real testers catch 80% of usability issues (Nielsen). No need for 50. 3+ people tripping on the same thing — must fix.

  6. Что такое think-aloud?Think-aloud?

    show

    Протокол user-testing: тестировщик решает задачу и проговаривает мысли вслух. «Вижу кнопку, нажимаю». Запутанность — это и есть данные. Не помогайте, наблюдайте.

    A user-testing protocol: the tester solves a task while speaking thoughts aloud. “I see a button, I tap.” Confusion is data. Don’t help; observe.

  7. Когда mockup, когда прототип?When mockup, when prototype?

    show

    Mockup — на ранних этапах для утверждения стиля, до того, как обсуждать поведение. Прототип — когда нужно проверить пользовательские решения: понятна ли иерархия, найдут ли путь, не запутаются ли в состояниях.

    Mockup early to lock style, before behaviour. Prototype to verify user decisions: is the hierarchy clear, will they find the path, do states confuse them.
