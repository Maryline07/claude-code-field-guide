Анатомия настоящего веб-приложенияAnatomy of a real web application

Frontend, backend и БД — «зал, кухня и кладовая»Frontend, backend and DB — "the dining room, the kitchen, the pantry"

Метафора, которую полезно держать в голове весь этап — ресторан:

  • Frontend = зал. Что видит и щупает посетитель: меню, столики, оформление. В нашем случае — страницы, кнопки, формы, графики. Живёт в браузере пользователя.
  • Backend = кухня. Готовит блюда. Принимает заказы из зала, лезет на склад, отдаёт еду. В нашем случае — сервер, который принимает запросы со страниц, читает базу, считает, отправляет результат.
  • База = кладовая. Тут лежат продукты. В нашем случае — пользователи, их операции, настройки. Кладовую видит только кухня, гости туда не ходят.

Эта метафора важна, потому что:

  • Frontend и backend — это две разные программы. Они часто живут в двух разных папках, на двух разных серверах, и общаются между собой через API.
  • Секреты (ключи, пароли) живут только в backend. Frontend их не видит — иначе любой гость может украсть их через DevTools.
  • База — отдельный сервис, к которому подключается backend. Управлять ей можно отдельно: бэкапить, мигрировать, переключать.

The metaphor to keep in mind through this etape — a restaurant:

  • Frontend = the dining room. What guests see and touch: menu, tables, decor. For us — pages, buttons, forms, charts. Lives in the user’s browser.
  • Backend = the kitchen. Cooks dishes. Takes orders from the dining room, hits the pantry, sends food out. For us — a server that takes page requests, reads the database, computes, returns results.
  • Database = the pantry. Stocks live here. For us — users, their operations, settings. The pantry is seen only by the kitchen, never by guests.

The metaphor matters because:

  • Frontend and backend are two different programs. Often two different folders, two different servers, talking via an API.
  • Secrets (keys, passwords) live only in the backend. The frontend never sees them — otherwise any guest can steal through DevTools.
  • The database is a separate service the backend connects to. You manage it independently: back up, migrate, switch.
Backend на FastAPI: «кухня» на PythonBackend on FastAPI: a Python "kitchen"

Для backend в курсе используем FastAPI — современный python-фреймворк, который Claude хорошо знает и собирает за минуты. Минимум кода — и у вас работающий сервер с документацией.

Сервер на FastAPI — это набор «эндпоинтов». Пример:

# читать список операцийread the operations list
GET  /api/operations         → список JSON

# добавить операциюadd an operation
POST /api/operations         → принимает JSON, возвращает созданное

# войти в аккаунтsign in
POST /api/login              → принимает email + password, возвращает токен

Каждый эндпоинт — это функция в Python, которую пишет Claude. Вы её не трогаете, но понимаете что она делает.

FastAPI бесплатно даёт автоматическую документацию по /docs — вы можете тыкать кнопками и тестировать сервер без написания клиента.

For the backend we use FastAPI — a modern Python framework Claude knows well and assembles in minutes. Minimal code — a working server with docs.

A FastAPI server is a set of endpoints. Example:

# list operations
GET  /api/operations         → JSON list

# add an operation
POST /api/operations         → takes JSON, returns the created one

# sign in
POST /api/login              → takes email + password, returns a token

Each endpoint is a Python function Claude writes. You don’t touch it — but you understand what it does.

FastAPI gives free auto-docs at /docs — you can click buttons and test the server without writing a client.

База данных: SQLite в разработке, PostgreSQL в интернетеDatabase: SQLite in dev, PostgreSQL in production

SQLite — это «база в файле». Один файл app.db — и там вся ваша информация. Идеальна для разработки и учебных проектов: нечего настраивать, всё работает.

PostgreSQL — это «настоящая» база. Отдельный процесс, к которому подключается ваш backend. Поддерживает множество пользователей одновременно. Берём её на Neon — бесплатно, без карты, проект создаётся в один клик.

Стратегия в курсе:

  • На вашем компьютере (разработка) — SQLite. Просто файл.
  • В интернете — PostgreSQL на Neon.
  • Между ними переключаемся одной строкой в .env: DATABASE_URL=sqlite:///./app.db локально и DATABASE_URL=postgresql://... (строка от Neon) в production. Код один и тот же — меняется только переменная окружения.

Базой управляет ORM (например, SQLAlchemy) — библиотека, которая превращает Python-объекты в запросы SQL. Вам не нужно знать SQL: вы говорите Claude «создай модель User с email, password_hash, created_at», и он делает всё остальное.

SQLite is a "database in a file." One app.db file — all your info inside. Ideal for development and learning projects: nothing to set up.

PostgreSQL is a "real" database. A separate process the backend connects to. Handles many concurrent users. We take it on Neon — free, no card, a project in one click.

Strategy in this course:

  • On your machine (dev) — SQLite. Just a file.
  • On the internet — PostgreSQL on Neon.
  • Switch by one line in .env: DATABASE_URL=sqlite:///./app.db locally and DATABASE_URL=postgresql://... (the Neon string) in production. Same code — only the env var changes.

The database is managed by an ORM (e.g. SQLAlchemy) — a library that turns Python objects into SQL. You don’t need SQL: you say to Claude "create a User model with email, password_hash, created_at," and it does the rest.

Авторизация: регистрация, вход, защитаAuth: sign-up, sign-in, protection

Чтобы у каждого пользователя были свои данные, нужна авторизация. Минимум:

  1. Регистрация: email + пароль → создаётся запись в таблице users. Пароль хешируется (превращается в нечитаемую строку) — в базе никогда не лежит «как есть».
  2. Вход: email + пароль → сервер проверяет хеш, выдаёт JWT-токен.
  3. Защита эндпоинтов: когда браузер делает запрос к /api/operations, он шлёт токен. Сервер декодирует токен, узнаёт, какой это пользователь, и возвращает только его данные.
  4. Выход: браузер просто стирает токен у себя.

Что Claude делает за вас:

  • пишет регистрацию и вход;
  • хеширует пароли через bcrypt или argon2;
  • выпускает и проверяет JWT;
  • защищает эндпоинты «декоратором» require_auth;
  • добавляет «забыл пароль», 2FA, OAuth — если попросите.

Что важно понимать вам:

  • пароли в базе — только в виде хеша. Если кто-то украдёт базу, ваши пароли всё равно не видны;
  • JWT-секрет (ключ для подписи токенов) живёт только в backend, в .env;
  • HTTPS обязателен для любых страниц с вводом пароля — и Vercel/Render включают его автоматически.

For each user to have their own data, you need auth. Minimum:

  1. Sign-up: email + password → a row in the users table. Password is hashed (turned into an unreadable string) — never stored "as is."
  2. Sign-in: email + password → the server checks the hash, issues a JWT token.
  3. Protected endpoints: when the browser hits /api/operations, it sends the token. The server decodes it, learns the user, and returns only their data.
  4. Sign-out: the browser just wipes the token.

What Claude does for you:

  • writes sign-up and sign-in;
  • hashes passwords via bcrypt or argon2;
  • issues and verifies JWTs;
  • protects endpoints with a require_auth decorator;
  • adds "forgot password," 2FA, OAuth on request.

What you must understand:

  • passwords in the database — hash only. If the DB leaks, passwords are still not readable;
  • the JWT secret (token signing key) lives only in the backend, in .env;
  • HTTPS is required on any page with password input — Vercel/Render enable it automatically.
Деплой одной кнопкой: Vercel + Render + NeonOne-click deploy: Vercel + Render + Neon

«Деплой» больше не значит «арендовать сервер, настраивать nginx, ставить SSL». Современный подход — три бесплатных сервиса, каждый делает свою часть:

  1. Связали аккаунт Vercel с вашим GitHub.
  2. В Vercel «Import Project» — выбираете репозиторий с frontend. Vercel сам собирает и публикует. Через ~30 секунд — живая ссылка my-app.vercel.app.
  3. В Neon создаёте проект — получаете строку подключения PostgreSQL.
  4. В Render: «New Web Service» → ваш backend-репозиторий.
  5. В настройках Render-сервиса добавляете environment variables: DATABASE_URL (строка от Neon), ANTHROPIC_API_KEY и т. п.
  6. Каждый git push — новый деплой. На каждый PR — своя preview-ссылка для ревью.

Что важно:

  • Почему три сервиса, а не один. Vercel хорош для frontend, но не умеет backend. У Render бесплатный веб-сервис есть, а бесплатная база удаляется через 30 дней — для курса не годится. Neon даёт постоянную бесплатную базу. Каждый инструмент — на своём месте.
  • Никогда не кладите секреты в код. Только Environment Variables через интерфейс платформы.
  • Настройте CORS — чтобы ваш Vercel-frontend имел право обращаться к Render-backend. Claude настроит, но знать про это надо.
  • Render-backend засыпает после 15 минут неактивности — первый запрос после сна занимает ~1 минуту. Это нормально для учебных проектов.

"Deploy" no longer means "rent a server, configure nginx, install SSL." The modern flow — three free services, each doing its part:

  1. Link Vercel to your GitHub.
  2. In Vercel "Import Project" — pick the frontend repo. Vercel builds and publishes. ~30 seconds later — a live URL my-app.vercel.app.
  3. In Neon create a project — you get a PostgreSQL connection string.
  4. In Render: "New Web Service" → your backend repo.
  5. In the Render service settings add environment variables: DATABASE_URL (the Neon string), ANTHROPIC_API_KEY, and so on.
  6. Every git push — a new deploy. Each PR gets its own preview URL for review.

What matters:

  • Why three services, not one. Vercel is great for the frontend but can’t do a backend. Render has a free web service, but its free database is deleted after 30 days — no good for a course. Neon gives a permanent free database. Each tool in its place.
  • Never put secrets in code. Only environment variables via the platform UI.
  • Configure CORS so your Vercel frontend can call your Render backend. Claude sets it, but you should know it’s a thing.
  • The Render backend sleeps after 15 min of inactivity — first request after sleep takes ~1 minute. Fine for learning projects.
Claude API в своём приложении: модели и prompt cachingClaude API in your app: models and prompt caching

Три модели, между которыми выбираем для разных задач:

  • Opus — самая умная, самая дорогая. Для сложных аналитических задач, где качество важнее всего (например, финансовая аналитика капстоуна).
  • Sonnet — «золотая середина». 80% качества Opus за 20% денег. Большинство приложений — на ней.
  • Haiku — быстрая и дешёвая. Для простых классификаций («это что — еда или транспорт?») и высоких объёмов.

В коде это одна строка:

model="claude-sonnet-4-6"   # для типичных задачfor typical tasks
model="claude-opus-4-7"     # для сложного анализаfor complex analysis

Prompt caching экономит деньги и время:

  • Если в каждом запросе вы шлёте один и тот же «контекст» (например, описание AI-консультанта, схему ваших данных, last 30 операций пользователя) — Claude API кэширует эту часть и на повторных запросах берёт за неё в 10 раз меньше.

  • Для нашего AI-консультанта это критично: пользователь шлёт по 15 вопросов подряд, кэшируется системный промпт + его портфель.

Three models to choose between for different jobs:

  • Opus — smartest, priciest. For hard analytic work where quality is paramount (e.g., capstone financial analytics).
  • Sonnet — the "sweet spot." 80% of Opus quality at 20% of the cost. Most apps run here.
  • Haiku — fast and cheap. For simple classifications ("food or transport?") and high volumes.

  • The cache lives 5 minutes. If requests come more often — you’re almost always cache-hit.
  • Critical for our AI advisor: a user fires 15 questions in a row, the system prompt + their portfolio get cached.

На этом этапе впервые «настоящая» сложность. Не всё работает с первого раза. Что чаще всего ломается:

This is the first etape with "real" complexity. Not everything works first try. The usual culprits:

  • Secrets didn’t load. Works locally, breaks on Vercel/Render. Confirm env vars are set in the platform dashboard (not just in your local .env).
  • Database migrations. On first deploy the tables are not there yet. Claude usually handles this, but sometimes ask explicitly: "add a startup hook that creates tables on server boot."

§ V · Словарь этапаEtape glossary

Frontend
«Зал ресторана» — часть приложения, которая работает в браузере пользователя. То, что человек видит, кликает, заполняет. "The dining room" — the part of the app running in the user’s browser. What the person sees, clicks, fills in.
Backend
«Кухня» — программа на сервере, принимающая запросы от frontend, обращающаяся к базе и возвращающая данные. Видит секреты и логику. Пользователь его не видит. "The kitchen" — a program on the server taking frontend requests, hitting the DB, returning data. Sees secrets and logic. The user never sees it.
База данныхDatabase
«Кладовая» — постоянное хранилище данных приложения: пользователи, их операции, настройки. Доступна только backend. "The pantry" — persistent storage for app data: users, ops, settings. Reachable only by the backend.
SQLite
База данных «в одном файле». Идеальна для разработки. Один файл app.db — вся ваша информация. A "database in one file." Ideal for dev. One app.db — all your info.
PostgreSQL
Полноценная база как отдельный сервис. Для production. Берём её на Neon — создаётся в один клик. A full database as its own service. For production. We take it on Neon — created in one click.
FastAPI
Современный python-фреймворк для backend. Минимум кода, бесплатная авто-документация по /docs. Используем как «кухню». A modern Python framework for backend. Minimal code, free auto-docs at /docs. We use it as the "kitchen."
ЭндпоинтEndpoint
Одна функция backend, доступная по URL. Например, POST /api/login — «эндпоинт логина». Имеет HTTP-метод и формат данных. One backend function reachable via a URL. E.g., POST /api/login — the "login endpoint." Has an HTTP method and a data format.
JWT-токенJWT token
JSON Web Token — короткая строка, которая «доказывает» серверу, что вы вошли. Хранится в браузере, шлётся на каждый запрос. Подписан секретом backend. JSON Web Token — a short string proving to the server you’re signed in. Stored in the browser, sent on every request. Signed by the backend secret.
CORS
Cross-Origin Resource Sharing — защита браузера: страница с одного домена по умолчанию не может звонить серверу с другого. Настраивается в backend, чтобы Vercel-frontend мог обращаться к Render-backend. Cross-Origin Resource Sharing — a browser safety: a page on one domain can’t call a server on another by default. Configured in the backend so the Vercel frontend can reach the Render backend.
ДеплойDeploy
Процесс «отправить ваше приложение на сервер в интернете». Современный подход — в один git push через Vercel (frontend) и Render (backend). The process of "sending your app to a server on the internet." Modern way — one git push via Vercel (frontend) and Render (backend).
Vercel
Платформа для деплоя frontend. Подключаете GitHub-репо — каждый push автоматически публикует новую версию. Бесплатный тариф щедрый. A frontend-deploy platform. Connect a GitHub repo — each push publishes a new version. Generous free tier.
Render
Платформа для деплоя backend-серверов. Бесплатный веб-сервис — 750 ч/мес; засыпает после 15 минут простоя. Может попросить карту для разовой верификации ($1, возвращается). A backend-server deploy platform. Free web service — 750 hrs/month; sleeps after 15 min idle. May ask for a card for a one-off verification ($1, refunded).
Neon
Бесплатный PostgreSQL как отдельный сервис. Без кредитной карты, тариф не истекает, 0.5 ГБ на проект. Сюда подключается backend через DATABASE_URL. Free PostgreSQL as its own service. No credit card, the plan never expires, 0.5 GB per project. The backend connects here via DATABASE_URL.
Claude API
Программный интерфейс к моделям Claude от Anthropic. Через него ваше приложение само вызывает Claude (а не только вы через Claude Code). Ключ в ANTHROPIC_API_KEY. A programmatic interface to Anthropic’s Claude models. Through it your app calls Claude (not just you via Claude Code). The key is ANTHROPIC_API_KEY.
Prompt caching
Кэширование одинаковых частей промпта между запросами. Если в начале каждого запроса одно и то же, Claude API кэширует и берёт за это в 10 раз меньше денег и времени. Кэш живёт 5 минут. Three models of different power and price. Opus — smartest, for hard analytics. Sonnet — the sweet spot, most tasks. Haiku — fast and cheap, for classifications.
§ VI · Quiz

Семь вопросов на закреплениеSeven questions for reinforcement

  1. Frontend, backend, БД — в метафоре «ресторан» что чему соответствует?In the "restaurant" metaphor, what maps to frontend, backend, DB?

    показать ответshow answer

    Frontend = зал (что видит и щупает гость). Backend = кухня (готовит блюда, лезет на склад). БД = кладовая (там лежат продукты, видит её только кухня).

    Frontend = dining room (what guests see and touch). Backend = kitchen (cooks, hits the pantry). DB = pantry (stocks live here, only the kitchen sees it).

  2. Почему пароли в базе хранятся хешированными, а не «как есть»?Why are passwords stored hashed, not as-is?

    показать ответshow answer

    Если кто-то украдёт базу — ваши пароли всё равно не видны. Хеш — необратимая «мясорубка»: из пароля считается строка, а из строки пароль обратно не вытащить. При входе сравнивается хеш введённого пароля с хешем в базе.

    If the DB leaks — passwords still aren’t readable. A hash is an irreversible "meat grinder": a password yields a string, but you can’t recover the password from the string. On sign-in, the hash of the typed password is compared to the stored one.

  3. Что такое CORS-ошибка и где её чинят?What is a CORS error and where do you fix it?

    показать ответshow answer

    Браузер блокирует запросы от страницы на одном домене (Vercel) к серверу на другом (Render), если backend явно не разрешил это. Чинится в backend: добавляется CORS middleware с разрешённым origin (вашим Vercel-доменом).

    The browser blocks requests from a page on one domain (Vercel) to a server on another (Render) unless the backend explicitly allows it. Fixed in the backend: add CORS middleware with the allowed origin (your Vercel domain).

  4. Почему ANTHROPIC_API_KEY никогда нельзя класть во frontend?Why must ANTHROPIC_API_KEY never live in the frontend?

    показать ответshow answer

    Frontend живёт в браузере пользователя. Любой откроет DevTools и увидит ваш ключ. Через минуту его украдут — и будут жечь ваши деньги через ваш аккаунт. Ключ должен быть только в backend, в env vars Render. Frontend обращается к вашему backend-у, backend — к Anthropic.

    Frontend lives in the user’s browser. Anyone opens DevTools and sees your key. A minute later it’s stolen — and your money burns through your account. The key must live only in the backend, in Render env vars. Frontend calls your backend, backend calls Anthropic.

  5. Какую модель Claude взять для «классификация транзакции» и какую — для «годовой финансовый прогноз с аргументацией»?Which Claude model for "classify a transaction" vs. "annual financial forecast with reasoning"?

    показать ответshow answer

    Классификация транзакций — Haiku (быстрая, дешёвая, простая задача). Годовой прогноз с аргументацией — Opus (сложно, нужна качественная аргументация). Большинство «средних» задач — на Sonnet.

    Transaction classification — Haiku (fast, cheap, simple). Annual forecast with reasoning — Opus (complex, needs strong arguments). Most "middle" tasks — Sonnet.

  6. Зачем prompt caching в AI-консультанте?Why prompt caching in the AI advisor?

    показать ответshow answer

    В каждом запросе мы шлём один и тот же системный промпт + портфель пользователя (десятки строк). Без кэша Claude API обрабатывает их каждый раз заново — долго и дорого. С кэшем эти части обрабатываются один раз, дальше используются «как есть» — в 10 раз быстрее и дешевле. Особенно важно, когда пользователь шлёт серию вопросов подряд.

    Each request sends the same system prompt + the user’s portfolio (dozens of lines). Without cache Claude API processes them anew each time — slow and expensive. With cache those parts process once, then reuse — 10× faster and cheaper. Especially when the user fires a series of questions.

  7. Что лучше для разработки и что для production: SQLite или PostgreSQL?SQLite or PostgreSQL — which fits dev, which fits production?

    показать ответshow answer

    SQLite — для разработки и учебных проектов: один файл, ничего настраивать. PostgreSQL — для production, особенно когда несколько пользователей одновременно. Хорошая практика — код один и тот же, разница только в DATABASE_URL в .env: sqlite:///./app.db локально, postgresql://... (строка от Neon) в production.

    SQLite — for dev and learning: one file, no setup. PostgreSQL — for production, especially with concurrent users. Good practice — same code, only DATABASE_URL differs in .env: sqlite:///./app.db locally, postgresql://... (the Neon string) in production.
