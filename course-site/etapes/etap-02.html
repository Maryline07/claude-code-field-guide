Папка проекта как «рабочее пространство»A project folder as a “workspace”

До этого этапа мы работали так: «одна папка — одна задача» (карточка обо мне, калькулятор процента и т. д.). Теперь — «одна папка — многоэтапный проект, в который мы возвращаемся неделями».

Что это даёт:

  • Claude видит все ваши файлы сразу и может между ними переключаться;
  • правила проекта пишутся один раз — больше не нужно повторять «всегда категоризируй по X» в каждой сессии;
  • проект можно отдать Claude «свежему» (новая сессия) — и он через 30 секунд снова в контексте.

Главный инструмент для этого — файл CLAUDE.md в корне проекта. О нём — следующая тема.

Up to this etape we worked “one folder — one task” (about-me card, compound calculator, etc.). Now — “one folder — a multi-step project we return to over weeks.”

This gives:

  • Claude sees all your files at once and can switch between them;
  • project rules are written once — no need to repeat “always categorise by X” every session;
  • you can hand the project to a “fresh” Claude (new session) — and within 30 seconds it’s back in context.

The key tool for this is the CLAUDE.md file in the project root. Next topic.

CLAUDE.md: правила, которые Claude помнит самCLAUDE.md: rules Claude reads on its own

CLAUDE.md — это обычный markdown-файл, который вы кладёте в корень папки проекта. Claude читает его автоматически в начале каждой новой сессии в этой папке.

Что туда писать:

  • Стиль и тон: «комментарии на русском», «короткие имена переменных», «никаких лишних emoji».
  • Структура проекта: где у нас данные, где код, где отчёты.
  • Правила работы: «всегда добавляй дату в ISO», «не коммить файлы из data/».
  • Тематические допущения: «валюта по умолчанию — рубль», «ставка налога — 13%».
  • Что не делать: «не используй сторонние API без спроса», «не удаляй файлы старше 30 дней без подтверждения».

Главное правило: пишите коротко и только то, чего нельзя угадать из кода. Не пишите туда историю проекта, биографию или то, что и так очевидно.

Пример хорошего CLAUDE.md для инвест-проекта:

# Investing Journal — правила проекта

## Стек
- одностраничные html, без сборщиков
- chart.js через CDN, никаких других зависимостей

## Данные
- источник: data/operations.csv (date, ticker, side, qty, price, currency)
- валюта по умолчанию — USD
- даты в ISO: YYYY-MM-DD

## Стиль
- editorial / paper-and-ink, тёплая палитра (как сайт курса)
- засечковый шрифт, IBM Plex Mono для цифр

## Правила
- НЕ добавляй emoji в код
- комментарии на английском, тексты UI на русском
- перед удалением файлов из data/ — спрашивай

CLAUDE.md is a plain markdown file at the project root. Claude reads it automatically at the start of every new session in that folder.

What to put inside:

  • Style and tone: “comments in English,” “short variable names,” “no decorative emoji.”
  • Project layout: where data is, where code is, where reports go.
  • Working rules: “always add ISO dates,” “never commit files from data/.”
  • Domain assumptions: “default currency USD,” “tax rate 13%.”
  • What not to do: “don’t use third-party APIs without asking,” “don’t delete files older than 30 days without confirmation.”

Key rule: keep it short and only put things that can’t be guessed from the code. Don’t add history, biography, or the obvious.

Example of a good CLAUDE.md for an investing project:

# Investing Journal — project rules

## Stack
- single-page html, no bundlers
- chart.js via CDN, no other deps

## Data
- source: data/operations.csv (date, ticker, side, qty, price, currency)
- default currency — USD
- ISO dates: YYYY-MM-DD

## Style
- editorial / paper-and-ink, warm palette
- serif type, IBM Plex Mono for numerals

## Rules
- NO emoji in code
- comments in English, UI text in Russian
- ask before deleting from data/
@-ссылки: указать Claude конкретный файл@-mentions: pointing Claude to a specific file

Если в вашем проекте десятки файлов, Claude может тратить время на их «сканирование» при каждой задаче. Когда вы уже знаете, какой именно файл нужен, проще указать его напрямую через @:

> посмотри @data/operations.csv и собери short-отчёт по тикерам.
> используй формат, описанный в @docs/report-template.md.

Что Claude делает:

  • Прочитает указанные файлы сразу, без сканирования папки;
  • Не будет «угадывать», какие ещё файлы могут пригодиться;
  • В ответе явно укажет, что использовал именно эти.

Удобно для:

  • длинных проектов с десятками файлов;
  • задач, где важна точность («только этот CSV, не трогай остальные»);
  • повторяющихся задач — быстрее, чем заново описывать структуру.

If your project has dozens of files, Claude may waste time “scanning” them every task. When you already know which file matters, point at it directly with @:

> look at @data/operations.csv and produce a per-ticker short report.
> use the layout described in @docs/report-template.md.

What Claude does:

  • reads those files right away, no folder-scan;
  • doesn’t “guess” which other files might matter;
  • states which files it used in the reply.

Handy for:

  • long projects with many files;
  • tasks where precision matters (“only this CSV, leave the rest”);
  • repeating tasks — faster than re-explaining structure.
Auto-memory: что Claude помнит между сессиямиAuto-memory: what Claude remembers across sessions

CLAUDE.md живёт в проекте — его читают только когда работают именно с этой папкой. Auto-memory — глобальная: всё, что Claude в ней запомнил, доступно в любой папке, любой сессии.

Файлы лежат в ~/.claude/memory/. Главный из них — MEMORY.md, индекс. Туда Claude сам пишет, когда узнаёт что-то полезное на будущее.

Четыре типа записей:

  1. user — о вас: роль, опыт, предпочтения. «Пользователь — финансовый аналитик, любит таблицы, не любит длинные комментарии».
  2. feedback — ваши указания, как работать или как НЕ работать. «Не используй mock-данные для интеграционных тестов — в прошлый раз пропустили баг».
  3. project — контекст текущих работ: дедлайны, инициативы, причины решений. «Переход с старого API — до 1 июня, по требованию compliance».
  4. reference — «где искать»: ссылки на Linear, Jira, Slack, дашборды. «Пайплайн-баги — в Linear-проекте INGEST».

Полезные команды:

# просмотреть, что Claude помнит
> покажи мою auto-memory

# попросить запомнить
> запомни: я предпочитаю tabular layout для финансовых отчётов

# попросить забыть
> забудь, что я говорил про предпочтение dark theme

Важное: Claude сам решает, когда писать в auto-memory. Но вы можете прямо попросить.

CLAUDE.md lives in the project — only read inside that folder. Auto-memory is global: anything Claude saved there is available in any folder, any session.

Files live in ~/.claude/memory/. The main one is MEMORY.md, the index. Claude writes there on its own when it learns something useful for the future.

Four record types:

  1. user — about you: role, experience, preferences. “User is a financial analyst, likes tables, dislikes long comments.”
  2. feedback — your guidance on how to work or NOT work. “Don’t use mock data in integration tests — last time it hid a bug.”
  3. project — current-work context: deadlines, initiatives, reasons. “Old-API migration — by June 1st, per compliance.”
  4. reference — “where to look”: pointers to Linear, Jira, Slack, dashboards. “Pipeline bugs — in Linear project INGEST.”

Useful commands:

# show what Claude remembers
> show my auto-memory

# ask to remember
> remember: I prefer a tabular layout for financial reports

# ask to forget
> forget what I said about preferring dark theme

Important: Claude decides when to write to auto-memory. But you can ask explicitly.

Чтение существующего проекта: «расскажи, что тут происходит»Reading an existing project: “tell me what’s going on here”

Когда вы открываете чужой проект (или свой, в который не заглядывали полгода), не нужно «вчитываться вручную». Запустите Claude и попросите ввести в курс:

> расскажи, что это за проект:
> — что он делает (одна-две фразы),
> — какая структура папок,
> — какие главные файлы,
> — что нетипично, на что мне стоит обратить внимание.

Claude пройдётся по файлам и даст вам краткий «бриф». Это особенно полезно перед тем, как:

  • добавлять новую функцию в старый проект;
  • возвращаться к курсу после паузы;
  • взять чужой код (например, скачали с GitHub) и хотите понять, что там.

When you open someone else’s project (or your own, untouched for half a year), do not “read manually.” Launch Claude and ask for an intro:

> tell me about this project:
> — what it does (one or two sentences),
> — folder structure,
> — the key files,
> — anything unusual I should pay attention to.

Claude walks the files and gives you a short brief. Useful right before:

  • adding a new feature to an old project;
  • returning to the course after a break;
  • opening third-party code (e.g. cloned from GitHub) and wanting to know what’s inside.
Multi-file edits: правки в нескольких файлах сразуMulti-file edits: changes across several files at once

В многофайловом проекте часто одно изменение влияет на несколько файлов: переименовали колонку в CSV — нужно поправить и парсер, и дашборд, и документацию.

Claude умеет делать это одной просьбой:

> в operations.csv поле «side» переименовалось в «direction».
> найди все упоминания «side» во всех файлах проекта и обнови.
> покажи список изменений перед применением.

Чем такой подход лучше «по файлу за раз»:

  • Claude видит все связи — не забывает обновить какой-то файл;
  • можно потребовать «покажи план перед применением» — и сначала посмотреть список;
  • откат проще: «верни всё, что ты только что изменил».
Хорошая привычка: для крупных правок ставить plan mode (этап 00) или просить «сначала покажи список изменений, я утвержу — потом применяй». Good habit: for large edits switch to plan mode (etape 00) or ask “show the change list first, I’ll approve, then apply.”

In a multi-file project one change often touches many files: rename a column in a CSV — and you need to update the parser, the dashboard, and the docs.

Claude does this in one request:

> in operations.csv the field "side" became "direction".
> find every mention of "side" across all files and update.
> show the change list before applying.

Why this beats one-file-at-a-time:

  • Claude sees all the connections — won’t forget a file;
  • you can demand “show the plan before applying” — review the list first;
  • rollback is simpler: “undo everything you just changed.”

Проект A · по инструкцииProject A · guided

Дневник инвестиционных операцийInvesting operations journal

Цель: многофайловый проект с CLAUDE.md. Хранилище — CSV. Веб-страница для добавления операций (покупка/продажа/дивиденд) и их просмотра с фильтрами.

Goal: a multi-file project with CLAUDE.md. Storage — CSV. A web page to add operations (buy / sell / dividend) and view them with filters.

Шаги

Steps

  1. Создайте папку portfolio-journal в claude-course, запустите там claude.
  2. Сначала — CLAUDE.md: 
    > составь CLAUDE.md для этого проекта:
>   - стек: одна html-страница + один csv-файл рядом, без бэкенда
>   - данные: data/operations.csv с колонками
>     date, ticker, side (buy/sell/dividend), qty, price, currency, note
>   - стиль: editorial, тёплая палитра, IBM Plex Mono для цифр
>   - правила: даты ISO, currency по умолчанию USD, без emoji
  3. Сгенерируйте тестовые данные (если нет своих): 
    > создай data/operations.csv с 30 реалистичными операциями
> за последние 12 месяцев. шесть тикеров (AAPL, MSFT, GOOGL,
> NVDA, BRK.B, VOO), buy/sell/dividend смешаны.
  4. Сделайте страницу-просмотрщик: 
    > сделай index.html — страница «дневник операций».
> читает data/operations.csv (через fetch), показывает таблицу
> с сортировкой по дате (новое сверху).
> фильтры сверху: тикер (выпадашка), тип операции, период.
> форма «добавить» внизу — сохранение в localStorage,

§ V · Словарь этапаEtape glossary

Что добавилось к словарюNew entries in the glossary

Восемь новых терминов — вокруг памяти и многофайловых проектов. Подчёркнуты пунктиром в тексте, наведите/коснитесь — всплывёт справка.

Eight new terms — around memory and multi-file projects. Underlined with dots in the text, hover/tap to see a tooltip.

CLAUDE.md
Markdown-файл в корне папки проекта. Claude автоматически читает его в начале каждой сессии и применяет как «правила проекта»: стек, структуру, стиль, ограничения. Главный способ передать Claude контекст без повторения. A markdown file at the project root. Claude reads it automatically at the start of every session and applies it as “project rules”: stack, layout, style, constraints. The main way to give Claude context without repetition.
@-ссылка@-mention
Точечная ссылка на файл в промпте: @data/operations.csv. Claude читает именно его, не сканируя всю папку. Удобно в больших проектах и для точности. A pointer to a specific file in your prompt: @data/operations.csv. Claude reads exactly that file, no folder scan. Handy in large projects and for precision.
Auto-memoryAuto-memory
Долгосрочная память Claude, общая для всех проектов. Хранится в ~/.claude/memory/. Claude сам решает, что туда записать — но вы можете попросить «запомни…» или «забудь…». Claude’s long-term memory, shared across all projects. Stored in ~/.claude/memory/. Claude decides what goes in — but you can ask “remember…” or “forget….”
Тип памяти (user / feedback / project / reference)Memory type (user / feedback / project / reference)
Четыре категории auto-memory:
user — о вас, ваших предпочтениях.
feedback — ваши указания, как работать.
project — контекст текущих инициатив.
reference — «где искать» (Linear, Slack, дашборды). Four auto-memory categories:
user — about you, your preferences.
feedback — your guidance on how to work.
project — context of current initiatives.
reference — “where to look” (Linear, Slack, dashboards).
Multi-file editMulti-file edit
Изменения в нескольких файлах одной просьбой. Claude видит все связи и обновляет файлы согласованно. Обычно лучше попросить «покажи план перед применением». Changes across several files in one request. Claude sees all the links and updates them consistently. Usually best paired with “show the plan before applying.”
CSVCSV
Comma-separated values — «таблица в текстовом файле». Каждая строка — запись, поля разделены запятой. Открывается в Excel или Google Sheets, но при этом видна «как есть» в любом редакторе. Comma-separated values — a “table in a text file.” One line per row, fields separated by commas. Opens in Excel or Google Sheets, yet readable “as is” in any editor.
ДашбордDashboard
Одна страница с ключевыми показателями и графиками — чтобы охватить состояние «одним взглядом». В этом этапе — дашборд по инвестиционному портфелю. A single page with key metrics and charts — to cover state “at a glance.” On this etape — a portfolio dashboard.
P&L (Profit and Loss)P&L (Profit and Loss)
Прибыли и убытки. Реализованный P&L — по закрытым сделкам (продали с плюсом или минусом). Нереализованный — по открытым позициям (на бумаге). Profit and loss. Realised P&L — on closed trades (sold at a gain or loss). Unrealised — on open positions (paper).
FIFO
First-In-First-Out — «первым купил, первым продал». Стандартный способ закрытия позиций при подсчёте P&L: первая sell-операция «закрывает» самую старую buy. First-In-First-Out — “bought first, sold first.” The standard way to close positions when computing P&L: the first sell “closes” the oldest buy.
