§ I · ТемыTopics

Что мы пройдём по шагамWhat we will walk through

Что Claude Design извлекает из кодбазыWhat Claude Design extracts from a codebase

Дайте Claude Design ссылку на GitHub-репозиторий, прикрепите .zip с кодом или конкретные файлы — и он вытащит дизайн-сигналы автоматически. Что именно он ищет:

  • CSS-переменные в :root блоках (--bg, --ink, --accent...) — это «золотой стандарт» tokens, как в вашем основном курсе.
  • Tailwind-config (tailwind.config.js) с extended themes — цвета, шрифты, размеры.
  • JSON-токены (tokens.json, design-tokens.json) — формат W3C/Style Dictionary, очень популярный сейчас.
  • SCSS/Less variables ($primary, $body-font...).
  • Figma Tokens плагин-экспорт — если у вас есть.

Промпт обычно такой:

> вот наша кодбаза: https://github.com/yourorg/yourrepo
> (или: прикреплён styles.css)
> извлеки tokens: палитру, типографику, размерные шкалы, отступы.
> покажи мне, что нашёл, в виде сводной таблицы.
> не делай дизайн пока — мне нужна сверка.

Claude отвечает: «нашёл 12 цветов (--bg, --ink, --accent...), 3 шрифта (Fraunces, Newsreader, IBM Plex Mono), spacing scale 0.25/0.5/1/2/3 rem, breakpoints 640/1024/1280...». Вы сверяете и при необходимости поправляете. Только после этого — запускаете дизайн в этой системе.

Этот «сверочный шаг» критичен. Если Claude ошибся в палитре или типографике (например, не увидел dark-mode override), все 50 будущих ассетов унаследуют ошибку. Лучше потратить 5 минут на проверку, чем переделывать.

Hand Claude Design a GitHub URL, a .zip, or specific files — and it pulls out design signals automatically. What it looks for:

  • CSS variables in :root blocks (--bg, --ink, --accent...) — the “gold standard” for tokens, as in your main course.
  • Tailwind config (tailwind.config.js) with extended themes — colours, fonts, sizes.
  • JSON tokens (tokens.json, design-tokens.json) — W3C / Style Dictionary format, very common now.
  • SCSS / Less variables ($primary, $body-font...).
  • Figma Tokens plugin export — if you have one.

The prompt usually:

> here is our codebase: https://github.com/yourorg/yourrepo
> (or: styles.css attached)
> extract tokens: palette, typography, size scales, spacing.
> show me what you found as a summary table.
> do not design yet — I want to verify.

Claude replies: “found 12 colours (--bg, --ink, --accent...), 3 fonts (Fraunces, Newsreader, IBM Plex Mono), spacing scale 0.25/0.5/1/2/3 rem, breakpoints 640/1024/1280...” You verify and patch if needed. Only then — run a design in that system.

The “verify step” is critical. If Claude misread the palette or typography (e.g. missed a dark-mode override), all 50 future assets inherit the error. Five minutes of verification beats redoing.
Figma как источник: что вытаскивает Claude из файлаFigma as source: what Claude pulls from the file

Если у вашей команды дизайн уже живёт в Figma, Claude Design умеет читать его напрямую: connect Figma file в интерфейсе или дайте share-URL в промпте. Что он вытащит:

  • Color styles (если они оформлены как shared styles, не просто HEX в слоях).
  • Text styles — пары «название → шрифт+размер+интерлиньяж».
  • Effect styles — тени, размытия.
  • Components — кнопки, карточки, поля ввода. Claude использует их как «строительные блоки» для новых проектов.
  • Variables (Figma 2024+) — современный аналог design tokens, читается напрямую.

Главное условие: Figma-файл должен быть аккуратно собран. Если стили локальные, компоненты разбросаны, цвета захардкожены в слоях — Claude вытащит мусор. Перед загрузкой в Claude Design стоит:

  1. Запустить Figma Variables Migrator или хотя бы вынести цвета в color styles.
  2. Проверить, что компоненты называются осмысленно (не «Frame 1247», а «Button / Primary»).
  3. Удалить устаревшие/дубликаты стилей.

Когда Figma-источник лучше кодбазы:

  • Команда работает «design-first» — дизайн опережает код.
  • В коде токенов нет, или они неполные.
  • Нужны визуальные компоненты — иллюстрации, иконки в одном стиле.

If your team’s design already lives in Figma, Claude Design can read it directly: connect Figma file in the UI or give a share URL in the prompt. What it pulls:

  • Colour styles (if defined as shared styles, not HEXes scattered across layers).
  • Text styles — named “font + size + leading” pairs.
  • Effect styles — shadows, blurs.
  • Components — buttons, cards, inputs. Claude uses these as building blocks for new projects.
  • Variables (Figma 2024+) — modern equivalent of design tokens, read directly.

Main precondition: the Figma file must be tidy. If styles are local, components scattered, colours hardcoded into layers — Claude pulls junk. Before loading into Claude Design:

  1. Run the Figma Variables Migrator or at least move colours into colour styles.
  2. Make sure components are named meaningfully (“Button / Primary,” not “Frame 1247”).
  3. Delete legacy/duplicate styles.

When Figma-source beats the codebase:

  • The team is “design-first” — design leads code.
  • Tokens in code don’t exist or are incomplete.
  • You need visual components — illustrations, one-family icons.
Workspace inheritance: бренд один, проектов — десятьWorkspace inheritance: one brand, ten projects

Самая сильная возможность этапа: загруженный однажды бренд применяется ко всем будущим проектам в этом workspace, без повторной загрузки.

Технически: workspace в Claude Design — это «папка», в которой живут связанные проекты. Загруженная brand-system привязывается к workspace, не к конкретному проекту. Создаёте новый проект в том же workspace — Claude по умолчанию использует tokens этого бренда.

Практически:

  1. Создаёте workspace «Северный край — кофейня». Загружаете tokens (цвета, шрифты, логотип).
  2. В этом workspace делаете 10 проектов: меню, лендинг, постер открытия, серия Instagram, баннер email, оформление чашек, табличка на двери, sticker-pack, OG-image, дек для инвестора.
  3. Все 10 артефактов выглядят как один бренд. Не потому что вы старались, а потому что Claude применил tokens из workspace автоматически.

Когда нужен новый workspace:

  • Новый клиент / новый проект (другой бренд).
  • Те же темы, но принципиально другая стилистика (например, кофейня vs её же чайный филиал, который позиционируется иначе).
  • Эксперимент со стилем без риска перезаписать основной бренд.

The most powerful capability of the etape: a brand uploaded once applies to every future project in that workspace, without re-uploading.

Technically: a workspace in Claude Design is a “folder” where related projects live. The uploaded brand-system attaches to the workspace, not to a specific project. Make a new project in the same workspace — Claude uses the workspace tokens by default.

In practice:

  1. Create workspace “Northern Coast — café.” Upload tokens (colours, fonts, logo).
  2. Inside that workspace ship 10 projects: menu, landing, opening poster, Instagram series, email banner, cup design, door sign, sticker pack, og-image, investor deck.
  3. All 10 look like one brand. Not because you tried, but because Claude applied the workspace tokens automatically.

When you need a new workspace:

  • New client / new project (different brand).
  • Same theme but a fundamentally different style (e.g. a café vs. its tea spin-off positioned differently).
  • An experiment with style without risk of overwriting the main brand.
Эволюция бренда: добавляем токены по ходуBrand evolution: adding tokens as you go

Реальный бренд редко стартует «полным». Чаще: загружаете базу (палитра + типографика + spacing), а дальше по ходу проектов нужны новые токены: «вторичный акцент для распродаж», «warm-mode палитра для зимних кампаний», «новый шрифт для презентаций».

Claude Design позволяет дотягивать бренд:

> добавь к нашей brand system второй акцент: тёплый горчичный
> для seasonal-кампаний. имя token: --accent-warm.
> покажи мне обновлённую сводную таблицу tokens.
> экспортируй обновление как tokens.json для backup.

Claude обновляет workspace-bound tokens, и все следующие проекты получают новый акцент. Старые проекты в этом же workspace остаются нетронутыми, но при следующей итерации могут использовать новый token.

Хороший workflow для зрелого бренда:

  1. Каждый раз, когда добавляете новый токен — экспортируйте tokens.json.
  2. Храните этот json в своей кодбазе (или в Figma Variables).
  3. Раз в квартал синхронизируйте: загружаете обновлённый файл в workspace.

Это закрывает цикл «дизайн обновляет систему → код подтягивает обновление → код снова кормит Claude Design». Brand system становится живым артефактом, а не статичным «я один раз настроил».

A real brand rarely starts “complete.” More often: you load a base (palette + typography + spacing), and along the way new tokens come up — “a secondary accent for sales,” “warm-mode palette for winter campaigns,” “a new font for slides.”

Claude Design lets you extend the brand:

> add to our brand system a second accent: warm mustard
> for seasonal campaigns. token name: --accent-warm.
> show me the updated tokens summary table.
> export the update as tokens.json for backup.

Claude updates the workspace-bound tokens; every future project picks up the new accent. Older projects in the same workspace remain untouched but can use the new token on next iteration.

A good workflow for a mature brand:

  1. Every time you add a token — export tokens.json.
  2. Store the json in your codebase (or Figma Variables).
  3. Sync quarterly: re-upload the refreshed file to the workspace.

This closes the loop: “design updates system → code pulls the update → code feeds Claude Design again.” The brand system becomes a living artefact, not a static “set up once.”

Когда design system нужен, а когда — излишествоWhen a design system is needed, when it is overkill

Design system — это инвестиция, не бесплатная. Время на настройку, дисциплина в поддержке, риск over-engineering.

Нужен:

  • 3+ проекта будут в одной стилистике (бренд кофейни, серия презентаций фонда).
  • Командная работа: дизайнер + разработчик + маркетолог должны видеть один и тот же набор tokens.
  • Долгая жизнь продукта (1+ год).
  • У вас уже есть код или Figma с tokens — глупо не использовать.

Излишество:

  • Один артефакт (одна обложка плейлиста). Просто опишите стиль в промпте.
  • Эксперимент со стилем — вы ещё не знаете, какой бренд хотите.
  • Pet-project на вечер. Не надо строить design system для лендинга для себя.

Простое правило: если вы собираетесь делать <3 проектов в одной стилистике — design system не стройте. Описывайте стиль в каждом промпте отдельно. Это нормально.

A design system is an investment, not free. Setup time, maintenance discipline, over-engineering risk.

Needed:

  • 3+ projects will share the style (a café brand, a fund deck series).
  • Team work: designer + developer + marketer all need the same token set.
  • Long product life (1+ year).
  • You already have code or Figma with tokens — silly not to use them.

Overkill:

  • One artefact (one playlist cover). Just describe style in the prompt.
  • A style experiment — you don’t yet know which brand you want.
  • An evening pet project. No need for a design system for your own landing.

Simple rule: if you’re shipping <3 projects in one style — skip the design system. Describe style per prompt. That’s fine.

Эволюция бренда на 5 недельA 5-week brand evolution

Цель: сделать «дневник эволюции» бренда. Пятнадцатиминутное упражнение каждую неделю в течение 5 недель: добавляете один новый токен, экспортируете обновлённый tokens.json. К концу — бренд из 15+ tokens, эволюционировавший естественно, а не «всё сразу».

Goal: a “brand evolution diary.” 15-minute exercise weekly for 5 weeks: add one new token, export updated tokens.json. By the end — a 15+-token brand grown naturally, not “all at once.”

  • Хорошие кандидаты на еженедельные дополнения: warm-mode палитра, второй акцент для распродаж, презентационный шрифт, иконки в одном стиле, моушен-токены (длительность анимаций).
  • Каждую неделю — коммитьте обновлённый tokens.json в любой git-репозиторий (даже личный). Через 5 недель у вас будет история.
  • Good weekly additions: warm-mode palette, secondary accent for sales, slide-display font, one-family icons, motion tokens (animation durations).
  • Each week — commit the refreshed tokens.json to any git repo (personal works). After 5 weeks you have a history.

Main-course graduates saw them in CLAUDE.md; in Claude Design the same idea, lifted into the UI.

How Claude Design reads a codebase or Figma and pulls out design signals (CSS variables, Tailwind config, JSON tokens, SCSS variables). The intermediate step between “Claude extracted tokens” and “Claude designs.” Ask for a summary table of what was found, so you can patch errors before they inherit into 50 artefacts.

The natural extension of a brand system with new tokens as projects grow. § VI · Quiz

Семь вопросовSeven questions

  1. Какие 5 источников tokens читает Claude Design в кодбазе?What 5 token sources does Claude Design read in a codebase?

    показать ответshow answer

    CSS-переменные в :root, Tailwind-config (tailwind.config.js), JSON-tokens (tokens.json, формат W3C), SCSS/Less variables ($primary), Figma Tokens плагин-экспорт.

    CSS variables in :root, Tailwind config (tailwind.config.js), JSON tokens (tokens.json, W3C format), SCSS/Less variables ($primary), Figma Tokens plugin export.

  2. Зачем нужен «verify-шаг» перед запуском дизайна?Why the “verify step” before designing?

    показать ответshow answer

    Чтобы поймать ошибки извлечения до того, как они унаследуются всеми будущими артефактами в workspace. Если Claude не увидел dark-mode палитру или перепутал основной шрифт, лучше узнать это сейчас, чем после 30 экспортов.

    To catch extraction errors before they inherit into every future artefact in the workspace. If Claude missed a dark-mode palette or got the display font wrong, better find out now than after 30 exports.

  3. Что такое workspace inheritance?What is workspace inheritance?

    показать ответshow answer

    Brand-system привязывается к workspace, не к конкретному проекту. Все будущие проекты в этом workspace автоматически используют те же tokens. Один загруженный файл → десять согласованных артефактов без повторной настройки.

    A brand system attaches to a workspace, not a specific project. All future projects in that workspace automatically use the same tokens. One uploaded file → ten consistent artefacts without re-setup.

  4. Чем Figma-источник отличается от кодбазы?How is a Figma source different from a codebase?

    показать ответshow answer

    Figma даёт визуальные компоненты (иллюстрации, иконки, кнопки в конкретной графике), которых нет в tokens из кода. Зато требует аккуратности самого Figma-файла: shared styles, осмысленные названия компонентов, без дубликатов. Лучше для design-first команд.

    Figma offers visual components (illustrations, icons, button graphics) that code tokens don’t carry. But requires a tidy Figma file: shared styles, meaningful component names, no duplicates. Better for design-first teams.

  5. Как добавить новый цвет к существующей brand system?How do you add a new colour to an existing brand system?

    показать ответshow answer

    Промптом в workspace: «добавь токен --accent-warm такой-то, экспортируй обновлённый tokens.json». Claude дотягивает workspace-bound tokens; следующие проекты получают новый акцент. Старые проекты не трогаются, но при следующей итерации могут использовать.

    A prompt in the workspace: “add token --accent-warm, export refreshed tokens.json.” Claude extends the workspace-bound tokens; future projects pick up the new accent. Existing projects are untouched but can use it on next iteration.

  6. Когда design system — это излишество?When is a design system overkill?

    показать ответshow answer

    Один артефакт; стилевой эксперимент, где вы ещё не знаете направления; pet-project на вечер. Простое правило: <3 проектов в одной стилистике — brand system не стройте, описывайте стиль в промпте отдельно.

    One artefact; a style experiment where direction is unclear; an evening pet project. Simple rule: <3 projects in one style — skip the brand system, describe per prompt.

  7. Как закрыть «дизайн ↔ код» цикл для живой brand system?How do you close the “design ↔ code” loop for a live brand system?

    показать ответshow answer

    После каждого добавления токена — экспорт tokens.json, коммит в кодбазу (или Figma Variables). Раз в квартал — синхронизация: загружаете обновлённый файл обратно в workspace. Brand system становится живым артефактом, а не статичной «настроил один раз» вещью.

    After each token addition — export tokens.json, commit to the codebase (or Figma Variables). Quarterly — sync: re-upload the refreshed file to the workspace. The brand system becomes a living artefact, not a static “set up once.”
