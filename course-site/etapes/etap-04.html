§ I · ТемыTopics

Дисциплина больших задачThe discipline of large tasks

Зачем нужен план перед действиемWhy a plan before action

На этапах 00–03 мы работали короткими шагами: один промпт — одна правка — видим результат. Этого хватает на задачи в ~5–10 шагов.

Но есть задачи на 30–50 шагов: «проанализировать 12 банковских выписок, категоризировать ~600 операций, сделать отчёт с графиками и тремя инсайтами». Если делать «одним промптом» — Claude быстро «уплывает»: забывает критерии, теряет цель, оставляет половину работы.

Решение — двухфазная работа:

  1. Фаза «продумай» — никаких изменений в файлах. Claude составляет план, перечисляет шаги, оценивает риски, спрашивает уточнения.
  2. Фаза «сделай» — вы утверждаете план, Claude его исполняет, отмечая прогресс.

Эта философия пронизывает весь этап. Plan Mode, spec.md, TodoWrite, verification — все они служат одному: сначала договориться о том, что будет сделано, потом делать.

On etapes 00–03 we worked in short steps: one prompt — one edit — see the result. Enough for ~5–10-step tasks.

But there are 30–50-step tasks: “parse 12 bank statements, categorise ~600 ops, make a report with charts and three insights.” Done “in one prompt,” Claude drifts fast: forgets criteria, loses goal, leaves half the work.

The fix is two-phase work:

  1. “Think it through” — no file changes. Claude drafts a plan, lists steps, weighs risks, asks for clarification.
  2. “Do it” — you approve the plan, Claude executes, marking progress.

This idea runs through the whole etape. Plan Mode, spec.md, TodoWrite, verification — all serve one goal: agree on what will be done, then do it.

Plan Mode подробно: как и когдаPlan Mode in detail: how and when

Plan Mode — это специальный режим Claude Code, в котором он гарантированно не трогает ваши файлы, пока вы не утвердите план. Включается через Shift+Tab (см. этап 00).

Что Claude делает в Plan Mode:

  • Читает существующие файлы и CLAUDE.md (это разрешено);
  • Изучает зависимости, структуру, ваши данные;
  • Составляет план: фазы, шаги, риски, вопросы;
  • Останавливается, ждёт вашего «да / нет / поправь X».

После «да» план попадает в обычный режим, и Claude его выполняет. Если план слишком большой — Claude сам предложит разбить на несколько подзадач.

Когда обязательно использовать Plan Mode:

  • задача займёт больше 10 шагов;
  • задача завязана на ваши данные и ошибка дорого стоит (бюджет, инвестиции, налоги);
  • вы не уверены, что Claude правильно понял требования;
  • задача затрагивает много файлов одновременно.

Когда не нужно: простые правки, мелкие итерации — они тратят время на «пустой план».

Plan Mode is a special Claude Code mode in which it is guaranteed not to touch your files until you approve the plan. Toggled with Shift+Tab (etape 00).

In Plan Mode Claude:

  • reads existing files and CLAUDE.md (allowed);
  • examines dependencies, structure, your data;
  • drafts a plan: phases, steps, risks, questions;
  • pauses, waits for your “yes / no / change X.”

After “yes” the plan moves to normal mode and Claude executes it. If the plan is too big — Claude itself suggests splitting into sub-tasks.

Definitely use Plan Mode when:

  • the task takes more than 10 steps;
  • the task uses your data and a mistake is expensive (budget, investing, taxes);
  • you’re not sure Claude grasped the requirements;
  • the task touches many files at once.

Skip when: simple edits, small iterations — they waste time on a “trivial plan.”

TodoWrite: видимый прогресс по шагамTodoWrite: visible step-by-step progress

TodoWrite — встроенный инструмент Claude Code. Когда задача состоит из 3+ шагов, Claude автоматически создаёт список задач и показывает его в диалоге. Каждая задача имеет статус:

  • pending — ещё не начата;
  • in_progress — делается прямо сейчас (только одна за раз);
  • completed — готова.

Зачем это вам:

  1. Видите, где Claude. «Сейчас в шаге 3 из 7» — не нужно гадать.
  2. Видите, что забыто. Если в конце все шаги completed, кроме одного pending — это явный сигнал.
  3. Можете остановить и вернуться. Сессия запомнит todo-лист — через --continue вы возвращаетесь к тому же месту.

Можно явно попросить Claude использовать TodoWrite:

> разбей задачу «анализ выписок» на 5–8 фаз
> через TodoWrite. потом приступай.

TodoWrite is a built-in Claude Code tool. When a task has 3+ steps, Claude creates a task list and shows it in the dialogue. Each task has a status:

  • pending — not started;
  • in_progress — happening right now (only one at a time);
  • completed — done.

What it gives you:

  1. Visible position. “Now on step 3 of 7” — no guessing.
  2. Visible misses. If everything is completed but one pending at the end — that’s a clear signal.
  3. You can stop and come back. The session remembers the todo list — --continue returns to the same place.

You can explicitly ask Claude to use TodoWrite:

> split the "statements analysis" task into 5–8 phases
> via TodoWrite. then proceed.
Спецификация (spec.md): техзадание для ClaudeSpec (spec.md): a brief for Claude

Когда задача очень большая (десятки шагов, недели работы), даже Plan Mode мало. В таких случаях работает spec-driven development: вы вместе с Claude пишете spec.md — документ, который описывает задачу до начала работы.

Что в spec обязательно:

  1. Цель. «Что мы строим» в одном абзаце.
  2. Входы. Какие данные на входе. Где они лежат, в каком формате.
  3. Выходы. Что должно появиться. Файлы, страницы, отчёты, цифры.
  4. Шаги. Высокоуровневые фазы работы (5–10 пунктов).
  5. Критерии готовности. «Считаем готовым, когда…» — список измеримых условий.
  6. Что вне scope. Что мы не будем делать (важно для Claude, чтобы он не уплывал).
  7. Риски и вопросы. Открытые места, где нужно ваше решение.

Spec живёт в specs/ или прямо в корне проекта — рядом с CLAUDE.md. На него ссылаются через @spec.md:

> работаем по @spec.md.
> начни с фазы 1, через TodoWrite веди прогресс.
> останавливайся в конце каждой фазы — я буду ревьюить.

When a task is very big (dozens of steps, weeks of work), even Plan Mode is not enough. Spec-driven development works there: together with Claude you write a spec.md — a document describing the task before any work starts.

What a spec must include:

  1. Goal. “What we’re building” in one paragraph.
  2. Inputs. What data, where it lives, in what format.
  3. Outputs. What should appear. Files, pages, reports, numbers.
  4. Steps. High-level phases (5–10 items).
  5. Acceptance criteria. “Done when…” — measurable conditions.
  6. Out of scope. What we will not do (matters so Claude doesn’t drift).
  7. Risks and questions. Open issues that need your decisions.

The spec lives in specs/ or in the project root next to CLAUDE.md. Reference it with @spec.md:

> work to @spec.md.
> start with phase 1, track via TodoWrite.
> stop at the end of each phase — I’ll review.
Compaction и управление длинными сессиямиCompaction and managing long sessions

В долгой сессии (часы работы, десятки промптов) у Claude накапливается «контекстный мусор» — старые ответы, отменённые попытки, тестовые запросы. Это замедляет Claude (больше текста на обработку) и сбивает: он может вспомнить отменённое решение и применить заново.

Команда /compact — «уплотни сессию»:

  1. Claude саммаризирует всё, что было: цель, ключевые решения, текущее состояние;
  2. Удаляет промежуточные сообщения;
  3. Оставляет суммарный «отчёт о текущем положении» как первое сообщение в новом, чистом контексте.

Когда уплотнять:

  • после каждой большой фазы плана (например, «парсинг готов — /compact, дальше категоризация»);
  • когда Claude начал «забывать» решения, принятые часом ранее;
  • в конце рабочего дня — чтобы завтра --continue начался с чистого, но информированного контекста.

Альтернативы:

  • /clear — полная очистка. Контекст теряется. Полезно, когда хотите начать «с нуля» в той же папке.
  • --resume — выбрать одну из недавних сессий, если случайно открыли новую.

In a long session (hours, dozens of prompts) Claude collects “context litter” — old replies, abandoned attempts, test queries. That slows Claude (more text to process) and misleads: it may recall an abandoned decision and reapply it.

The /compact command — “compact the session”:

  1. Claude summarises the whole thing: goal, key decisions, current state;
  2. removes intermediate messages;
  3. keeps the summary as the first message of a clean new context.

When to compact:

  • after each big plan phase (e.g., “parsing done — /compact, then categorisation”);
  • when Claude starts “forgetting” decisions made an hour ago;
  • end of day — so tomorrow’s --continue starts on a clean but informed context.

Alternatives:

  • /clear — wipe completely. Context is lost. Useful when you want a fresh start in the same folder.
  • --resume — pick a recent session if you accidentally opened a new one.
Verification: «готово» только после проверкиVerification: “done” only after the check

Самая частая ошибка крупных задач — «соврать про готовность». Claude может написать код, увидеть, что он выглядит правильно, и объявить задачу законченной. Запустите — не работает.

Дисциплина verification-before-completion требует прежде чем сказать «готово»:

  1. Запустить — программу, страницу, расчёт.
  2. Прогнать тестовый сценарий — известный вход, ожидаемый выход.
  3. Сверить с критериями готовности из spec.md.
  4. Через Playwright сделать скриншот, посмотреть на UX.
  5. Только потом — объявить задачу выполненной.

Skill superpowers:verification-before-completion встраивает эту дисциплину в поведение Claude. Без skill можно делать вручную:

> прежде чем сказать «готово», прогони такой тест:
> на входе X — должно получиться Y. сверь.
> через playwright открой страницу, сделай скриншот.
> покажи мне результат каждой проверки.

The most common big-task failure — “lying about being done.” Claude can write code, see it looks right, declare done. Run it — broken.

The verification-before-completion discipline requires, before saying “done”:

  1. Run — the program, page, calculation.
  2. Walk a test scenario — a known input, expected output.
  3. Compare to acceptance criteria in spec.md.
  4. Through Playwright, screenshot and judge UX.
  5. Only then — declare done.

The superpowers:verification-before-completion skill bakes this into Claude’s behaviour. Without the skill — do it manually:

> before saying "done", run this check:
> with input X — output should be Y. verify.
> through playwright open the page, screenshot.
> show me each check’s result.
Декомпозиция: когда и как разбивать задачуDecomposition: when and how to break tasks down

Признаки, что задача слишком большая для одного промпта:

  • больше 10 шагов;
  • требует нескольких источников данных;
  • содержит несколько независимых выходов (отчёт + дашборд + автоматизация);
  • вы сами не уверены, что должен сделать Claude.

Как разбивать:

  1. На фазы по «данным». Парсинг → нормализация → агрегация → визуализация → отчёт.
  2. На фазы по «артефактам». Сначала data/operations.csv, потом dashboard.html, потом report.md.
  3. На фазы по «ревью». Каждая фаза заканчивается /compact и вашим утверждением.

Каждая фаза должна быть:

  • самостоятельно полезной (можно остановиться и использовать промежуточный результат);
  • измеримой («готово, когда X существует и показывает Y»);
  • укладываться в 1–3 часа.

Signs a task is too big for a single prompt:

  • more than 10 steps;
  • needs several data sources;
  • has several independent outputs (report + dashboard + automation);
  • you yourself aren’t sure what Claude must do.

How to break it down:

  1. Phases by “data.” Parse → normalise → aggregate → visualise → report.
  2. Phases by “artefacts.” First data/operations.csv, then dashboard.html, then report.md.
  3. Phases by “review.” Each phase ends with /compact and your approval.

Every phase should be:

  • independently useful (you can stop and use the partial result);
  • measurable (“done when X exists and shows Y”);
  • 1–3 hours long.

A list of measurable conditions for “done.” E.g.: “chart shows all 12 months,” “total spend reconciles with category sums.” Recorded in spec.md.
Out of scope
Раздел spec, в котором перечислено, что мы не будем делать в этой задаче. Важно, чтобы Claude не уплывал за границы. A spec section listing what we will not do in this task. Important so Claude does not drift past the boundary.
Sanity-check
Быстрая проверка результата на «разумность»: сходятся ли суммы, есть ли все ожидаемые поля, не показывает ли система абсурдных значений. Часть verification. A quick result check for “reasonableness”: do sums add up, are all expected fields present, no absurd values? Part of verification.
§ VI · Quiz

Семь вопросов на закреплениеSeven questions for reinforcement

  1. Какие признаки говорят, что задача слишком большая для одного промпта?Which signs tell a task is too big for a single prompt?

    показать ответshow answer

    Больше 10 шагов; задача завязана на несколько источников данных; есть несколько независимых выходов (отчёт + дашборд + автоматизация); вы сами не уверены, что Claude должен сделать. В этих случаях — Plan Mode, spec, фазы.

    More than 10 steps; depends on several data sources; has several independent outputs (report + dashboard + automation); you yourself aren’t sure what Claude should do. In those cases — Plan Mode, spec, phases.

  2. Чем Plan Mode отличается от обычного режима с точки зрения файлов?How does Plan Mode differ from default in terms of files?

    показать ответshow answer

    В Plan Mode Claude гарантированно не изменяет ни одного файла. Только читает существующие, изучает структуру, составляет план. Файлы трогаются только после вашего «утверждено» и выхода в default.

    In Plan Mode Claude is guaranteed not to change any file. It only reads existing ones, examines structure, drafts a plan. Files get touched only after your “approved” and switch to default.

  3. Что обязательно должно быть в spec.md?What must be in spec.md?

    показать ответshow answer

    Семь обязательных разделов: цель, входы, выходы, шаги (фазы), критерии готовности, что вне scope, риски и вопросы. Без каждого из них Claude может «уплыть» по-своему.

    Seven required sections: goal, inputs, outputs, steps (phases), acceptance criteria, out of scope, risks and questions. Without any of these Claude can drift its own way.

  4. Что делает /compact и чем отличается от /clear?What does /compact do, and how does it differ from /clear?

    показать ответshow answer

    /compact саммаризирует диалог: сохраняет цель, ключевые решения и текущее состояние, удаляет промежуточные сообщения. Контекст «лёгкий», но Claude помнит суть. /clear — полная очистка, контекст теряется. Compact используется чаще — между фазами; clear — редко, для «старта с нуля».

    /compact summarises the dialogue: keeps the goal, key decisions and current state, drops intermediate messages. Context is light, but Claude remembers the gist. /clear wipes everything. Compact is used often — between phases; clear is rare, for a “fresh start.”

  5. Зачем нужно verification-before-completion?Why verification-before-completion?

    показать ответshow answer

    Самая частая ошибка крупных задач — «соврать про готовность»: код выглядит правильным, объявляется «готово», но при запуске не работает. Verification требует прежде чем сказать «готово»: запустить, прогнать тестовый сценарий, сверить с критериями spec, сделать Playwright-скриншот. Только потом — «done».

    The most common big-task failure — “lying about being done”: code looks right, declared done, breaks at runtime. Verification requires, before saying “done”: run, walk a test scenario, compare to spec criteria, take a Playwright screenshot. Only then — “done.”

  6. Как разбивать крупную задачу на фазы — назовите три способа.Three ways to phase a large task?

    показать ответshow answer

    По данным (парсинг → нормализация → агрегация → визуализация); по артефактам (сначала data/operations.csv, потом dashboard.html, потом report.md); по ревью (каждая фаза заканчивается /compact и вашим утверждением).

    By data (parse → normalise → aggregate → visualise); by artefacts (first data/operations.csv, then dashboard.html, then report.md); by review (each phase ends with /compact and your approval).

  7. Какой простой sanity-check вы бы прогнали для «анализа выписок за год»?What simple sanity-check would you run for a “year of statements” analysis?

    показать ответshow answer

    Минимум: число строк в data/all-ops.csv равно сумме строк во всех 12 исходных файлах. Total spend в отчёте равна сумме всех «расход»-строк. Сумма по категориям равна total. Все 12 месяцев присутствуют в графике тренда. Если хоть один пункт «✗» — «готово» нельзя говорить.

    At minimum: row count in data/all-ops.csv equals the sum of rows across all 12 source files. Total spend in the report equals the sum of all “spend” rows. Per-category sums equal the overall total. All 12 months appear in the trend chart. If any of these is “✗” — you can’t say “done.”
