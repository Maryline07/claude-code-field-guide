§ I · ТемыTopics

Что мы пройдёмWhat we will walk through

Что такое handoff bundle и что внутриWhat a handoff bundle is and what it contains

Handoff bundle — это структурированный экспорт из Claude Design, заточенный под чтение Claude Code. В отличие от обычного «Export → HTML», bundle содержит не только разметку, но и:

  • HTML-структура каждого экрана — семантическая, с именами классов как у компонентов, не «div div div».
  • CSS с design tokens в :root и body.dark (если у вас тёмная тема). Прямо как в этом тьюториале.
  • Ассеты: SVG-иконки, PNG-изображения, шрифты (если кастомные) в папке assets/.
  • Типографические правила: font-stack, type-scale, font-features.
  • Навигационный граф (если был прототип): JSON или README с описанием переходов — что куда ведёт.
  • Состояния: empty, loading, error, success — как варианты разметки или CSS-классы.
  • README.md — текстовое описание для разработчика: «вот что это, вот компонентная карта, вот граф навигации».

Это формат, заточенный под Claude Code как «принимающую сторону». Когда вы откроете bundle в Claude Code и скажете «реализуй», он увидит README и поймёт контекст.

Альтернативный экспорт «Export → folder» даёт ту же папку, но без README и без ориентированного-под-Claude формата. Для ручной работы с Figma/Webflow — нормально, для Claude Code — берите именно handoff bundle.

A handoff bundle is a structured export from Claude Design, optimised for Claude Code consumption. Unlike a vanilla “Export → HTML,” the bundle contains:

  • HTML structure per screen — semantic, component-named classes, not “div div div.”
  • CSS with design tokens in :root and body.dark (if you have a dark theme). Just like this tutorial.
  • Assets: SVG icons, PNG images, fonts (if custom) in assets/.
  • Typography rules: font-stack, type-scale, font-features.
  • Navigation graph (if a prototype): JSON or README describing transitions.
  • States: empty, loading, error, success — as markup variants or CSS classes.
  • README.md — a developer-facing description: “here is what it is, here is the component map, here is the navigation graph.”

A format optimised for Claude Code as the “receiving end.” Opening the bundle in Claude Code and saying “implement” means it sees the README and gets the context.

The alternative “Export → folder” gives the same files without README and without the Claude-oriented format. Fine for hand-coding via Figma/Webflow; for Claude Code take the handoff bundle exactly.

Что переживает handoff, что — нетWhat survives handoff, what does not

Реалистичные ожидания экономят часы:

Переходит хорошо:

  • Структура страницы (hero, секции, футер).
  • Палитра, типографика, отступы (как design tokens в CSS).
  • Простые компоненты: кнопки, карточки, формы.
  • Адаптивные брейкпоинты (если вы делали и mobile, и desktop в Claude Design).
  • Иконки в виде SVG.

Переходит частично:

  • Сложные иллюстрации — SVG может оказаться громоздким, лучше переделать вручную или экспортировать в PNG для прототипа.
  • Анимации и переходы — основные передаются как CSS-transitions, сложные motion-эффекты Claude Code обычно реализует «по памяти», не по bundle.
  • 3D-сцены (этап 08) — передаются как описание, не как готовый код. Реализация ложится на Claude Code.

Не переходит:

  • Бизнес-логика (вход, авторизация, валидации, расчёты) — это работа Claude Code с нуля.
  • Бэкенд (база, API) — см этап 06 основного курса.
  • Production-ready performance optimisation — Claude Code дотачивает после имплементации.

Главное правило: handoff передаёт «что», не «как». Claude Code решает «как» имплементировать — на React/Vanilla/Astro, с каким бэкендом, и т.д.

Realistic expectations save hours:

Carries well:

  • Page structure (hero, sections, footer).
  • Palette, typography, spacing (as design tokens in CSS).
  • Simple components: buttons, cards, forms.
  • Responsive breakpoints (if you did mobile + desktop in Claude Design).
  • Icons as SVG.

Carries partially:

  • Complex illustrations — SVG may be bloated; better redo manually or export as PNG for the prototype.
  • Animations and transitions — basics carry as CSS transitions; complex motion is usually rebuilt by Claude Code “from memory.”
  • 3D scenes (etape 08) — transferred as description, not ready code. Implementation falls on Claude Code.

Does not carry:

  • Business logic (auth, validation, calculations) — Claude Code from scratch.
  • Backend (DB, API) — see etape 06 of the main course.
  • Production-ready performance — Claude Code dials in after implementation.

Headline rule: handoff transfers “what,” not “how.” Claude Code decides “how” to implement — React/Vanilla/Astro, which backend, etc.

Принимающая сторона: первый промпт в Claude CodeThe receiving end: first prompt in Claude Code

Вы экспортировали handoff bundle. Это .zip или папка, скачанная локально. Теперь — знакомая среда CLI:

$ mkdir my-cafe-site && cd my-cafe-site
$ unzip ~/Downloads/booking-prototype-handoff.zip -d ./
$ claude

Первый промпт в Claude Code (с помощью @-references из этапа 02 основного курса):

> в этой папке handoff bundle из claude design.
> прочитай @README.md, посмотри @screens/ и @tokens/styles.css.
> реализуй прототип как статичный сайт на vanilla html/css.
> используй tokens как есть, без изменений.
> для всех 5 экранов сделай отдельные страницы с навигацией по графу.
> формы пока без бэкенда — только клиентская валидация.
> деплой через github pages.

Claude Code прочитает структуру, поймёт задачу, и начнёт реализацию — обычным way вы уже знаете по основному курсу. Может, например, в plan mode сначала набросать план файлов, потом начать писать.

Что критично: не давайте Claude Code править tokens. Они — источник истины из дизайна. Если что-то нужно поправить в палитре — вернитесь в Claude Design, обновите brand-system, перегенерируйте bundle. Это сохраняет согласованность.

You exported a handoff bundle. A .zip or folder downloaded locally. Now — familiar CLI:

$ mkdir my-cafe-site && cd my-cafe-site
$ unzip ~/Downloads/booking-prototype-handoff.zip -d ./
$ claude

First Claude Code prompt (using @-references from etape 02 of the main course):

> this folder is a handoff bundle from claude design.
> read @README.md, look at @screens/ and @tokens/styles.css.
> implement the prototype as a static site, vanilla html/css.
> use tokens as-is, no changes.
> for all 5 screens make separate pages with graph-based navigation.
> forms client-side validation only, no backend yet.
> deploy to github pages.

Claude Code reads the structure, understands the task, starts implementing — the usual way you know from the main course. Might use plan mode first to outline files, then write.

What’s critical: do not let Claude Code edit tokens. They’re the design source of truth. If a palette tweak is needed — back to Claude Design, update the brand system, regenerate the bundle. Keeps consistency.

Полный цикл: бриф → дизайн → код → деплойFull loop: brief → design → code → deploy

Этот этап — первый, где у вас в руках полный pipeline двух инструментов. Канонический путь нового продукта:

  1. Бриф: .docx от клиента или ваш собственный (этап 02).
  2. Brand system: tokens из кода или Figma (этап 03).
  3. Прототип: интерактивный, тестируется на 5 пользователях (этап 04).
  4. Handoff: bundle экспортируется и принимается Claude Code (этот этап).
  5. Имплементация: код, бэкенд, тесты — Claude Code (этап 06 основного курса).
  6. Деплой: GitHub Pages, Vercel, или ваш хостинг — одной командой (этап 05 основного курса).
  7. Мониторинг: PostHog, аналитика, фоновые задачи — этап 10 основного.

Это и есть «команда стажёров» в действии. Вы — арт-директор + продакт-менеджер. Claude Design — дизайнер. Claude Code — разработчик. Pipeline закрывается за неделю-две вместо квартала.

Для капстоуна (последний этап тьюториала) вы будете проходить этот pipeline целиком на одном выбранном продукте: бренд → 5-экранный прототип → handoff → живой сайт → публичный URL.

This etape is the first where you hold the full two-tool pipeline. The canonical path of a new product:

  1. Brief: .docx from client or your own (etape 02).
  2. Brand system: tokens from code or Figma (etape 03).
  3. Prototype: interactive, tested on 5 users (etape 04).
  4. Handoff: bundle exported and read by Claude Code (this etape).
  5. Implementation: code, backend, tests — Claude Code (main course etape 06).
  6. Deploy: GitHub Pages, Vercel, or your hosting — one command (main course etape 05).
  7. Monitoring: PostHog, analytics, background tasks — main course etape 10.

That is the “intern team” in action. You — art director + PM. Claude Design — designer. Claude Code — developer. Pipeline closes in 1–2 weeks instead of a quarter.

For the capstone (final etape of this tutorial) you will run this pipeline end-to-end on one chosen product: brand → 5-screen prototype → handoff → live site → public URL.
