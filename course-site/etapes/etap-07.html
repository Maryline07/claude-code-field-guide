Как нанимать стажёровHow to hire interns

Что такое subagent: «стажёр на одну задачу»What a subagent is: an “intern for one task”

Subagent — это отдельный экземпляр Claude, которого вы запускаете на одну подзадачу. У него:

  • Свой контекст. Он не видит вашей переписки с главным Claude. Получает только тот brief, который вы ему дали.
  • Свой набор инструментов. Можно дать ему доступ к файлам, можно ограничить только web-search — зависит от типа.
  • Один результат. Он возвращает одно сообщение с итогом своей работы. Не ведёт диалог — делает задачу и уходит.

Метафора, которую полезно держать в голове весь этап: вы нанимаете стажёра для одной задачи. Стажёр умный, но не знает, чем занят остальной офис. Вы даёте ему brief: «вот данные, вот задача, вот формат ответа», ждёте, получаете отчёт.

Главные плюсы:

  • Параллелизм. Можно запустить 3–4 стажёра одновременно. Что вручную заняло бы 4 часа, делается за один.
  • Изоляция контекста. Вы экономите свой основной контекст: «грязная» работа делается в контексте стажёра, вам приходит только итог.
  • Специализация. Можно нанять «code-reviewer» с его настроенным промптом и поведением, не объясняя каждый раз, что вы хотите от ревью.

A subagent is a separate Claude instance you launch for one sub-task. It has:

  • Its own context. It does not see your dialogue with main Claude. It gets only the brief you give it.
  • Its own tool set. You can grant file access, or restrict to web search — depending on type.
  • One result. It returns one message with its outcome. No back-and-forth — does the task and leaves.

The metaphor for the whole etape: you hire an intern for one task. They’re smart, but unaware of the rest of the office. You give them a brief: “here is the data, here is the task, here is the output format,” you wait, you get a report.

The main wins:

  • Parallelism. Run 3–4 interns at once. What would take 4 hours by hand finishes in 1.
  • Context isolation. You spare your main context: the “messy” work happens inside the intern’s, only the outcome reaches you.
  • Specialisation. Hire a “code-reviewer” with its tuned prompt and behaviour, no need to re-explain what you want every time.
Типы subagents в Claude CodeSubagent types in Claude Code

Claude Code приносит несколько готовых типов. Названия могут отличаться по вашей конфигурации, но логика общая:

  • general-purpose — универсальный стажёр без специализации. Можно дать любую задачу.
  • Explore / code-explorer — «следователь по коду»: пройди по файлам, найди где определена функция X, что в этом проекте необычного. Только чтение, без правок.
  • code-reviewer — ревью кода: ищет баги, проблемы безопасности, отклонения от конвенций.
  • code-architect — архитектура: проектирует решение, прежде чем писать код.
  • code-simplifier — упрощает код: чище, короче, читаемее.
  • plan-agent — «планировщик»: разбивает крупную задачу на шаги.
  • и другие, которые могут появиться от установленных плагинов.

Вызов — через инструмент Task (главный Claude использует его сам, вы просто говорите):

> используй subagent code-reviewer на @backend/auth.py.
> задача: найди риски безопасности и предложи исправления.
> формат: список из 3-5 пунктов, по убыванию серьёзности.

Список доступных — через /agents (или эквивалент в вашей конфигурации).

Claude Code ships a few ready types. Names may differ in your setup, the logic is the same:

  • general-purpose — a generalist intern. Any task.
  • Explore / code-explorer — a “code investigator”: walk files, find where X is defined, note unusual things. Read only, no edits.
  • code-reviewer — code review: bugs, security, convention drift.
  • code-architect — architecture: design first, code later.
  • code-simplifier — simplifies code: cleaner, shorter, more readable.
  • plan-agent — planner: splits a big task into steps.
  • others may appear from installed plugins.

Invocation goes through the Task tool (main Claude uses it, you just say):

> run subagent code-reviewer on @backend/auth.py.
> task: find security risks and propose fixes.
> format: 3-5 bullets, descending severity.

Brief: как нанять стажёра правильноBrief: hiring an intern correctly

Brief — это технический «вход» для subagent. От его качества напрямую зависит, насколько полезен будет результат. Хороший brief содержит:

  1. Цель в одной фразе. «Найди три риска безопасности в auth.py».
  2. Входы. Какие файлы читать, какие данные использовать. Лучше через @-ссылки.
  3. Чего НЕ делать. «Не трогай файлы, только читай». «Не используй сторонние сервисы». «Не давай советов по UX, только по безопасности».
  4. Формат ответа. «Список из 3–5 пунктов в Markdown». «JSON со структурой X».
  5. Объём. «Одна страница максимум». «По 5–7 строк на каждый риск».

Пример хорошего brief для инвест-аналитика:

> запусти subagent general-purpose со следующим brief:
>
> РОЛЬ: фундаментальный аналитик акций.
> ЦЕЛЬ: оцени фундаменталы AAPL за последний квартал.
> ВХОДЫ: можешь использовать WebSearch и WebFetch для
>   сайта investor.apple.com и финансовых лент.
> НЕ ДЕЛАЙ: не давай прогнозов. не используй технический анализ.
>   не пиши «по моему мнению» — только факты.
> ФОРМАТ: markdown-блок с разделами:
>   - выручка (yoy +/-);
>   - чистая прибыль (yoy +/-);
>   - P/E vs средний по сектору;
>   - три ключевых факта из последнего квартала.
> ОБЪЁМ: одна страница максимум.

Принцип «fail fast в brief»: лучше потратить 30 секунд на точный brief, чем 5 минут на разбор результата, в котором стажёр сделал «не то».

A brief is the technical “input” for the subagent. The result’s usefulness scales directly with brief quality. A good brief has:

  1. Goal in one sentence. “Find three security risks in auth.py.”
  2. Inputs. Which files to read, which data to use. Use @-mentions where you can.
  3. What NOT to do. “Read only, don’t edit.” “No third-party services.” “Don’t give UX advice — security only.”
  4. Output format. “3–5 markdown bullets.” “JSON of shape X.”
  5. Length. “One page max.” “5–7 lines per risk.”

An example brief for an investing analyst:

> spawn a general-purpose subagent with this brief:
>
> ROLE: fundamental stock analyst.
> GOAL: assess AAPL fundamentals for the last quarter.
> INPUTS: you can use WebSearch and WebFetch for
>   investor.apple.com and financial feeds.
> DO NOT: don't forecast. no technical analysis.
>   no "in my opinion" — facts only.
> FORMAT: a markdown block with sections:
>   - revenue (yoy +/-);
>   - net income (yoy +/-);
>   - P/E vs sector average;
>   - three key facts from the latest quarter.
> LENGTH: one page max.

The “fail fast in the brief” rule: better spend 30 seconds on a precise brief than 5 minutes parsing a result where the intern did “the wrong thing.”

Параллельный запуск: несколько стажёров одновременноParallel dispatch: several interns at once

Главный плюс subagents — параллелизм. Если задачи независимы (один анализирует фундаменталы, второй — график, третий — новости), запускаем их одновременно.

В Claude Code для этого — параметр run_in_background: true. Один промпт — четыре стажёра в работе:

> для AAPL запусти ПАРАЛЛЕЛЬНО (run_in_background) четырёх subagents:
>
> 1) ФУНДАМЕНТАЛЬНЫЙ — выручка, прибыль, P/E (brief 1).
> 2) ТЕХНИЧЕСКИЙ — тренд, индикаторы, поддержки (brief 2).
> 3) НОВОСТНОЙ — sentiment по 30 свежим заголовкам (brief 3).
> 4) МАКРО — отрасль и экономический фон (brief 4).
>
> ЖДИ всех. КАЖДЫЙ возвращает свой markdown-блок.
> ПОТОМ собери в один файл memo-aapl.md с разделами по каждому.

Что произойдёт:

  • Главный Claude отправит четыре Task-вызова, каждый с run_in_background: true;
  • Они работают параллельно (~2–4 минуты вместо ~10–15);
  • Главный Claude дожидается всех, читает результаты;
  • Сшивает в единый отчёт.

Когда параллелизм не нужен: если результат одного стажёра — вход для другого. Тогда последовательно (foreground): дождались первого, передали второму.

The main subagents win is parallelism. When tasks are independent (one for fundamentals, another for the chart, a third for news), run them simultaneously.

In Claude Code use the run_in_background: true parameter. One prompt — four interns working:

> for AAPL launch IN PARALLEL (run_in_background) four subagents:
>
> 1) FUNDAMENTAL — revenue, profit, P/E (brief 1).
> 2) TECHNICAL — trend, indicators, supports (brief 2).
> 3) NEWS — sentiment over 30 fresh headlines (brief 3).
> 4) MACRO — sector and macro backdrop (brief 4).
>
> WAIT for all. EACH returns its own markdown block.
> THEN stitch into one memo-aapl.md with sections per analyst.

What happens:

  • Main Claude fires four Task calls, each with run_in_background: true;
  • They run in parallel (~2–4 minutes instead of ~10–15);
  • Main Claude waits for all, reads the results;
  • Stitches into a single report.

When parallelism doesn’t fit: when one intern’s output is another’s input. Then sequential (foreground): wait the first, hand to the second.

Foreground vs background: когда какоеForeground vs background: which when

Foreground — «жду стажёра, прежде чем продолжить». Главный Claude посылает задачу, останавливается, дожидается результата.

  • Используется, когда результат подзадачи нужен прямо сейчас для следующего шага;
  • Удобно для «одного стажёра на разовую задачу»;
  • Плюс: проще читать — результат сразу в диалоге.

Background — «отправил задачу, продолжаю работать, заберу результат позже».

  • Используется для параллельных независимых задач (как четыре аналитика);
  • Удобно, когда задача длинная (например, анализ всех файлов проекта);
  • Плюс: общее время работы сокращается в разы.

Простое правило:

  • Один стажёр + результат сейчас → foreground.
  • 3+ стажёра одновременно → background.
  • Длинная задача (5+ минут), хочется параллельно делать что-то ещё → background, через какое-то время «проверь, готово ли».

Foreground — “wait for the intern before continuing.” Main Claude sends the task, pauses, waits for the result.

  • For when the sub-result is needed right now for the next step;
  • Good for “one intern on a one-off task”;
  • Plus: easier to read — the result lands in the dialogue.

Background — “fired the task, kept working, will pick up the result later.”

  • For parallel independent tasks (the four analysts);
  • Good for long tasks (e.g., analysing the whole project);
  • Plus: total wall-time drops several-fold.

Simple rule:

  • One intern + result needed now → foreground.
  • 3+ interns at once → background.
  • Long task (5+ min), want to do something else in parallel → background, ask later “is it ready.”
Worktree-isolation: «грязные» стажёры в своей песочницеWorktree isolation: “messy” interns in their own sandbox

Если subagent читает — его можно запускать «как есть»: он ничего не ломает. Но если он пишет файлы (например, рефакторит код), вы не хотите, чтобы три параллельных стажёра одновременно правили один и тот же app.py.

Решение — worktree-isolation: каждый «грязный» стажёр работает в собственной копии репозитория (worktree, см. этап 05). После завершения вы смотрите на каждый результат отдельно, выбираете, что мержить, что отбрасывать.

> для рефакторинга frontend/auth.tsx запусти параллельно:
> — code-simplifier в worktree feature/auth-simplify;
> — code-reviewer в worktree review/auth (только чтение, ему worktree не обязателен);
> — code-architect в worktree feature/auth-redesign.
>
> ЖДИ всех. покажи мне три предложенных версии,
> чтобы я выбрал, что мержить в main.

Skill superpowers:using-git-worktrees добавляет это автоматически: «грязные» subagents всегда получают свой worktree, чистые — нет.

If a subagent reads — you can run it as is: it breaks nothing. But if it writes files (e.g. refactors code), you don’t want three parallel interns editing the same app.py at once.

The fix is worktree isolation: every “messy” intern works in its own repo copy (worktree, see etape 05). When done you review each result separately and decide what to merge and what to drop.

> for refactoring frontend/auth.tsx, dispatch in parallel:
> — code-simplifier in worktree feature/auth-simplify;
> — code-reviewer in worktree review/auth (read only, worktree not required);
> — code-architect in worktree feature/auth-redesign.
>
> WAIT for all. show me the three proposals
> so I pick what to merge into main.

The superpowers:using-git-worktrees skill makes this automatic: “messy” subagents always get a worktree, “clean” ones don’t.

Multi-agent teams: координация и передача результатовMulti-agent teams: coordination and handoffs

Простой шаблон — «один главный, четыре параллельных» — работает для independent задач. Но есть и более сложные сценарии:

  1. Конвейер. Агент A собирает данные → агент B анализирует → агент C пишет отчёт. Каждый — в foreground, выход одного передаётся как вход следующему.
  2. Конкурс. Три агента решают одну задачу разными способами, главный выбирает лучший. Полезно, когда подход неочевиден.
  3. Дискуссия. Агент A пишет план, агент B критикует, агент A переписывает с учётом критики. Полезно для сложных архитектурных решений.

Главный принцип: главный Claude — всегда один. Он дирижёр оркестра, а subagents — музыканты. Стажёр стажёру не звонит — всё координирует главный.

Когда задача требует именно «дирижёра», полезен skill superpowers:dispatching-parallel-agents: он учит главного Claude правильно строить такие схемы и собирать результаты.

The simple pattern — “one main, four in parallel” — fits independent tasks. There are richer scenarios too:

  1. Pipeline. Agent A collects data → agent B analyses → agent C writes the report. Each is foreground, the output of one feeds the next.
  2. Contest. Three agents solve one task different ways, the main picks the best. Useful when the approach isn’t obvious.
  3. Debate. Agent A drafts a plan, agent B critiques, agent A rewrites with critique in mind. Useful for hard architectural decisions.

Core principle: main Claude is always single. It’s the conductor; subagents are musicians. Interns don’t call interns — the conductor coordinates.

When the task wants a real conductor, the superpowers:dispatching-parallel-agents skill helps: it teaches main Claude to build such schemes and gather results.

Когда subagents не нужныWhen subagents are not needed

Subagents — не «серебряная пуля». Они платят накладные расходы (запуск контекста, передача brief), и иногда проще без них.

Не используйте subagents:

  • Для мелких правок, на которые главный Claude отвечает за 30 секунд.
  • Когда задачи зависят друг от друга в нелинейном порядке (постоянная необходимость «в ответ на X сделать Y» — на это нужен диалог, а не одноразовый brief).
  • Для интерактивных сценариев («давай попробуем», «нет, не так»). Subagent делает один ответ — вы можете запросить ещё, но это уже новый стажёр со своим контекстом.

Используйте, когда:

  • Задача независима и «крупная» (минут на 5–15);
  • Нужны 2+ независимых результата;
  • Вы не хотите засорять основной контекст «грязной» работой (логи, поиск по коду, парсинг страниц);
  • Есть специализированный тип, идеально подходящий (code-reviewer для ревью, и т. д.).

Subagents are not a silver bullet. They have overhead (context spin-up, brief handover), and sometimes simpler is better.

Don’t use subagents:

  • For small edits main Claude handles in 30 seconds.
  • When tasks depend on each other in a non-linear way (constant “in response to X do Y” — needs dialogue, not a one-shot brief).
  • For interactive flows (“let’s try,” “no, not like that”). A subagent gives one reply — you can ask again, but that’s a new intern with new context.

Use when:

  • The task is independent and “large” (5–15 minutes);
  • You need 2+ independent results;

  1. Чем subagent отличается от «просто другой задачи в том же диалоге»?How is a subagent different from “just another task in the same dialogue”?

    показать ответshow answer

    Subagent — отдельный экземпляр Claude со своим контекстом. Не видит вашей переписки. Получает только brief, делает одну задачу, возвращает один ответ. «Просто другая задача» — всё в том же контексте, всё видит. Subagent — чистая «коробка», изолированная от шума.

    A subagent is a separate instance of Claude with its own context. Doesn’t see your dialogue. Gets only the brief, does one task, returns one reply. “Just another task” shares the context, sees everything. A subagent is a clean “box,” isolated from noise.

  2. Что обязательно должно быть в хорошем brief?What must a good brief include?

    показать ответshow answer

    Пять пунктов: цель в одной фразе; входы (какие файлы/данные читать, через @); что НЕ делать (явные ограничения); формат ответа (markdown / JSON / список); объём (одна страница / 5 пунктов). Без каждого — стажёр сделает «не то».

    Five items: goal in one sentence; inputs (which files/data, via @); what NOT to do (explicit limits); output format (markdown / JSON / list); length (one page / 5 bullets). Skip any — the intern does “the wrong thing.”

  3. Когда foreground, когда background?When foreground, when background?

    показать ответshow answer

    Foreground: один стажёр, результат нужен сейчас для следующего шага. Главный Claude ждёт. Background: 3+ независимых стажёра, или одна длинная задача, рядом с которой можно делать ещё что-то. Главный отправляет всех и ждёт всех вместе — общее время равно времени самого медленного.

    Foreground: one intern, result needed now for the next step. Main Claude waits. Background: 3+ independent interns, or one long task while you can do something else. Main fires all and waits for them together — total time equals the slowest.

  4. Зачем worktree-isolation, и для каких subagents она не нужна?Why worktree isolation, and for which subagents is it not needed?

    показать ответshow answer

    Нужна для «грязных» субагентов, которые пишут в файлы (рефакторинг, генерация кода). Если их запускать без isolation, два параллельных могут конфликтовать — один перезатрёт правки другого. Не нужна для «чистых» — которые только читают (code-reviewer, аналитики, новостные агенты).

    Needed for “messy” subagents that write to files (refactor, code-gen). Without isolation, two in parallel may collide — one overwrites the other. Not needed for “clean” ones that only read (code-reviewer, analysts, news agents).

  5. Кто координирует команду subagents?Who coordinates a team of subagents?

    показать ответshow answer

    Главный Claude. Он «дирижёр»: даёт brief каждому, ждёт ответы, собирает в общий результат. Subagents не общаются друг с другом напрямую — они «музыканты», каждый смотрит только в свои ноты. Это упрощает координацию: один источник истины.

    Main Claude. It’s the “conductor”: hands a brief to each, waits, stitches into one result. Subagents don’t talk to each other directly — they’re “musicians,” each reads only its score. Simpler coordination: one source of truth.

  6. Что такое «red team» и зачем он на Due Diligence?What is a “red team” and why on a Due Diligence?

    показать ответshow answer

    Дополнительный subagent в конце процесса с brief'ом «найди слабости в этом результате». Помогает заметить «слепые пятна»: где аргументация хрупкая, какие риски недооценили, что бы спросил скептик. На DD особенно ценно, потому что цена ошибочного вердикта — реальные деньги.

    An extra subagent at the end with a “find weaknesses” brief. Helps notice blind spots: where the argument is fragile, which risks were under-weighted, what a sceptic would ask. Especially valuable on a DD, where the cost of a bad verdict is real money.

  7. Когда subagents не нужны?When are subagents not needed?

    показать ответshow answer

    Для мелких правок, которые главный Claude делает за 30 секунд. Для интерактивных сценариев («давай попробуем», «нет, не так») — subagent даёт один ответ, диалога не ведёт. Для задач с нелинейными зависимостями — на них нужна гибкость, а subagent работает по брифу. Накладные расходы запуска не окупаются.

    For tiny edits main Claude handles in 30 seconds. For interactive flows (“let’s try,” “no, not like that”) — a subagent gives one reply, no dialogue. For tasks with non-linear dependencies — they need flexibility, a subagent works to a brief. The startup overhead doesn’t pay off.
