Семь вещей, которые надо понять перед кодомSeven things to grasp before code

В Telegram-боте легко увязнуть в деталях библиотеки. Сначала — модель: что такое бот в голове Telegram, как он узнаёт пользователей, и где живёт. Потом инструменты.

It’s easy to get lost in library details with a Telegram bot. First — the mental model: what a bot is in Telegram’s eyes, how it identifies users, where it lives. Then the tools.

1. Что такое Telegram Bot API: long polling vs webhook1. What the Telegram Bot API is: long polling vs webhook

«Бот» на стороне Telegram — это просто аккаунт, к которому привязан токен и который умеет получать сообщения программно. Telegram сам ничего вашему коду не шлёт по умолчанию: вы выбираете один из двух режимов.

  • Long polling — ваш бот раз за разом спрашивает у Telegram «есть новые сообщения?» и ждёт ответа до 30 секунд. Telegram отвечает либо «нет», либо отдаёт пачку обновлений. Удобно для учебного бота: ничего настраивать не надо, работает из коробки.
  • Webhook — вы даёте Telegram свой публичный URL, и Telegram сам присылает туда POST-запрос на каждое событие. Быстрее, экономит ресурсы, но требует HTTPS-эндпоинт. Подходит для production.

Что выбираем сейчас: long polling. На Railway бот будет жить как отдельный worker-сервис (см. тему 3), который запускается командой python bot.py — и всё. Webhook добавим в этапе 11, когда речь пойдёт об агентах и надёжности.

> объясни мне long polling простыми словами,
> сравни с webhook, и подскажи: для нашего учебного бота
> на Railway что выбрать и почему?

A “bot” on Telegram’s side is just an account with an attached token, designed to receive messages programmatically. Telegram doesn’t push to you by default: you pick one of two modes.

  • Long polling — your bot asks Telegram “any new messages?” over and over, waiting up to 30 seconds for an answer. Telegram replies either “none” or with a batch of updates. Convenient for a learning bot: nothing to configure, works out of the box.
  • Webhook — you give Telegram a public URL, and Telegram POSTs each event to it. Faster, cheaper on resources, but requires an HTTPS endpoint. Suits production.

What we pick now: long polling. On Railway the bot lives as a worker service (see topic 3) launched with python bot.py — and that’s it. We’ll add webhooks in etape 11 when we cover agents and reliability.

> compare to webhook, and recommend: for our learning bot

2. aiogram 3: маршрутизатор, хендлеры, FSM2. aiogram 3: router, handlers, FSM

aiogram — асинхронный python-фреймворк для Telegram-ботов. Архитектурно похож на FastAPI, который вы уже знаете: есть «приложение» (Dispatcher), есть «роутеры», на них регистрируются обработчики. Только вместо HTTP-эндпоинтов — обработчики команд и текстов.

# bot.py — упрощённая структура
import asyncio
from aiogram import Bot, Dispatcher, Router
from aiogram.filters import Command

router = Router()

@router.message(Command("start"))
async def start(message):
    await message.answer("Привет! Это ваш инвест-бот.")

@router.message(Command("portfolio"))
async def portfolio(message):
    # сюда позже добавим запрос в backend
    await message.answer("Портфель пока недоступен — войдите через /start")

async def main():
    bot = Bot(token=os.environ["TELEGRAM_BOT_TOKEN"])
    dp = Dispatcher()
    dp.include_router(router)
    await dp.start_polling(bot)

asyncio.run(main())

Три понятия, которые надо запомнить:

Handler
асинхронная функция, которая обрабатывает одно событие (команду /start, текст «привет», кнопку). Декоратор @router.message(...) привязывает её к фильтру.
Router
группа handlers. У нас будет два: auth_router (логин, привязка) и main_router (команды кабинета).
FSM
конечный автомат. У нас понадобится только в одном месте — для команды /login с кодом подтверждения. Для большинства команд состояние не нужно: бот не помнит контекст, каждая команда самостоятельная.
> покажи минимальный пример aiogram 3:
> один handler на /start, один на /portfolio.
> не используй FSM пока. объясни каждую строку.

aiogram is an async Python framework for Telegram bots. Architecturally similar to FastAPI, which you already know: there’s an “app” (Dispatcher), routers, and handlers attached to them. Instead of HTTP endpoints — command and text handlers.

Three concepts to remember:

Handler
an async function that handles one event (command /start, text “hi”, a button). The decorator @router.message(...) binds it to a filter.
Router
a group of handlers. We’ll have two: auth_router (login, linking) and main_router (office commands).
FSM
finite state machine. We only need it in one place — the /login command with a confirmation code. Most commands don’t need state: each is self-contained.
> show the minimal aiogram 3 example:
> one handler on /start, one on /portfolio.
> no FSM yet. explain each line.
3. Worker-сервис на Railway: бот — не web-сервер3. Worker service on Railway: a bot is not a web server

Backend этапа 06 был web-сервисом: Railway видел, что он слушает порт $PORT, посылал на него healthcheck, и при успехе считал сервис «живым». Бот же ничего не слушает — он просто крутит цикл polling. Это worker-сервис.

На Railway это решается через файл Procfile или поле «Start Command» в настройках сервиса:

# Procfile в корне репозитория бота:
worker: python bot.py

Когда Claude будет деплоить, напомните ему:

> это worker-сервис, не web. на Railway:
> — не указывай PORT, healthcheck отключи;
> — Procfile должен быть с типом worker;
> — TELEGRAM_BOT_TOKEN и BACKEND_URL — через env vars в Railway dashboard;
> — добавь логирование: при старте печатай «бот запущен, опрашиваю Telegram»,
>   при ошибке — стектрейс через logging.exception().

Один backend, два сервиса. У вас в Railway появятся два отдельных сервиса: web (backend этапа 06) и worker (этот бот). Они общаются через HTTP — бот ходит на https://my-investing-backend.up.railway.app/api/.... БД у них одна и та же.

The etape-06 backend was a web service: Railway saw it listening on $PORT, sent healthchecks, and treated it as alive on success. A bot listens to nothing — it just runs a polling loop. That’s a worker service.

On Railway you set this via a Procfile or the “Start Command” field:

# Procfile at the bot repo root:
worker: python bot.py

When Claude deploys, remind it:

> this is a worker service, not web. on Railway:
> — no PORT, disable healthcheck;
> — Procfile must declare type 'worker';
> — TELEGRAM_BOT_TOKEN and BACKEND_URL — via env vars in Railway dashboard;
> — add logging: on start print "bot up, polling Telegram",
>   on error — stacktrace via logging.exception().

One backend, two services. You’ll have two Railway services: web (etape-06 backend) and worker (this bot). They talk over HTTP — the bot calls https://my-investing-backend.up.railway.app/api/.... They share one DB.

4. Авторизация без паролей в чате: deep-link4. Auth without passwords in chat: deep-link

Антипаттерн: команда /login email password. Пароль остаётся в истории чата у пользователя, в логах Telegram, в логах бота. Если телефон украдут — пароль найдут текстом. Этап 06 учил «никогда не светить пароли в логах» — в Telegram это правило ещё строже.

Anti-pattern: a /login email password command. The password lives in the user’s chat history, in Telegram’s logs, in your bot’s logs. Steal the phone — find the password as plain text. Etape 06 taught “never log passwords” — in Telegram that rule is stricter.

Правильный приём — deep-link с одноразовым токеном. Идея: настоящая авторизация остаётся в вебе, бот получает только короткий «ключ» с истекающим сроком жизни.

  1. Пользователь логинится в вашем веб-кабинете (этап 06, проект A) обычным образом.
  2. На странице /profile жмёт кнопку «Привязать Telegram». Backend генерирует одноразовый токен (например, 32 случайных символа), сохраняет в таблице TelegramLinkToken(token, user_id, expires_at) со сроком жизни 5 минут.
  3. Backend возвращает ссылку: https://t.me/my_investing_bot?start=<token>. Пользователь нажимает.
  4. Telegram открывает чат с ботом и шлёт ему специальное событие /start <token>.
  5. Бот в handler /start читает command.args, идёт в backend на POST /api/telegram/link с этим токеном и своим telegram_id.
  6. Backend проверяет токен, сохраняет связку telegram_id ↔ user_id, удаляет одноразовый токен. Возвращает боту JWT с коротким TTL (например, 24 часа).
  7. Бот хранит JWT в aiogram-storage по ключу telegram_id.
  1. User logs into your web office (etape-06 project A) the regular way.
  2. On /profile they click “Link Telegram”. Backend generates a one-time token (32 random chars), stores it as TelegramLinkToken(token, user_id, expires_at) with a 5-minute TTL.
  3. Backend returns the link https://t.me/my_investing_bot?start=<token>. User taps it.
  4. Telegram opens the bot and sends a special /start <token> event.
  5. The bot reads command.args in the /start handler and calls POST /api/telegram/link with this token and its telegram_id.
  6. Backend validates the token, stores telegram_id ↔ user_id, deletes the one-time token. Returns a JWT to the bot with a short TTL (e.g. 24h).
  7. The bot stores the JWT in aiogram-storage keyed by telegram_id.

Что мы выиграли: ни email, ни пароль никогда не идут через Telegram. Если кто-то завладеет вашим Telegram — он увидит JWT, но он истечёт через сутки. Веб-кабинет вы всегда можете «отключить от Telegram» одной кнопкой — backend удаляет связь.

> в проекте A добавь:
> — модель TelegramLinkToken (id, token, user_id, expires_at, used)
> — эндпоинт POST /api/telegram/link-init: генерит токен, возвращает t.me-URL
> — эндпоинт POST /api/telegram/link: принимает token + telegram_id, валидирует
>   срок и used=False, помечает used=True, привязывает telegram_id к User,
>   возвращает JWT (TTL 24 часа)
> — эндпоинт POST /api/telegram/unlink (под /profile): отвязка
> ВАЖНО: один раз использованный токен — мёртв. срок 5 минут. в логах не светим.

The gain: no email or password ever passes through Telegram. If your Telegram gets hijacked, the attacker sees a JWT that expires in 24h. You can always “unlink Telegram” from the web with one click — the backend kills the binding.

5. Хранение состояния: MemoryStorage и что бывает при рестарте5. State storage: MemoryStorage and what restarts do to it

aiogram даёт два варианта storage «из коробки»:

MemoryStorage
состояние и FSM в памяти процесса. Рестарт сервиса = всё пропало. На Railway worker может перезапуститься после деплоя, обновления окружения, OOM-kill. Все JWT-токены, которые бот «держал» для пользователей, обнулятся — им придётся снова жать «Привязать Telegram».
RedisStorage
состояние в Redis. Переживает рестарт. Требует Redis-сервиса (Railway его умеет, $5/мес. на старте).

Прагматичный выбор учебного этапа: начинаем с MemoryStorage, потому что просто, бесплатно и учит понимать, что такое «состояние». В projects A/B этого этапа явно описываем UX: «после долгого простоя, возможно, попросит снова привязать Telegram — это нормально». Если потом захочется production-надёжности — добавляете Redis. Это одна строка в коде:

from aiogram.fsm.storage.redis import RedisStorage
storage = RedisStorage.from_url(os.environ["REDIS_URL"])
dp = Dispatcher(storage=storage)

aiogram offers two storage options out of the box:

MemoryStorage
state and FSM in process memory. Restart = everything gone. On Railway a worker can restart after deploy, env update, OOM-kill. All JWTs the bot held for users vanish — they’ll have to re-link.
RedisStorage
state in Redis. Survives restarts. Requires a Redis service (Railway provides one, ~$5/month).

Pragmatic call for this etape: start with MemoryStorage — simple, free, and teaches “what is state”. In projects A/B we make the UX explicit: “after a long idle, you may be asked to re-link Telegram — that’s normal.” For production robustness — add Redis. One line:

from aiogram.fsm.storage.redis import RedisStorage
storage = RedisStorage.from_url(os.environ["REDIS_URL"])
dp = Dispatcher(storage=storage)
6. Миграция схемы: добавляем telegram_id в User6. Schema migration: adding telegram_id to User

Backend этапа 06 уже задеплоен на Railway, в PostgreSQL уже есть пользователи. Теперь нужно добавить колонку telegram_id к модели User — и применить это к живой БД, не потеряв данные. Это миграция.

Инструмент — Alembic, который вы уже видели в этапе 06. Каждое изменение схемы — отдельный файл в alembic/versions/, у каждого свой ID и ссылка на «родителя».

> в backend проекта A создай alembic-миграцию add_telegram_id:
> — добавляет nullable колонку telegram_id (BigInteger, unique=True) в User;
> — добавляет таблицу TelegramLinkToken (id, token, user_id, expires_at, used);
> — индекс по telegram_id для быстрого поиска при логине из бота.
>
> покажи и upgrade(), и downgrade(). объясни, почему nullable, а не NOT NULL.

Почему nullable: старые пользователи ещё не привязали Telegram, у них колонка должна быть пустой. Если поставить NOT NULL без default — миграция упадёт.

Накат на production. Локально: alembic upgrade head. На Railway: либо в Procfile добавляем pre-deploy release: alembic upgrade head, либо запускаем команду из Railway shell вручную. Для маленького проекта — вторая опция проще.

Перед накатом — бэкап. Railway делает автоматические снапшоты БД, но учить руки лучше сразу: «прежде чем накатить миграцию, сделай pg_dump через Railway shell, сохрани локально как backup-before-tg.sql».

The etape-06 backend is already deployed on Railway, with users in PostgreSQL. Now we add a telegram_id column to User and apply it to a live DB without data loss. That’s a migration.

Tool — Alembic, which you saw in etape 06. Each schema change is a file in alembic/versions/ with its own ID and a parent reference.

Why nullable: existing users haven’t linked Telegram, so the column must be empty. NOT NULL without a default would fail.

Applying on production. Locally: alembic upgrade head. On Railway: either add release: alembic upgrade head as a pre-deploy in Procfile, or run from the Railway shell manually. For a small project, option two is simpler.

Back up first. Railway snapshots automatically, but train the muscle: “before applying the migration, run pg_dump via Railway shell, save locally as backup-before-tg.sql”.

7. Безопасность токена бота7. Bot token security

Токен бота — это пароль, который даёт полный контроль над ботом. С ним можно отправлять сообщения от имени бота, читать всё, что ему пишут, узнать список всех привязанных пользователей. Если утечёт:

  • злоумышленник напишет вашим пользователям от имени «вашего» бота с фейковой ссылкой «обнови привязку Telegram»;
  • пользователь, привыкший доверять боту, нажмёт — и уедет на фишинг-страницу «инвест-кабинета»;
  • введёт там настоящий пароль — и всё, аккаунт уведён.

Защита. Простое, но всё — правда:

  1. Токен только в TELEGRAM_BOT_TOKEN в Railway env. Никогда в коде, в .env.example, в скриншотах, в коммитах.
  2. .gitignore содержит .env. Перед каждым коммитом — git diff --cached | grep -i token, либо включите hook (этап 08a).
  3. Если токен утёк (или подозреваете) — в BotFather: /revoke, выберите бота, «Revoke current token». Telegram выдаст новый, старый умрёт сразу. Обновите Railway env.
  4. Внутри бота: при логировании сообщений никогда не печатайте message.text целиком, если это /start <token> — одноразовые токены тоже секрет.
> пройдись по моему боту как security-эксперт. найди:
> — где может утечь TELEGRAM_BOT_TOKEN (логи, переменные, ошибки),
> — где может утечь одноразовый link-token (printf, exception),
> — где может утечь JWT пользователя.
> для каждого — конкретный фикс.

A bot token is a password granting full control over the bot. With it you can send messages on the bot’s behalf, read everything sent to it, list every linked user. On leak:

  • attacker DMs your users from “your” bot with a fake “update your Telegram link” URL;
  • a user who trusts the bot taps it — lands on a phishing page;
  • enters their real password — account compromised.

Defense. Boring but true:

  1. Token only in TELEGRAM_BOT_TOKEN in Railway env. Never in code, .env.example, screenshots, commits.
  2. .gitignore contains .env. Before each commit: git diff --cached | grep -i token, or install a hook (etape 08a).
  3. On leak (or suspicion): in BotFather /revoke, pick the bot, “Revoke current token”. Telegram issues a new one, the old dies immediately. Update Railway env.
  4. Inside the bot: when logging messages, never print full message.text if it’s /start <token> — one-time tokens are secrets too.

Токен ботаBot token

Секретная строка вида 1234567890:AAH..., через которую ваш код подключается к Telegram и «притворяется» ботом. Утечка = полный контроль над ботом. Хранится только в TELEGRAM_BOT_TOKEN в env, ротируется через BotFather /revoke.

A secret string like 1234567890:AAH... your code uses to authenticate as the bot. Leak = full bot control. Lives only in TELEGRAM_BOT_TOKEN env, rotated via BotFather /revoke.

Long polling

Режим работы, при котором ваш бот сам непрерывно спрашивает Telegram «есть новые события?». Простой, не требует HTTPS-эндпоинта, годится для учебных и маленьких ботов.

A mode where your bot continuously asks Telegram “any new events?”. Simple, no HTTPS endpoint required, fine for learning and small bots.

Webhook

Альтернатива long polling: вы даёте Telegram свой публичный HTTPS-URL, и он сам присылает на него POST на каждое событие. Дешевле по ресурсам, нужен публичный домен с SSL.

Alternative to long polling: you give Telegram a public HTTPS URL, it POSTs each event to you. Cheaper on resources, needs a public domain with SSL.

aiogram

Python-фреймворк для Telegram-ботов. Версия 3.x — современная, async, роутерная архитектура. Аналог FastAPI, но для Telegram-событий.

A Python framework for Telegram bots. Version 3.x — modern, async, router-based. Analogous to FastAPI but for Telegram events.

Worker-сервисWorker service

Долгоживущий процесс на сервере, который не слушает HTTP, а делает свою работу: опрашивает очередь, шлёт уведомления, считает по расписанию. На Railway указывается через Procfile с типом worker.

A long-running server process that doesn’t answer HTTP — it does background work: polls queues, pushes notifications, runs scheduled jobs. On Railway: Procfile with type worker.

Deep-link

Ссылка вида https://t.me/<bot>?start=<payload>: при клике Telegram открывает чат с ботом и передаёт ему payload как аргумент команды /start. Главный приём для безопасной авторизации — передаём одноразовый токен, не email/пароль.

A URL like https://t.me/<bot>?start=<payload>: tapping it opens the bot and passes payload as the /start argument. The key trick for secure auth — pass a one-time token, not email/password.

FSM (конечный автомат)FSM (finite state machine)

Способ запомнить, в каком «шаге» диалога находится конкретный пользователь («жду email» → «жду пароль» → «всё ввёл»). aiogram даёт встроенный FSM-механизм поверх storage.

A way to remember which step a specific user is at (“waiting for email” → “waiting for password” → “done”). aiogram has an FSM built on top of storage.

MemoryStorage / RedisStorage

Два варианта хранилища FSM в aiogram. Первое — в памяти процесса (исчезает при рестарте), второе — в Redis (переживает). Учебный этап на первом.

Two FSM storages in aiogram. First — in-process memory (lost on restart). Second — in Redis (survives). The learning etape uses the first.

Alembic

Инструмент миграций для SQLAlchemy. Каждое изменение схемы — отдельный файл с функциями upgrade() и downgrade(). Применяется к живой БД командой alembic upgrade head.

Migration tool for SQLAlchemy. Each schema change is a file with upgrade() and downgrade() functions. Applied to a live DB via alembic upgrade head.

APScheduler

Python-планировщик задач: cron-выражения, fixed-interval, one-shot. Подходит для «отправь дайджест в 9:00» внутри того же процесса бота, без внешнего сервиса.

A Python task scheduler: cron, fixed-interval, one-shot. Good for “send a digest at 9:00” inside the same bot process, no external service.

§ VI · Сам себе экзаменаторSelf-quiz

Семь вопросовSeven questions

Сначала ответьте про себя, потом раскройте — и нажмите «знал/не знал».

Answer in your head first, then open and click “knew / didn’t”.

  1. Почему мы не используем команду /login email password в боте? Why don’t we use a /login email password command in the bot?
    Пароль остаётся в истории чата у пользователя, в логах Telegram, в логах бота — всё равно, что записать его в блокнот. Правильно — deep-link с одноразовым токеном: пароль остаётся в вебе, бот получает только короткий «ключ» с истекающим сроком.
    The password lives in the user’s chat history, Telegram logs, bot logs — basically written down. Use deep-link with a one-time token instead: the password stays in the web, the bot gets only a short, expiring “key”.
  2. Чем worker-сервис на Railway отличается от web-сервиса? How does a worker service on Railway differ from a web service?
    Web слушает $PORT и проходит healthcheck. Worker не слушает порт, healthcheck отключён, в Procfile тип worker. Бот — worker, потому что он опрашивает Telegram, а не принимает HTTP-запросы.
    Web listens on $PORT and passes healthcheck. Worker doesn’t listen, healthcheck off, Procfile type worker. The bot is a worker because it polls Telegram rather than receiving HTTP.
  3. Что произойдёт с JWT пользователей при рестарте бота, если использовать MemoryStorage? What happens to user JWTs on a bot restart with MemoryStorage?
    Все JWT исчезнут — они жили в памяти процесса. При следующей команде пользователь увидит «привязка истекла, зайдите снова». Решение для production — RedisStorage: одна строка кода, переживает рестарт.
    All JWTs disappear — they were in process memory. On the next command the user sees “link expired”. Production solution — RedisStorage: one line of code, survives restarts.
  4. Почему колонку telegram_id в User делаем nullable, а не NOT NULL? Why is the telegram_id column on User nullable, not NOT NULL?
    У старых пользователей Telegram ещё не привязан, у них колонка должна быть пустой. Если поставить NOT NULL без default, миграция упадёт на первой же существующей строке. Nullable + unique — правильный компромисс.
    Existing users haven’t linked Telegram — the column must be empty for them. NOT NULL without a default would fail on the first existing row. Nullable + unique is the right compromise.
  5. Если кто-то завладел вашим TELEGRAM_BOT_TOKEN, что он может сделать прямо сейчас? If someone gets your TELEGRAM_BOT_TOKEN, what can they do right now?
    Притвориться вашим ботом и писать вашим пользователям — например, фишинговую ссылку «обнови привязку». Это страшно потому, что пользователи доверяют боту. Защита: /revoke в BotFather — старый токен умирает мгновенно, новый ставите в Railway env. Всё.
    Impersonate your bot and DM your users — e.g. a phishing “re-link” URL. Bad because users trust the bot. Defense: /revoke in BotFather — the old token dies instantly, set the new one in Railway env. Done.
  6. Зачем хранить часовой пояс отдельно от «часа отправки» в DigestSubscription? Why store time zone separately from “send hour” in DigestSubscription?
    Сервер на Railway может работать в UTC, и «9:00» без зоны интерпретируется как 9:00 UTC — в Москве это будет 12:00. Храните IANA-зону (Europe/Moscow), и APScheduler сам пересчитает.
    Railway runs in UTC, “9:00” without zone is 9:00 UTC — in Moscow that’s 12:00. Store an IANA zone (Europe/Moscow) and APScheduler does the math.
  7. Почему в проекте C мы завели отдельного бота через BotFather, а не использовали токен из проекта A? Why did project C use a separate bot via BotFather, not the project-A token?
    Один токен — один бот. Если запустить два процесса с одним токеном, Telegram отдаст событие случайному из них (или одному, отключив другого). Каждое приложение — свой бот, свой токен, свой @username.
    One token — one bot. Two processes with the same token: Telegram routes events to a random one (or kicks one off). Each app — its own bot, token, @username.
