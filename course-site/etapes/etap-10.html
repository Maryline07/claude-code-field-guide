Cron-расписание: как сказать машине «каждый день в 8:30»Cron expressions: telling the machine “every day at 8:30”

Cron-выражение — пять полей через пробел:

 ┌─────────── минута (0–59)
 │ ┌───────── час (0–23)
 │ │ ┌─────── день месяца (1–31)
 │ │ │ ┌───── месяц (1–12)
 │ │ │ │ ┌─── день недели (0–6, 0 = воскресенье)
 │ │ │ │ │
 30 8 * * *    # каждый день в 8:30every day at 8:30
 0 19 * * 5    # каждую пятницу в 19:00every Friday at 19:00
 0 9 1 * *     # первое число месяца в 9:001st of month at 9:00
 */15 * * * *  # каждые 15 минутevery 15 minutes

Запоминать не нужно. Скажите Claude «каждый рабочий день в 8:30» — он сам построит выражение. Ваше дело — понимать, что эти пять цифр означают «когда».

Часовые пояса. Cron на сервере живёт в часовом поясе сервера. Если сервер в UTC, а вам нужно «в 8:30 по Москве» — ставите 30 5 * * *. Лучшая практика — задать TZ в конфиге cron.

A cron expression is five space-separated fields:

 ┌─────────── minute (0–59)
 │ ┌───────── hour (0–23)
 │ │ ┌─────── day of month (1–31)
 │ │ │ ┌───── month (1–12)
 │ │ │ │ ┌─── day of week (0–6, 0 = Sunday)
 │ │ │ │ │
 30 8 * * *    # every day at 8:30
 0 19 * * 5    # every Friday at 19:00
 0 9 1 * *     # 1st of month at 9:00
 */15 * * * *  # every 15 minutes

You don’t need to memorise it. Tell Claude “every weekday at 8:30” — it builds the expression. Your job is to know those five digits mean “when.”

Timezones. Cron on a server lives in the server’s TZ. A server in UTC, you want “8:30 Moscow time” — set 30 5 * * *. Best practice — set TZ in the cron config.

Где живёт расписание: cron, облачный хостинг, schedule-skillWhere schedules live: cron, a cloud host, the schedule skill

Три варианта запустить «каждый день в 8:30»:

  1. Системный cron на вашем компьютере. Подходит, если компьютер всегда включён. Минус — если ноутбук закрыт, расписание не сработает.
  2. Облачный cron (Render Cron Jobs и т. п.). Сервис всегда поднят, расписание в настройках. Лучше всего для долгоживущих автоматизаций, но на бесплатных тарифах обычно недоступен.
  3. Skill schedule от Anthropic. Создаёт «remote agent», работающий в облаке Anthropic. Без вашего сервера. Идеально для учебных и личных задач.

В Claude Code есть две нативные возможности для долгих задач:

  1. run_in_background: true — запускает Bash-команду в фоне. Главный Claude получает ID процесса и продолжает работать. Можно проверить статус позже.
  2. Monitor — инструмент «следи за потоком». Подходит, чтобы дождаться завершения процесса (например, build на 15 минут): Claude засыпает, просыпается на каждой строке stdout, реагирует.

Когда полезно:

  • Долгий деплой — запустили в фоне, продолжаете другие задачи, проверяете через 5 минут;
  • Цепочка из нескольких subagents — пока один работает, готовите brief для следующего;
  • Локальный сервер — запустили uvicorn, переключились на тестирование через playwright.

> если всё ок — оставь сервер работать, мы будем тестировать.

Claude Code has two native facilities for long tasks:

  1. run_in_background: true — runs a Bash command in the background. Main Claude gets a process ID and keeps working. You can check status later.
  2. Monitor — a “follow the stream” tool. Good for waiting on a process (e.g., a 15-minute build): Claude sleeps, wakes on each stdout line, reacts.

When useful:

Идемпотентность: запустилось дважды — ничего не сломалосьIdempotency: ran twice — no damage

Cron в реальной жизни срабатывает не всегда «один раз»: бывают дубли (сервер перезапустился, расписание сработало два раза), пропуски (сервер был выключен), сдвиги по timezone.

Правило: каждая cron-задача должна быть идемпотентной. Запуск дважды — не ломает ничего.

Что это значит на практике:

  • Утренний отчёт может прислать дубль — неприятно, но не катастрофа.
  • «Удалить старые файлы» — не идемпотентно: первый запуск удалил, второй удалит ещё. Решение — в скрипте проверять «есть ли что удалять».
  • «Купить акцию по сигналу» — опасно: дубль = двойная покупка. Нужен механизм блокировки.

Дисциплина: перед каждой cron-задачей задайтесь вопросом «что произойдёт, если она сработает дважды подряд?». Если ответ «ничего страшного» — идём дальше. Если «деньги пропадут» — добавляем защиту.

Cron in real life is not always “exactly once”: duplicates (server restart, fired twice), misses (server was off), TZ shifts.

Rule: every cron task must be idempotent. Running twice — no damage.

In practice:

  • The morning digest may double — annoying, not a disaster.
  • “Delete old files” — not idempotent: first run deletes, second deletes more. Fix — check “is there anything to delete” in the script.
  • “Buy on signal” — dangerous: a duplicate = double purchase. Needs a lock.

Discipline: before each cron task ask “what happens if it fires twice in a row?” If “no big deal” — go ahead. If “money disappears” — add protection.

Дисциплина автопилота: четыре правилаAutopilot discipline: four rules

Перед тем как ставить задачу «работать самостоятельно», прогоните её по четырём правилам:

  1. Идемпотентность. Что произойдёт, если задача сработает дважды? Если плохо — добавьте защиту (флаг-файл, lock в БД, проверка «уже было сегодня»).
  2. Граница времени. Сколько максимум может длиться задача? Если больше N минут — убивать. Иначе застрявший процесс будет жечь ресурсы.
  3. Кнопка «стоп». Как быстро остановить автоматизацию, если что-то пошло не так? Команда /stop_autopilot в вашем Telegram-боте — хорошая практика.
  4. Канал ошибок. Куда уходят сообщения «сбой»? Если только в логи — вы их не увидите. Хороший шаблон: ошибки приходят в тот же Telegram-чат, что и успехи, но с маркером ⚠.

Без этих правил «автопилот» превращается в «чёрную дыру», в которой что-то происходит, но вы не знаете что. С ними — вы контролируете автономный процесс с любого экрана.

Before sending a task to “run on its own,” walk it through four rules:

  1. Idempotency. What happens if it fires twice? If bad — add protection (flag file, DB lock, “already done today” check).
  2. Time bound. Max duration? If above N minutes — kill it. Otherwise a stuck process burns resources forever.
  3. “Stop” button. How fast can you halt it if it goes off the rails? A /stop_autopilot command in your Telegram bot is good practice.
  4. Error channel. Where do “failure” messages go? Only logs — you’ll never see them. Pattern: errors land in the same Telegram chat as successes, but with a ⚠ marker.

Without these rules, “autopilot” becomes a black hole — something happens, you don’t know what. With them — you control an autonomous process from any screen.

Семь вопросов на закреплениеSeven questions for reinforcement

  1. Что означает cron-выражение 30 8 * * 1-5?What does the cron expression 30 8 * * 1-5 mean?

    показать ответshow answer

    «Каждый рабочий день (понедельник — пятница) в 8:30». Поля: минута 30, час 8, любой день месяца, любой месяц, дни недели 1–5.

    “Every weekday (Mon–Fri) at 8:30.” Fields: minute 30, hour 8, any day-of-month, any month, weekdays 1–5.

  2. Где лучше держать cron-задачи: на своём ноутбуке, в облачном cron или через skill schedule? Почему?Where to keep cron tasks: on your laptop, in a cloud cron, or via the schedule skill? Why?

    показать ответshow answer

    Зависит. Ноутбук — только если он всегда включён (редко). Облачный cron (например, Render Cron Jobs) — надёжно, но обычно платно. schedule skill — самое простое для учебных и личных задач, не нужен свой сервер и бесплатно. В курсе бесплатные пути — системный cron и schedule.

    Depends. Laptop — only if always-on (rarely). Cloud cron (e.g. Render Cron Jobs) — reliable but usually paid. The schedule skill — simplest for learning and personal tasks, no server, free. The free paths in this course are system cron and schedule.

  3. Что такое идемпотентность и почему она обязательна для cron-задач?What is idempotency and why is it mandatory for cron tasks?

    показать ответshow answer

    Идемпотентность — задача можно запустить дважды, и результат тот же, ничего не ломается. Обязательна, потому что cron реально срабатывает дважды (перезапуск сервера, сдвиги timezone). «Купить акцию по сигналу» неидемпотентно: дубль = двойная покупка. Защита — флаг-файл «уже сделано».

    Idempotency — a task can fire twice, same result, no damage. Mandatory because cron really does double-fire (server restart, TZ shifts). “Buy on signal” is not idempotent: a duplicate = double buy. Defence — a flag file “already done.”

  4. Чем отличается background task от cron-задачи?How does a background task differ from a cron task?

    показать ответshow answer

    Background task — запуск программы в фоне внутри текущей сессии, она работает «параллельно» с Claude. Cron — запуск по расписанию, в будущем, без вашего участия. Background для «сейчас, но не блокирует», cron для «регулярно».

    Background task — runs a program in the background of the current session, “in parallel” with Claude. Cron — runs on schedule, in the future, unattended. Background = “now, but non-blocking,” cron = “recurring.”

  5. Почему обязательно подключать PostHog к автопилоту?Why connect PostHog to your autopilot?

    показать ответshow answer

    Чтобы видеть, во что обходится автопилот: токены, ошибки, длительность, тренды. Без мониторинга одна сломавшаяся задача может тихо жечь деньги недели, прежде чем заметите. PostHog — глаза, без которых автономный процесс «слепой».

    To see what your autopilot costs: tokens, errors, latency, trends. Without monitoring, one broken task can silently burn money for weeks before you notice. PostHog is the eyes — without it the autonomous process is blind.

  6. Почему в «Дневнике на автопилоте» содержательные поля шифруются, а в PostHog уходят только метрики?Why are content fields encrypted in the “Diary” while only metrics go to PostHog?

    показать ответshow answer

    Содержание дневника — чувствительные личные данные. Они не должны быть доступны никому, кроме вас. Шифрование с ключом в env гарантирует, что даже если БД утечёт — данные не прочитают. В PostHog отправляем только обезличенные метрики (количество ответов, длительность, ошибки) — они полезны для мониторинга, но не утекают в стороннюю аналитику ваши мысли.

    Diary content is sensitive personal data. Nobody but you should access it. Encryption with the key in env ensures that even a DB leak yields unreadable data. PostHog gets only anonymised metrics (response counts, latencies, errors) — useful for monitoring without leaking your thoughts to third-party analytics.

  7. Какие четыре правила «дисциплины автопилота»?What are the four “autopilot discipline” rules?

    показать ответshow answer

    1) Идемпотентность — повторный запуск не ломает. 2) Граница времени — задача убивается, если работает дольше N минут. 3) Кнопка стоп — одна команда останавливает всё. 4) Канал ошибок — ошибки приходят туда же, куда успехи (например, в Telegram), с маркером ⚠.

    1) Idempotency — re-running breaks nothing. 2) Time bound — task killed if it runs over N minutes. 3) Stop button — one command halts everything. 4) Error channel — errors land in the same channel as successes (Telegram), with a ⚠ marker.
