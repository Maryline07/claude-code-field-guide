§ I · ТемыTopics

Как Claude получает «новые навыки»How Claude gains “new abilities”

MCP изнутри: что отдаёт сервер и что получает ClaudeMCP from the inside: what the server returns, what Claude receives

В этапе 03 мы ставили готовые MCP. Сейчас разбираемся, что именно происходит внутри. У MCP-сервера три типа «выдач»:

  1. Tools — функции, которые Claude может вызвать. Например, get_price(ticker: str) возвращает текущую цену акции. Имя, параметры, описание — всё это сервер сообщает Claude при подключении.
  2. Resources — «документы», которые Claude может прочитать. Например, portfolio://my-current-positions — ваш текущий портфель. Сервер отдаёт содержимое по запросу.
  3. Prompts — готовые шаблоны промптов от сервера. Полезно, чтобы пользователю было проще «нажать кнопку», а не писать промпт с нуля.

Минимум для нашего «Daily Trader» MCP — четыре tool'а:

  • get_price(ticker) — текущая цена акции (через Alpha Vantage);
  • get_history(ticker, days) — история за N дней;
  • compute_signal(ticker) — технический сигнал (например, простое moving average crossover);
  • set_alert(ticker, threshold) — записать алерт в файл, чтобы хук этапа 10 проверил утром.

On etape 03 we installed ready MCPs. Now we look inside. An MCP server returns three kinds of things:

  1. Tools — functions Claude can call. E.g. get_price(ticker: str) returns the current stock price. Name, parameters, description — all this the server tells Claude on connect.
  2. Resources — “documents” Claude can read. E.g. portfolio://my-current-positions — your current portfolio. The server returns the content on request.
  3. Prompts — ready prompt templates from the server. Useful so users “press a button” instead of writing prompts from scratch.

Minimum for our “Daily Trader” MCP — four tools:

  • get_price(ticker) — current stock price (via Alpha Vantage);
  • get_history(ticker, days) — history for N days;
  • compute_signal(ticker) — a technical signal (e.g., simple moving-average crossover);
  • set_alert(ticker, threshold) — write an alert to a file for the etape-10 hook to read in the morning.
Transports: как сервер общается с ClaudeTransports: how the server talks to Claude

Четыре способа соединения:

  • stdio — через стандартный ввод-вывод процесса. Самый простой и частый. Сервер запускается локально, общается с Claude через stdin/stdout.
  • http — по HTTP. Сервер живёт на удалённой машине.
  • sse — Server-Sent Events. Тоже по HTTP, но со стримингом.
  • websocket — двусторонний канал по WebSocket.

Для учебного MCP, который ходит к Alpha Vantage, выбираем stdio: процесс запускается локально, никаких серверов поднимать не надо. Sublime для старта.

Когда понадобится http/sse: если ваш MCP делится между несколькими пользователями (например, корпоративный) или его нужно вызывать из web-интерфейса. Это уже второй проект для вашей команды — не этого этапа.

Four ways to connect:

  • stdio — through the process’s stdin/stdout. The simplest and most common. The server runs locally, talks to Claude via standard streams.
  • http — over HTTP. The server lives on a remote machine.
  • sse — Server-Sent Events. Also HTTP, with streaming.
  • websocket — full duplex over WebSocket.

For a learning MCP hitting Alpha Vantage, pick stdio: the process runs locally, no servers needed. Perfect to start.

You’ll want http/sse later: when the MCP is shared (e.g. corporate), or when you call it from a web app. That’s a second project for your team — not this etape.

Плагин: упаковка skills + commands + hooks + MCP в одноPlugin: bundling skills + commands + hooks + MCP in one

До этого мы ставили skills и MCP отдельно. Это работает, но неудобно: пользователь должен помнить, какие из них «работают вместе».

Плагин — это бандл. Один пакет, в котором лежат:

  • skills/ — ваши skills (как в этапе 08a);
  • commands/ — ваши slash-команды;
  • hooks/ — ваши hooks с настройками;
  • mcp/ — ваш MCP-сервер;
  • plugin.json — манифест: что есть в плагине, какие версии, описание.
my-personal-office/

├── README.md
├── skills/

│   ├── personal-cfo/
│   └── auto-publisher/

├── hooks/
│   └── settings.json     # hook на SessionStart/End
└── mcp/
    └── daily-trader/

        └── server.ts

Публикация на Smithery: один git push — и ваш плагин в магазинеPublishing on Smithery: one git push and your plugin is in the store

Smithery подцепляется к вашему публичному GitHub-репо с plugin.json. Шаги:

  1. Залить ваш плагин в публичный репо my-personal-office.
  2. На smithery.ai в вашем профиле — «Submit a plugin», указать ссылку на репо.
  3. Smithery валидирует plugin.json, проверяет, что зависимости работают.
  4. Через 5–10 минут плагин виден на витрине, ставится одной командой:
$ npx @smithery/cli@latest install @<you>/personal-office --client claude

Что важно:

  • Никаких секретов в коммитах. ANTHROPIC_API_KEY, ALPHAVANTAGE_KEY и т. п. требуйте через ENV. plugin.json в mcp_servers[].env перечисляет, какие переменные нужны — Smithery предложит пользователю их ввести.
  • Семантическое версионирование. 1.0.0 → 1.0.1 (багфикс) → 1.1.0 (фича) → 2.0.0 (breaking). Каждый push с новой версией — автоматический re-publish.
  • README обязателен. С примерами «что делать» и описанием каждого инструмента.
  • Лицензия. MIT — самый частый выбор для open-source плагинов.

Smithery hooks into your public GitHub repo with a plugin.json. Steps:

  1. Push your plugin to the public my-personal-office repo.
  2. On smithery.ai in your profile — “Submit a plugin,” point to the repo.
  3. Smithery validates the plugin.json, checks deps work.
  4. 5–10 minutes later your plugin shows up in the store, installable with one command:
$ npx @smithery/cli@latest install @<you>/personal-office --client claude

What matters:

  • No secrets in commits. Require ANTHROPIC_API_KEY, ALPHAVANTAGE_KEY etc. via ENV. plugin.json declares them in mcp_servers[].env — Smithery prompts the user.
  • Semver. 1.0.0 → 1.0.1 (bugfix) → 1.1.0 (feature) → 2.0.0 (breaking). Each push with a new version — auto re-publish.
  • README required. Examples of usage and descriptions for each tool.
  • Licence. MIT is the most common pick for open-source plugins.
Безопасность плагинов: «доверяй, но проверяй»Plugin safety: trust but verify

Плагин получает доступ ко всем инструментам Claude Code. Это опасно, если вы ставите чужой плагин и не знаете, что внутри.

Что делает Smithery для безопасности:

  • Все плагины — в публичных GitHub-репо, можно прочитать код.
  • При установке — явно показывается, какие env переменные плагин запрашивает.
  • Сообщества помечают «verified» статусы для плагинов от известных авторов.

Что должны делать вы:

  • Перед установкой — зайдите в репо плагина, прочитайте plugin.json, посмотрите mcp/server.ts;
  • Если плагин запрашивает доступ к вашим API-ключам или файлам — понимайте, зачем;
  • Свежие плагины от неизвестных авторов — ставьте в тестовой папке, не в основном проекте.

Особое внимание: hook на PreToolUse может перехватить любой ваш Bash. Хук на UserPromptSubmit — видит каждое ваше сообщение. Не ставьте плагин с этими хуками вслепую.

A plugin gets access to all Claude Code tools. That’s dangerous if you install a stranger’s plugin without checking.

Smithery’s safety:

  • All plugins are in public GitHub repos — you can read the code.
  • On install — env variables the plugin requests are shown explicitly.
  • The community flags “verified” statuses for known-author plugins.

Your part:

  • Before installing — open the plugin’s repo, read plugin.json, look at mcp/server.ts;
  • If the plugin asks for API keys or files — understand why;
  • Fresh plugins from unknown authors — install in a sandbox folder, not your main project.

Especially: a PreToolUse hook can intercept any Bash you run. A UserPromptSubmit hook sees every message. Don’t install a plugin with those blindly.

MCP «Daily Trader»“Daily Trader” MCP

Цель: MCP с четырьмя tool'ами (get_price, get_history, compute_signal, set_alert) и двумя resources (portfolio://current-positions, alerts://pending). Подключаемый локально, готовый к упаковке в плагин (проект B).

Goal: an MCP with four tools (get_price, get_history, compute_signal, set_alert) and two resources (portfolio://current-positions, alerts://pending). Locally hooked up, ready to bundle into a plugin (project B).

Шаги

Steps

  1. Создайте папку daily-trader-mcp. Через Plan Mode попросите spec и план реализации.
  2. Реализация четырёх tool'ов:
    • get_price(ticker) — вызывает Alpha Vantage GLOBAL_QUOTE, возвращает { ticker, price, change_pct, currency, as_of };
    • get_history(ticker, days) — TIME_SERIES_DAILY, возвращает массив { date, close };
    • compute_signal(ticker) — считает MA-5 и MA-20, возвращает { signal: "buy" | "sell" | "hold", reason };
    • set_alert(ticker, threshold, direction) — пишет в ~/.claude/alerts.json.
  3. Реализация двух resources:
    • portfolio://current-positions — читает ~/portfolio-journal/data/operations.csv, считает текущие позиции;
    • alerts://pending — читает ~/.claude/alerts.json.
  4. Кэширование цен: ~/.claude/.daily-trader-cache/ с TTL 15 мин (как в этапе 03).
  5. Обработка ошибок: rate limit, отсутствие ключа, недоступная сеть — везде дружелюбные сообщения.

§ VI · Quiz

Семь вопросов на закреплениеSeven questions for reinforcement

  1. Чем плагин отличается от «просто MCP» и «просто skill»?How does a plugin differ from a “plain MCP” and a “plain skill”?

    показать ответshow answer

    Плагин — бандл: skills + commands + hooks + MCP в одной упаковке. Один плагин — одна команда установки. Просто MCP — только сервер с tools. Просто skill — только инструкция, без tools и hooks. Плагин — «продукт», MCP/skill — «компоненты».

    A plugin is a bundle: skills + commands + hooks + MCP in one package. One plugin — one install. A plain MCP — just a server with tools. A plain skill — just instructions, no tools or hooks. Plugin = “product,” MCP/skill = “components.”

  2. Какой transport выбрать для учебного MCP, и почему?Which transport for a learning MCP, and why?

    показать ответshow answer

    stdio. Сервер запускается локально, общается с Claude через стандартный ввод-вывод — никаких сетей, портов, SSL. Самый простой и надёжный вариант. http/sse/websocket нужны, когда сервер общий и удалённый.

    stdio. The server runs locally, talks to Claude via stdin/stdout — no networks, ports, SSL. Simplest and most reliable. http/sse/websocket are for shared, remote servers.

  3. Что такое resources в MCP, и чем они отличаются от tools?What are MCP resources, and how do they differ from tools?

    показать ответshow answer

    Tools — действия: функции, которые Claude вызывает с параметрами и получает результат (set_alert). Resources — документы: «открой по этому URL» (portfolio://current). Resources идут в контекст, как файлы; tools — вызываются ради побочного эффекта или вычисления.

    Tools — actions: functions Claude calls with parameters and gets a result (set_alert). Resources — documents: “open this URL” (portfolio://current). Resources flow into context like files; tools are called for side effects or computation.

  4. Почему секретные ключи нельзя класть в plugin.json?Why can secret keys not live in plugin.json?

    показать ответshow answer

    plugin.json публичен — лежит в вашем репо, который видят все. Любой ключ оттуда утечёт за часы. Правильно: в plugin.json в mcp_servers[].env перечислить имена переменных (ALPHAVANTAGE_KEY), а значения пользователь вводит при установке — они хранятся локально, не уходят в Smithery.

    plugin.json is public — it lives in your repo, visible to all. Any key there leaks within hours. Correct: in plugin.json’s mcp_servers[].env list names (ALPHAVANTAGE_KEY); the user enters values on install — they live locally, not on Smithery.

  5. Что в semver 1.0.0 → 2.0.0 — багфикс или ломающее изменение?In semver, is 1.0.0 → 2.0.0 a bugfix or a breaking change?

    показать ответshow answer

    MAJOR — ломающее. Если вы поменяли имя tool, выкинули resource, изменили формат данных — это 2.0.0. Багфикс — 1.0.0 → 1.0.1. Новая фича без ломки старых — 1.0.0 → 1.1.0.

    MAJOR — breaking. If you renamed a tool, dropped a resource, changed a data shape — that’s 2.0.0. Bugfix — 1.0.0 → 1.0.1. New feature without breaking old — 1.0.0 → 1.1.0.

  6. Как проверить, что плагин «по-настоящему» работает у другого пользователя?How do you verify the plugin “really works” for another user?

    показать ответshow answer

    Поставить его в чистом окружении (другая папка, другой компьютер, другой OS) и пройти базовый сценарий: /skills и /mcp — всё на месте; вызвать каждую slash-команду и инструмент MCP — работает. Если у себя «у меня всё работало», а на чистой машине — нет, обычно проблема в неуказанной зависимости или невыставленной env-переменной.

    Install in a clean environment (another folder, another machine, another OS) and walk a baseline: /skills and /mcp — all there; call each slash command and MCP tool — works. If it “worked on my machine” but not on a clean one, it’s usually a missing dependency or unset env variable.

  7. Что особенно опасно в чужих плагинах, и как защититься?What is especially dangerous in third-party plugins, and how do you defend?

    показать ответshow answer

    Hooks на PreToolUse (могут перехватить любой Bash) и UserPromptSubmit (видят каждое ваше сообщение). Также — MCP-серверы с запросом доступа к вашим ключам или файловой системе. Защита: читайте plugin.json и server.ts перед установкой; ставьте сначала в тестовой папке; для неизвестных авторов — смотрите «verified» статус и количество звёзд на GitHub.

    PreToolUse hooks (can intercept any Bash) and UserPromptSubmit hooks (see every message). Also — MCP servers asking for your keys or filesystem. Defence: read plugin.json and server.ts before install; sandbox-install first; for unknown authors — check “verified” status and GitHub stars.
