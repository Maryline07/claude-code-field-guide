§ I · ТемыTopics

Из CLI — в «Claude как библиотека»From CLI to “Claude as a library”

Claude Code, Claude API и Claude Agent SDK — в чём разницаClaude Code, Claude API, Claude Agent SDK — the difference

Три способа использовать Claude:

  1. Claude Code (CLI) — интерактивный диалог в терминале. Это всё, что мы делали с этапа 00 по 10. Удобно для работы человека с Claude.
  2. Claude API (chat completions) — HTTP-вызов «вход → текст». Этап 06: ваш backend шлёт промпт, получает ответ. Удобно для встраивания одного ответа внутрь приложения. Без tools, без цикла.
  3. Claude Agent SDK — полный «агентский» цикл из вашего скрипта: prompt → tool calls → tool results → next prompt → ... до завершения. С agent loop, tools, prompt caching, выбор модели.

Когда что:

СценарийЧто использовать
Я разрабатываю проект, направляя ClaudeClaude Code
Веб-приложение шлёт один промпт пользователюClaude API
Скрипт-демон ловит данные, использует tools, шлёт результатAgent SDK
CI/CD: автоматический code review на каждом PRAgent SDK

Three ways to use Claude:

  1. Claude Code (CLI) — interactive dialogue in the terminal. Everything we did from etape 00 through 10. Best for a human working with Claude.
  2. Claude API (chat completions) — an HTTP “input → text” call. Etape 06: your backend sends a prompt, gets a reply. Best for embedding a single reply inside an app. No tools, no loop.
  3. Claude Agent SDK — the full agent loop from your script: prompt → tool calls → tool results → next prompt → ... until done. With an agent loop, tools, prompt caching, model selection.

Agent loop: внутренний цикл агента в кодеAgent loop: the agent’s loop in your code

Когда вы пишете промпт в Claude Code и Claude вызывает tools — цикл крутится внутри Claude Code, вы его не видите. В Agent SDK этот же цикл — ваш собственный код:

messages = [{"role": "user", "content": "проанализируй акцию AAPL"}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        tools=my_tools,        # список tools, которыми может пользоваться
        messages=messages,
    )
    messages.append({"role": "assistant", "content": response.content})

    if response.stop_reason == "end_turn":
        # модель закончила
        break

    if response.stop_reason == "tool_use":
        # модель просит вызвать tool
        for tool_call in response.content:
            if tool_call.type == "tool_use":
                result = execute_tool(tool_call.name, tool_call.input)
                messages.append({"role": "user", "content": [
                    {"type": "tool_result", "tool_use_id": tool_call.id, "content": result}
                ]})
        # цикл повторяется: модель видит tool_result и продолжает

Это и есть «agent loop». Claude Code прячет его, SDK — даёт в руки. Вы контролируете каждую итерацию: можете остановить раньше, поменять tools на ходу, добавить логи, тайм-ауты, защиту.

Skill claude-api делает шаблон ещё лучше: добавляет prompt caching, retry на ошибках, ограничение по шагам цикла, защиту от «infinite tool call».

When you prompt in Claude Code and Claude uses tools, the loop runs inside Claude Code — you don’t see it. In the Agent SDK that same loop is your own code:

messages = [{"role": "user", "content": "analyse stock AAPL"}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        tools=my_tools,        # tool list the model may use
        messages=messages,
    )
    messages.append({"role": "assistant", "content": response.content})

    if response.stop_reason == "end_turn":
        # the model is done
        break

    if response.stop_reason == "tool_use":
        # the model wants a tool call
        for tool_call in response.content:
            if tool_call.type == "tool_use":
                result = execute_tool(tool_call.name, tool_call.input)
                messages.append({"role": "user", "content": [
                    {"type": "tool_result", "tool_use_id": tool_call.id, "content": result}
                ]})
        # loop repeats: the model sees tool_result and continues

That is the agent loop. Claude Code hides it; the SDK hands it to you. You control each iteration: stop early, swap tools mid-flight, add logs, timeouts, guards.

The claude-api skill makes the template even better: prompt caching, retries on errors, step caps, “infinite tool call” guards.

Headless mode и CI/CD: Claude как «функция»Headless mode and CI/CD: Claude as a “function”

Headless mode — запуск Claude в окружении без интерактивного терминала. Сервер, CI-pipeline, daemon — никто не ждёт ваших ответов на вопросы Claude.

Что это меняет:

  • Никаких «Claude спросил подтверждение, я нажал y». Все разрешения — в коде заранее.
  • Все секреты — через env vars, не через интерактивный prompt.
  • Тайм-ауты обязательны. Зависший агент в CI — падает билд.
  • Логирование — обязательное и структурированное (JSON), чтобы потом разобрать.

Пример — GitHub Action для auto code review:

# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - run: pip install anthropic
      - run: python .github/scripts/claude_review.py
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

Скрипт claude_review.py через Agent SDK:

  1. Получает diff PR через GitHub API;
  2. Шлёт в Claude с промптом «найди 3–5 проблем»;
  3. Через tool post_comment — постит замечания прямо в PR.

Headless mode — running Claude in an environment with no interactive terminal. A server, a CI pipeline, a daemon — nobody waits to answer Claude’s questions.

What changes:

  • No “Claude asked, I pressed y.” All permissions in code, upfront.
  • Secrets — via env vars, not interactive prompts.
  • Timeouts mandatory. A stuck agent in CI fails the build.
  • Logging — mandatory and structured (JSON), to parse later.

Example — a GitHub Action for auto code review:

# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - run: pip install anthropic
      - run: python .github/scripts/claude_review.py
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

The claude_review.py script via the Agent SDK:

  1. Pulls the PR diff via GitHub API;
  2. Sends to Claude with “find 3–5 issues” prompt;
  3. Via a post_comment tool — posts comments straight into the PR.
Tool definitions для SDK: не MCP, а Python-функцииTool definitions in the SDK: not MCP, but Python functions

В Claude Code мы давали Claude tools через MCP. В SDK — через определения прямо в коде:

my_tools = [
    {
        "name": "get_price",
        "description": "Возвращает текущую цену акции по тикеру.",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {"type": "string", "description": "тикер, e.g. AAPL"}
            },
            "required": ["ticker"]
        }
    },
    {
        "name": "post_telegram",
        "description": "Шлёт сообщение пользователю в Telegram.",
        "input_schema": {
            "type": "object",
            "properties": {
                "text": {"type": "string"}
            },
            "required": ["text"]
        }
    }
]

Когда Claude решает вызвать tool, ваш цикл получает tool_use с именем и аргументами — и вы сами вызываете соответствующую Python-функцию. То есть SDK не интегрируется с MCP — tools определяются и исполняются у вас в коде.

Зачем разница: в CI или daemon — нет процесса для MCP-сервера, нет конфига Claude Code. SDK даёт самодостаточный, переносимый агент в одном файле.

In Claude Code we gave Claude tools via MCP. In the SDK — via definitions in code:

my_tools = [
    {
        "name": "get_price",
        "description": "Returns the current stock price by ticker.",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {"type": "string", "description": "ticker, e.g. AAPL"}
            },
            "required": ["ticker"]
        }
    },
    {
        "name": "post_telegram",
        "description": "Sends a Telegram message to the user.",
        "input_schema": {
            "type": "object",
            "properties": {
                "text": {"type": "string"}
            },
            "required": ["text"]
        }
    }
]

When Claude decides to call a tool, your loop receives a tool_use with name and args — and you invoke the matching Python function. So the SDK does not integrate with MCP — tools are defined and executed in your code.

Why: in CI or a daemon there’s no MCP-server process, no Claude Code config. The SDK gives a self-contained portable agent in a single file.

Prompt caching и batch API: оптимизация стоимостиPrompt caching and the batch API: cost optimisation

Prompt caching — вы уже видели в этапе 06. В SDK активируется одним полем:

response = client.messages.create(
    model="claude-sonnet-4-6",
    system=[
        {"type": "text", "text": "Большой системный промпт...",
         "cache_control": {"type": "ephemeral"}}
    ],
    messages=messages,
)

Эта часть кэшируется на 5 минут. На повторных вызовах с тем же системным промптом стоимость падает в 10 раз. Идеально для Twitter-демона, где каждый твит идёт через один и тот же «инструктаж».

Batch API — асинхронный режим: отправляете 100 промптов одной пачкой, ответы готовы через ~30 минут, цена в 2 раза ниже. Подходит для «не срочных» задач: ночная классификация всех вчерашних твитов, переоценка портфеля раз в день, обработка длинной очереди задач.

Не подходит — для интерактивных бот-задач, где пользователь ждёт ответ.

Prompt caching — you saw it in etape 06. In the SDK it’s one field:

response = client.messages.create(
    model="claude-sonnet-4-6",
    system=[
        {"type": "text", "text": "A long system prompt...",
         "cache_control": {"type": "ephemeral"}}
    ],
    messages=messages,
)

This section is cached for 5 minutes. On repeats with the same system prompt the cost drops 10×. Ideal for the Twitter daemon, where every tweet runs through the same “briefing.”

Batch API — async mode: you ship 100 prompts in one batch, answers in ~30 minutes, 2× cheaper. Fits “not urgent” jobs: nightly classification of yesterday’s tweets, daily portfolio rescore, long task queues.

Does not fit — interactive bot tasks where the user is waiting.

Безопасность headless: что отличается от Claude CodeHeadless safety: how it differs from Claude Code

В Claude Code вы нажимаете «yes/no» на каждое опасное действие. В headless такого нет — всё решается заранее. Что обязательно:

  1. Список разрешённых tools — узкий. В Twitter-демоне нет смысла давать tool «удалить файл». Ограничьте только теми, что нужны.
  2. Ограничение шагов цикла. max_iterations = 10. Иначе при ошибке агент будет крутиться вечно, жгя токены.
  3. Тайм-аут на запрос. 60 секунд — и убиваем. В CI это критично: иначе билд висит.
  4. Аудит-лог. Каждое действие — в лог-файл с timestamp. PostHog (этап 10) для агрегации.
  5. Бюджет. На запуск и на день. Если превысили — останавливаемся, шлём алерт.
  6. Sandboxing. Если агент пишет в файлы — пусть пишет в tmp/, а не в корень репо. В CI — в artifact-папку.

В Claude Code эти правила «по умолчанию». В SDK — ваша ответственность.

In Claude Code you press “yes/no” for every dangerous action. Headless has none — everything is decided upfront. Mandatory:

  1. Narrow allowed-tools list. A Twitter daemon doesn’t need a “delete file” tool. Restrict to what’s needed.
  2. Loop step cap. max_iterations = 10. Otherwise on an error the agent spins forever, burning tokens.
  3. Per-request timeout. 60 seconds — kill. Critical in CI: builds would hang otherwise.
  4. Audit log. Every action — into a log file with a timestamp. PostHog (etape 10) for aggregation.
  5. Budget. Per run and per day. Over budget — stop, alert.
  6. Sandboxing. If the agent writes files — into tmp/, not the repo root. In CI — into an artifact folder.

Claude Code applies these by default. In the SDK — your responsibility.

Долговременная память агента: разговор «с продолжением»Long-term agent memory: a conversation with continuity

Агент в скрипте по умолчанию забывает всё после break. Для Telegram-консьержа нужно «помнить вас»: что любите, что искали вчера, какие задачи стоят.

Шаблоны:

  1. В БД — conversations(user_id, messages_json). Загружаете при старте сессии, сохраняете перед выходом.
  2. В файле на диск — ~/.concierge/<user_id>.json. Простое, для одного пользователя.
  3. Через summarisation — когда история длинная, прогоняете её через Claude и получаете «сжатую память». Дальше работаете с summary, а не с полной историей. Экономит токены.

Дисциплина:

  • Не храните пароли и ключи в истории сообщений — это утечка ждущая случиться;
  • Шифруйте, если хранится в облаке (этап 10, дневник на автопилоте);
  • Дайте пользователю команду /forget — удалить всю историю одной кнопкой. GDPR.

The script-agent forgets everything after break by default. A Telegram concierge needs to “remember you”: what you like, what you searched yesterday, what tasks are pending.

Patterns:

  1. DB — conversations(user_id, messages_json). Load at session start, save before exit.
  2. Disk file — ~/.concierge/<user_id>.json. Simple, single-user.
  3. Via summarisation — when the history is long, run it through Claude for a “compressed memory.” Then work with the summary, not the full history. Saves tokens.

Discipline:

  • Never store passwords or keys in message history — a leak waiting to happen;
  • Encrypt if cloud-stored (etape 10, the diary);
  • Give the user a /forget command to wipe history in one tap. GDPR.

Claude Agent SDK
Библиотека Anthropic (anthropic для Python, @anthropic-ai/sdk для Node), позволяющая запускать Claude как агента из вашего скрипта. Полный agent loop, tools, prompt caching — всё в ваших руках. An Anthropic library (anthropic for Python, @anthropic-ai/sdk for Node) that lets you run Claude as an agent from your script. Full agent loop, tools, prompt caching — all yours.
Agent loop
Внутренний цикл агента: prompt → tool calls → tool results → next prompt → ... до stop_reason: end_turn. В Claude Code скрыт, в SDK — ваш код. The agent’s inner loop: prompt → tool calls → tool results → next prompt → ... until stop_reason: end_turn. Hidden in Claude Code, your code in the SDK.
Headless mode
Запуск Claude без интерактивного TTY: на сервере, в CI, в daemon-скрипте. Все разрешения, секреты, тайм-ауты — в коде заранее. Running Claude without an interactive TTY: server, CI, daemon script. All permissions, secrets, timeouts — in code upfront.
Tool definition
Описание tool в SDK: имя, description, JSON-schema параметров. Передаётся в messages.create(tools=...). Когда модель решает вызвать tool — ваш код исполняет соответствующую функцию. A tool description for the SDK: name, description, JSON-schema parameters. Passed to messages.create(tools=...). When the model decides to call — your code runs the matching function.
Daemon
Долгоживущий процесс, постоянно работающий в фоне. На Railway — «worker»-сервис. Используется для стримов (Twitter), ботов (Telegram), мониторов. A long-running process always alive in the background. On Railway — a “worker” service. Used for streams (Twitter), bots (Telegram), monitors.
Batch API
Асинхронный режим Anthropic API: 100 промптов одной пачкой, ответы через ~30 минут, в 2 раза дешевле. Подходит для несрочных задач (ночная классификация, переоценка портфеля). Async Anthropic API mode: 100 prompts in one batch, replies in ~30 minutes, 2× cheaper. Fits non-urgent jobs (nightly classification, portfolio rescore).
CI/CD
Continuous Integration / Continuous Deployment — автоматизация сборки, тестов и деплоя. В курсе — через GitHub Actions: на каждый push/PR запускается набор скриптов, в т. ч. Claude code review. Continuous Integration / Continuous Deployment — build, test and deploy automation. Here — via GitHub Actions: on every push/PR a script set runs, including Claude code review.
GitHub Actions
Встроенная CI-система GitHub. Workflow описывается YAML-файлом в .github/workflows/. На триггер — запускает шаги: setup среды, выполнение скриптов, постинг результатов. GitHub’s built-in CI. Workflows are YAML files in .github/workflows/. On a trigger — runs steps: env setup, scripts, result posting.
max_iterations
Ограничение количества шагов agent loop. Без него агент при ошибке может крутиться вечно. Стандартно — 5–10 для большинства задач. A cap on agent-loop steps. Without it, on an error the agent loops forever. Default — 5–10 for most tasks.

    Когда вам нужно один ответ на один промпт, без tools и без цикла. Например: чат-приложение, которое получает вопрос и возвращает ответ. Claude API проще, дешевле и не требует определять tools.

    When you need one reply to one prompt, no tools, no loop. E.g., a chat app that takes a question and returns an answer. Claude API is simpler, cheaper, no tool definitions.

  2. Что делает stop_reason: tool_use в agent loop?What does stop_reason: tool_use mean in the agent loop?

    показать ответshow answer

    Это сигнал: «модель просит тебя вызвать tool». Ваш код должен исполнить запрошенные tool-функции, передать результаты обратно как tool_result, и продолжить цикл — модель увидит результаты и либо вызовет ещё tool, либо завершит с end_turn.

    It’s the signal “model asks you to call a tool.” Your code runs the requested tool functions, sends results back as tool_result, and continues the loop — the model sees the results and either calls another tool or finishes with end_turn.

  3. Зачем обязательно ограничение max_iterations в headless?Why is a max_iterations cap mandatory headless?

    показать ответshow answer

    При ошибке (плохой tool, неудачный prompt, цикл из «попробуй снова») агент может крутиться вечно, жгя токены. В Claude Code пользователь нажмёт Esc — в headless этого нет. max_iterations — страховка: после N шагов — принудительный выход с ошибкой, которую можно увидеть в логах.

    On an error (bad tool, unfortunate prompt, “try again” loop) the agent can spin forever, burning tokens. In Claude Code the user hits Esc — headless has none. max_iterations is the insurance: after N steps — a forced exit with an error visible in logs.

  4. Где живут tools в Agent SDK — в MCP-сервере или в вашем коде?Where do tools live in the Agent SDK — in an MCP server or in your code?

    показать ответshow answer

    В вашем коде. SDK — это агент в скрипте, без Claude Code-обёртки и без конфигов MCP. Tools определяются как Python-функции (или Node) и передаются в messages.create(tools=...). Это даёт переносимость — один файл-скрипт работает в CI, на сервере, в любом окружении.

    In your code. The SDK is an agent in a script, no Claude Code wrapper, no MCP configs. Tools are Python (or Node) functions passed to messages.create(tools=...). This gives portability — one script file runs in CI, on a server, anywhere.

  5. Когда уместен Batch API, а когда нет?When does the Batch API fit, and when not?

    показать ответshow answer

    Уместен: большие пачки несрочных задач — ночная классификация, переоценка портфеля раз в день, обработка очереди задач. В 2 раза дешевле, ответ через ~30 минут. Не уместен: интерактивные сценарии, где пользователь ждёт ответ (Telegram-бот, AI-консьерж).

    Fits: big batches of non-urgent tasks — nightly classification, daily portfolio rescore, queue processing. 2× cheaper, replies in ~30 minutes. Doesn’t fit: interactive flows where the user waits (Telegram bot, AI concierge).

  6. Зачем команда /forget в AI-консьерже?Why a /forget command in the AI concierge?

    показать ответshow answer

    GDPR и здравый смысл. Бот хранит факты о пользователе — пользователь должен иметь возможность их удалить одной кнопкой. Без /forget ваш бот не соответствует европейскому регулированию (а кроме того — неэтичен).

    GDPR and common sense. The bot stores facts about the user — the user must wipe them with one button. Without /forget your bot doesn’t meet EU rules (and beyond that — it’s unethical).

  7. Какие шесть мер безопасности обязательны в headless-агенте?What six safety measures are mandatory in a headless agent?

    показать ответshow answer

    1) Узкий список разрешённых tools; 2) max_iterations на agent loop; 3) тайм-аут на каждый запрос; 4) аудит-лог в файл/PostHog; 5) бюджет (на запуск, на день); 6) sandboxing файловых операций (tmp/ или artifact-папка). В Claude Code это «по умолчанию», в SDK — ваша ответственность.

    1) Narrow allowed-tools list; 2) max_iterations on the agent loop; 3) per-request timeout; 4) audit log to file/PostHog; 5) budget (per run, per day); 6) sandbox file operations (tmp/ or artifact folder). Default in Claude Code, your duty in the SDK.
